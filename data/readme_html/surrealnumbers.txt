<p dir="auto">This is a package implementing some parts of the Surreal Number system
invented by John Horton Conway, and explained by Knuth in <a href="http://www.math.harvard.edu/~knill/teaching/mathe320_2015_fall/blog15/surreal1.pdf" rel="nofollow">"Surreal
Numbers: How Two Ex-Students Turned on to Pure Mathematics and Found
Total
Happiness."</a></p>
<p dir="auto">It isn't intended to be useful, so much as educational, and an
interesting test of Julia itseld. It was educational for me to write,
in terms of learning Julia. I wanted a task that would be painful to
do in Matlab but "easy" to do in Julia. It might also be helpful, I
hope, for someone trying to learn about Surreal numbers. I certainly
did learn a lot about them that would have been easily brushed to the
side if I had only gone through the theorems.</p>
<h2 dir="auto"><a id="user-content-background-the-surreal-numbers" class="anchor" aria-hidden="true" href="#background-the-surreal-numbers"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Background: the Surreal Numbers</h2>
<p dir="auto">Surreal numbers aren't numbers as we are taught, but they have many of
the same properties. The tricky thing is that they are defined
recursively from the very start. The nice part is that they use only
set operations.</p>
<p dir="auto">The definition is as follows: a surreal number <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x$</math-renderer> is an ordered pair
of sets of surreal numbers (call them the left set <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_L$</math-renderer> and right set
<math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_R$</math-renderer>) such that every member of the left set is <code>&lt;</code> all of the
members of the right set (technically no member of the left set is
<math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$&amp;gt;=$</math-renderer> a member of the right).</p>
<p dir="auto">There is a starting point -- we can always use empty sets -- as so the
first surreal number (usually denoted zero, because it will turn out
to be the additive identity) is <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">${ | }$</math-renderer>, where I will use this
bracket and pipe notation to denote <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x = { X_L | X_R}$</math-renderer>. Also, to
make the empty spaces clearer (and the coding more efficient) I have
defined $ âˆ… = { } $ the empty set.</p>
<p dir="auto">Then on the "first day" a new generation of surreals can be created in
terms of the initial case. On the second day we create a new generation
and so on. Each has a meaning corresponding to traditional numbers in
order to place a consistent interpretation on standard mathematical
operators defined on the surreals.</p>
<p dir="auto">Just to reiterate, the tricky thing is that everything is
recursive. Even comparitors like <code>&lt;</code>, and hence, we can't even know
if something is a valid surreal in its own right, but only through
recursively investigating its component sets.</p>

<p dir="auto">The purpose here is simply to introduce the key reasons for
implementing this in Julia -- it enables users to define
high-performance types of their own, and those types can be
recursively defined.</p>
<p dir="auto">I know you can do this in other languages, and in some cases also
achieve high performance. But it's <em>so</em> easy in Julia.</p>

<p dir="auto">Let's start off with some small examples of using the package. After
installing the package we can create basic surreals using (i) the
constructor, (ii) conversion from another real number, or (iii) a couple of
special functions, e.g., <code>zero</code> and <code>one</code>. There are two constructors,
one includes an extra string we'll call the <em>shorthand</em> for the
surreal. It's use in printing out numbers. The second constructor, and
many other operators leave this blank. The empty set is designated by
âˆ…, which you can get in Julia by typing <code>\emptyset TAB</code>.</p>

<h2 dir="auto">
<a id="user-content-icky-bits----implementation-details" class="anchor" aria-hidden="true" href="#icky-bits----implementation-details"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Icky bits -- implementation details</h2>
<p dir="auto">While surreals use sets, and Julia has a Set type, implementation
wise, sorted arrays of surreals were a better bet because that
exchanges a once off cost when constructing a surreal with a reduced
cost for many operations throughout the lifetime of the surreal. Also
Julia creates a whole suite of array functions automagically when you
define scalars so using arrays was a low pain way to get things
working. Also, although the surreals use sets, i.e., order of the
elements is not important, almost all texts do write these sets in
order.</p>
<p dir="auto">The surreals include all real numbers (and infinity and epsilon and
others). However, many of these require infinitely large sets <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_L$</math-renderer>
and/or <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_R$</math-renderer>. I have some ideas about how to do that (using lazy
evaluation), but they aren't fleshed out yet so for the moment, I will
restrict myself to finite surreals, i.e., surreals with finite sets
<math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_L$</math-renderer> and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_R$</math-renderer>.</p>
<p dir="auto">So the type <code>Surreal</code> is an abstract type (a subtype of <code>Real</code>) with
at present only one useful subtype <code>SurrealShort</code>, where finite
(here) means that the representation is finite, not that the number
itself is finite.</p>
<p dir="auto">This is a pretty rich set by itself, but it doesn't cover even the
entire set of Rationals; only the <em>dyadic</em> rationals. So a few words
on them seem in order (see below).</p>
<p dir="auto">The plan is to add a <code>SurrealTrans</code> for the transfinites and other
surreals with infinite representations. But that is a little harder to
do, so I leave that for the moment.</p>
<p dir="auto">The actual type structure (minus the constructor -- see the code) is
just</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="struct SurrealShort &lt;: Surreal
    shorthand::String
    L::Array{SurrealShort,1} 
    R::Array{SurrealShort,1}  
    h::UInt64 
end "><pre class="notranslate"><code>struct SurrealShort &lt;: Surreal
    shorthand::String
    L::Array{SurrealShort,1} 
    R::Array{SurrealShort,1}  
    h::UInt64 
end 
</code></pre></div>
<p dir="auto">Note the addition of a <code>shorthand</code> string, which isn't necessary, but
carries a little bit of extra information to make pretty printing and
hence debugging a little easier.</p>
<p dir="auto">The variable <code>h</code> contains the hash value of the surreal, which is
calculated the first time it is used. This avoids incurring the cost
of this (recursive) calculation every time the value is needed, which
is often as the hash is used to speed up other operations.</p>
<h3 dir="auto">
<a id="user-content-dyadic-numbers" class="anchor" aria-hidden="true" href="#dyadic-numbers"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Dyadic numbers</h3>
<p dir="auto">The dyadic rational numbers are those that have a denominator that is
an exact power of 2, that is, numbers of the form (note seems that
GitHub doesn't let me use Mathjax to interpret maths).</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content=" \[ x = \frac{ n }{ 2^k } \]"><pre class="notranslate"><code> \[ x = \frac{ n }{ 2^k } \]
</code></pre></div>
<p dir="auto">It turns out that every dyadic has a finite representation as a
surreal, and every finite surreal represents a dyadic. The easiest way
to represent this is as a DAG (Directed Acyclic Graph) of the constructions as shown below:</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/examples/dyadic_tree_3_manual.svg"><img src="/examples/dyadic_tree_3_manual.svg" alt="dyadics" style="max-width: 100%;"></a></p>
<p dir="auto">It might seem a little limiting to be restricted to this set, but
remember that floating point numbers are dyadics. They are a (binary)
integer (the <em>significand</em>) multiplied by 2 to the power of a (binary)
exponent (just called the <em>exponent</em>). Thus all (finite) floating
point numbers have a finite surreal representation (though it may be
very, very long).</p>
<p dir="auto">I did start writing a type for dyadics, but I'm not sure what it would
be useful for, so it didn't get far (I realise the irony here -- the
surreals themselves aren't exactly useful).</p>
<h3 dir="auto">
<a id="user-content-converting-a-number-to-a-surreal" class="anchor" aria-hidden="true" href="#converting-a-number-to-a-surreal"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Converting a number to a surreal</h3>
<p dir="auto">The description of a surreal given above generates a <em>form</em>. The forms
satisfy the rules of arithmetic (<math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$+$</math-renderer>, <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$-$</math-renderer>, <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$\times$</math-renderer> and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$/$</math-renderer>), and so
we can identify these with the real numbers. However, there are many
<em>forms</em> that equate to the same real number. Thus there are sets of
<em>forms</em> that are equivalent in the sense that <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x \equiv y$</math-renderer> if and only
if <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x \leq y$</math-renderer> and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$y \leq x$</math-renderer>.</p>
<p dir="auto">I think of this loosely by analogy to the rationals, e.g., we can have
two different forms that have the same value, and we usually call the
same "number"</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="2 // 4 = 1 // 2"><pre class="notranslate"><code>2 // 4 = 1 // 2
</code></pre></div>
<p dir="auto">However, it seems important to distinguish equality (where two forms
are identical) from equivalence (where they have the same value). In
programming terms two "things" are equal when they are the same, not
some airy-fairy notion of equivalence, so equality and equivalence
have different meanings and uses. Hence, here we have the relation â‰…
defined to test equivalence separate from ==.</p>
<p dir="auto">BTW, here we hit one of the weirdnesses of Julia; 99% of the time,
you can redefine operators and comparators to do whatever you like on
your new type. But you can't redefine <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$===$</math-renderer> or <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$\equiv$</math-renderer>. The blog I
read suggests that this is because this is a core operation, that
might cause problems if a user broke it. From what I can tell, <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$===$</math-renderer>
tests that the things being tested are the same object, i.e. ,the same
piece of memory. That doesn't seem like something we want to mess
with.</p>
<ul dir="auto">
<li><a href="https://docs.julialang.org/en/stable/devdocs/functions/#Builtins-1" rel="nofollow">https://docs.julialang.org/en/stable/devdocs/functions/#Builtins-1</a></li>
<li><a href="https://discourse.julialang.org/t/overload-for-custom-type/4898" rel="nofollow">https://discourse.julialang.org/t/overload-for-custom-type/4898</a></li>
</ul>
<p dir="auto">Anyway, I have defined <em>congruence</em> <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$\cong$</math-renderer> or <code>â‰…</code> to do the same
thing, check for equivalence (equality of value). However, as there
are many possible surreal forms we could create to represent any given
real number, we have to chose one. Call that the <em>canonical</em> form. We
could define it in several ways, but the standard is</p>
<ul dir="auto">
<li>
<p dir="auto">zero =&gt; <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$0 = { | }$</math-renderer></p>
</li>
<li>
<p dir="auto">integers n =&gt; <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$n+1 = { n | }$</math-renderer></p>
</li>
<li>
<p dir="auto">dyadic fraction =&gt; $ x = \frac{ n }{ 2^k } $ (for n odd) becomes</p>
<p dir="auto">[  { x - 1/2^{k} | x + 1/2^{k} } ]</p>
</li>
<li>
<p dir="auto">negative number =&gt; use the identity that <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$-x = {-X_R | -X_L}$</math-renderer></p>
</li>
</ul>
<p dir="auto">This is the set of conversions implemented, and it includes all
dyadics, and hence floats. Note the recursion inherent in the
construction/conversion. So some care (particularly) with floats
should be taken not to create a variable that exhausts the stack. The
current code isn't very clever in checking for this another place some
work is needed.</p>

<h3 dir="auto">
<a id="user-content-converting-a-surreal-back-to-a-real" class="anchor" aria-hidden="true" href="#converting-a-surreal-back-to-a-real"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Converting a surreal back to a real</h3>
<p dir="auto">The conversion of canonical forms is relatively easy, but remember
that non-canonical forms are possible, and can be quite
counter-intuitive.</p>
<p dir="auto">For instance, naively, you might expect that the form ${ 3 |
17 }$ could be mapped to the mean of the two sets, i.e., 10.
However, <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x = { X_L | X_R }$</math-renderer> is the simplest number such that $X_L &lt; x
&lt; X_R$, so, in fact, this form is equivalent to 4.</p>
<p dir="auto">Note, often in texts it is written <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X \not \leq Y$</math-renderer> whereas I am
writing <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X &amp;gt; Y$</math-renderer>. The original definition is intended (I think) to take
care of the case where one or the other is the empty set, but we can
equally define <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$&amp;gt;$</math-renderer> to be synonomous with <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$\not \leq$</math-renderer>, and just move
on.</p>
<p dir="auto">Conway defines a surreal <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x = { X_L | X_R }$</math-renderer> to be the simplest
<math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x$</math-renderer> that satisfies <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_L &amp;lt; x &amp;lt; X_R$</math-renderer>, where simplest means comes from
the earliest generation.</p>
<p dir="auto">The secret of the conversion is again to use recursion, but that is
not quite enough in this case. We use several tricks along the way:</p>
<ul dir="auto">
<li>If <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x$</math-renderer> is equivalent to a known surreal such as 0 or 1, we convert
directly</li>
<li>If <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x$</math-renderer> is negative, we use the identity that <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$-x = { -X_R | -X_L }$</math-renderer>
</li>
<li>And, most importantly, if <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$0 &amp;lt; x &amp;lt; 1$</math-renderer> we know <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x$</math-renderer> will be the
"simplest" dyadic rational number such that <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x_L &amp;lt; x &amp;lt; x_R$</math-renderer>. In the
interval <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$(0,1)$</math-renderer> simplest means having the denominator with the lowest
power, i.e., in order of preference we would like the denomator to the
<math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$1,2,3, \ldots$</math-renderer>.</li>
</ul>
<p dir="auto">We can find the latter though a simple modification of the standard
binary search a simplified version of which is shown below.</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content=" a = 0; b = 1

   if xl &lt; c &lt; xr

 while true
   d = (a + b) / 2   
   if xl &lt; c &lt; xr
     return d
   elseif c &lt;= xl

   elseif c &gt;= xr 
     b = d
   end

</code></pre></div>
<p dir="auto">The first and last rules allow us to convert any number $x \in
[0,1]$. To convert numbers above this range, i.e., <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x &amp;gt; 1$</math-renderer>, we
subtract 1 (the surreal additive identity), convert the result
(recursively), and then add back 1 (this time a real). To convert
negative numbers we apply the second rule.</p>
<p dir="auto">The result is not the world's most beautiful code -- I'm sure it can
be improved, but there are so many other inefficiency's here, I am not
sure it warrants it.</p>
<h3 dir="auto">
<a id="user-content-implementations-of-operators-and-standard-functions" class="anchor" aria-hidden="true" href="#implementations-of-operators-and-standard-functions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Implementations of operators and standard functions</h3>
<p dir="auto">Most of the operators follow standard surreal definitions and defining
them in Julia is easy. They are all recursive, as you might guess,
and so very inefficient -- I wouldn't want to do demanding
computations this way, but they are easy to understand, for instance</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content=" +(x::SurrealShort, y::SurrealShort) = SurrealShort([x.X_L .+ y; x .+ y.X_L], [x.X_R .+ y; x .+ y.X_R] )"><pre class="notranslate"><code> +(x::SurrealShort, y::SurrealShort) = SurrealShort([x.X_L .+ y; x .+ y.X_L], [x.X_R .+ y; x .+ y.X_R] )
</code></pre></div>
<p dir="auto">Notice that we are exploiting here Julia's natural extension of
operators from scalar to vectors (this is one of the reasons that
using arrays instead of sets for <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_L$</math-renderer> and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_R$</math-renderer> is appealling). Thus we can write</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content=" x .+ [y_1, y_2]"><pre class="notranslate"><code> x .+ [y_1, y_2]
</code></pre></div>
<p dir="auto">once addition is defined on the scalar surreal type, without any additional definitions, and this is particular appealling here as the scalar addition operator is recursively defined in terms of the vector+scalar addition <code>.+</code>.</p>
<p dir="auto">Mutiplication was a bit of a bugbear to get right because I
misinterpreted the definition. The definition of multiplication of
<math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x= { X_L |X_R }$</math-renderer> and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$y={ Y_L | Y_R }$</math-renderer> had terms like</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="X_L y + x Y_L - X_L Y_L"><pre class="notranslate"><code>X_L y + x Y_L - X_L Y_L
</code></pre></div>
<p dir="auto">I assumed the <code>+</code> operator in this definition was the standard
(surreal) addition of sets of surreals, but it isn't. Instead the
entire expression should be interpreted as "for all pairs of elements
from X_L and Y_L perform the above computation, and form a set from
the results." That's easy enough to do once you work out what you are
trying to do. There is another underlying problem which is that
canonical forms all have sets with 1 or 0 elements, and my broken
multiplication worked for them. It was only when I created products of
non-canonical forms that I saw problems, and then only when I had
cleaned out another misunderstanding from the code.</p>
<p dir="auto">The one interesting thing to note is that even simple computations often
generate non-canonical forms. Part of the aim of this package was to
let people experiment and see such things. It is otherwise far to
laborious to calculate anything but the very simplest cases (none of
the above texts do any but the simplest).</p>
<p dir="auto">One really useful example (for me to understand the surreals) is the
following, which was obtained by multiplying the canonical form of 2
with itself. You might naively think that operators applied to
canonical forms resulted in canonical forms, but this is not the
case. The form is quite complex. Moreover, superficially it recurse
into itself, i.e., the surreal number 4 appears twice in the
tree. However, the form is defined in terms of the tree, so the
dependent "4" is not the same surreal as the parent "4". That is, they
are not  equal, but they are equivalent. Thus the parent "4" is
defined in terms of simpler forms (even though one of these is
equivalent).</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="/test/Data/test_dot_x43.svg"><img src="/test/Data/test_dot_x43.svg" alt="2x2 = 4" style="max-width: 100%;"></a></p>
<p dir="auto">The other pieces of the toolkit are the standard things you expect to
be able to do with numbers, e.g. round, sign, isinteger, ...  I
haven't implemented all of these, but hopefully enough that any others
can be easily added.</p>
<p dir="auto">There are two approaches: one is to use intrinsic surreal arithmetic,
e.g. <code>sign</code> and <code>abs</code> are implemented using native surreal arithmetic and operators. The result is that they looks almost exactly like it would for any other number.</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="sign(x::SurrealShort) = x&lt;zero(x) ? -one(x) : x&gt;zero(x) ? one(x) : zero(x)
abs(x::SurrealShort) = x&lt;zero(x) ? -x : x"><pre class="notranslate"><code>sign(x::SurrealShort) = x&lt;zero(x) ? -one(x) : x&gt;zero(x) ? one(x) : zero(x)
abs(x::SurrealShort) = x&lt;zero(x) ? -x : x
</code></pre></div>
<p dir="auto">Actually, I don't even have to define <code>abs</code> as I get this for free
because Julia has a similar operation defined <code>abs(x::Real)</code> for all
real numbers. This is a great feature of Julia's type hierarchy and
multiple dispatch function selection.</p>
<p dir="auto">The other approach to function definition is somewhat of a cheat. It
involves converting the number to a real, and then using the
appropriate operation on that field. I have tried to avoid that
approach when possible. At the moment, only the <code>simplify</code> function
uses this approach. But if I am going to do more complex math
functions, e.g., logs or trig functions, I think I will have to take
this approach. The mathematical definitions of such in pure surreal
terms are obscure.</p>
<p dir="auto">The cheat is used as part of one of the pieces of this that is hard to
implement, namely division. Division is well-defined on the surreals,
but, as even simple divisions such as 1/3 result in non-dyadic rational
numbers, and hence have a infinite representations as a surreal, the
class of "finite" surreals defined here is not <em>closed</em> under
division. Eventually, this can be solved by outputting a non-finite
surreal, but for the moment I have only implemented division when
dividing by <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$2^k$</math-renderer> or by a divisor of the numerator of the dyadic. The
first is easy (it can be implemented in terms of multiplication). The
second (at the moment) requires the cheat (see below) of converting
back into a rational number, but in either case the check to see which
case requires the conversion. Some work is needed here.</p>

sometimes we want to see deeper. In the case where shorthand is
defined we can use the command <code>pf</code> to see deeper, but it will stop at
the first layer below with a shorthand.</p>

<p dir="auto">There are two pieces that are unique to surreals:</p>
<p dir="auto"><strong>Generation:</strong> the generation of a surreal is 1 + the generation of
the surreals used to construct it. Again this is easy to implement
recursively.  Generation comes from Knuth's story where it's called
the "birth day" of the surreal. Generation can be thought of as a
function <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$g(\cdot)$</math-renderer></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content=" \[ g(x) = \sup_{y \in X_L \cup X_R} \big[ g(y) + 1 \big] \]"><pre class="notranslate"><code> \[ g(x) = \sup_{y \in X_L \cup X_R} \big[ g(y) + 1 \big] \]
</code></pre></div>
<p dir="auto">Implicitly, generation is the tree <em>depth</em>, if viewed as the recursive
tree shown above.</p>
<p dir="auto">Generation is important as <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$x = { X_L | X_R }$</math-renderer> is defined to be the
simplest number such that <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="af09e9246bb39f82753e789d8a672b11">$X_L &amp;lt; x &amp;lt; X_R$</math-renderer>, i.e., the number with the
lowest birth day.</p>
<p dir="auto">Note that within an equivalence class of forms, not all forms have the
same generation.</p>
<p dir="auto"><strong>Canonicalise:</strong> convert a surreal into its equivalent canonical
form. The easiest way to implement this was to use a similar cheat to
that above, i.e., convert to a real, and then convert back to the
equivalent surreal in canonical form.</p>

<p dir="auto">We mentioned arrays of surreals earlier -- they are used in the
implementation instead of sets. One of the nice things about Julia us
that you get many of the array operators and functions for free when
you create scalar operators. So, for instance, you can write</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="convert.(SurrealShort, [-1, 0, 1, 2] )"><pre class="notranslate"><code>convert.(SurrealShort, [-1, 0, 1, 2] )
</code></pre></div>
<p dir="auto">which will broadcasts the convert function across the array of
integers to create an array containing the respective
surreals. Likewise, we can use <em>comprehensions</em> to construct arrays,
e.g.,</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="[ convert(SurrealShort, i) for i=-1:2 ]"><pre class="notranslate"><code>[ convert(SurrealShort, i) for i=-1:2 ]
</code></pre></div>
<p dir="auto">Or we could construct and iterator for the same thing (once <code>floor</code>
and some promotion rules are defined), e.g.., the iterator from -1 to 2 is</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="convert(SurrealShort, -1 ):convert(SurrealShort, 2)"><pre class="notranslate"><code>convert(SurrealShort, -1 ):convert(SurrealShort, 2)
</code></pre></div>
<p dir="auto">However, in order to use arrays as sets we need, in the constructor
for a surreal to reduce the "set" to a sorted array containing unique
elements. Julia has nice sort and unique functions, but they rely on
hash functions, so we have to add a hash. These need to be recursive,
and work for arrays of surreals as well. The hash function help says
that we should implement such for new types such that <code>isequal(x,y)</code>
implies <code>hash(x)==hash(y)</code>, with a second argument to be mixed in the
results. This is linked to the idea that we should separate the == and
the <em>congruent</em> comparisons -- my hash is based on two forms being
equal, not being equivalent. In any case, Julia's syntax is again
simple and concise for specifying the hashes need (believe it or not
its another recursive function), but I need the hash to be really
fast. I use it to speed other pieces of code up, so I can't afford to
recurse every time. So instead, the hash value is calculated the first
time it is needed, and then stored in the surreal data structure
itself, so that we only need call the function once at most. We could
have defined the hash in the constructor, but it turns out that many
temporary surreals are created in multiplication (in particular), and
so avoiding the overhead when it isn't needed is just fine.</p>
