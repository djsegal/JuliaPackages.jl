<p><code>Multirate</code> is a package for the creation and execution of streaming (stateful) FIR filters which can also perform sample rate conversion.</p>

<p>These methods allow one-time filter of a vector without the creating of a filter object.</p>
<p>In these examples, <code>x</code> and <code>h</code> are previously defined signal &amp; filter-taps vectors.</p>

<h2><a id="user-content-streaming" class="anchor" aria-hidden="true" href="#streaming"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Streaming</h2>
<p>To use <code>Multirate</code>'s streaming filter functionality, you must first create a filter object. Every time you call <code>filt</code> with that object, the filtering processes picks up where it left off. This is good for processing large vectors from a file, or filtering a stream of samples of indefinite length.</p>
<p>Each filter object is of type <code>FIRFilter{Tk&lt;:FIRKernel}</code>. There are four subtypes of <code>FIRKernel</code>:</p>
<ul>
<li><code>FIRStandard</code>: normal single-rate FIR filter</li>
<li><code>FIRInterpolator</code>: resampling with a ratio of <code>L//1</code></li>
<li><code>FIRDecimator</code>: resampling with a ratio of <code>1//M</code></li>
<li><code>FIRRational</code>: resampling with a ratio of <code>L/M</code></li>
</ul>
<p>You do not need to specify the kernel type. It is chosen for you when you based on the resampling ratio you specify when creating a new <code>FIRFilter</code> object.</p>
<p>One thing to note is that for decimation and rational resampling, there are instances where <code>filt</code> will return an empty vector. This is because those filters require a certain number of inputs to compute an output. If you  call <code>filt</code> and it returns an empty vector, it is working normally, and will eventually return processed samples when it receives enough inputs. However, your code should be written in a way that does not assume that the returned vectors are not empty.</p>

<h1><a id="user-content-speed" class="anchor" aria-hidden="true" href="#speed"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Speed</h1>
<p>The naive approach to resampling is a two to three step process, depending on whether you are decimating, interpolating, or both in case of rational resampling. For a rational factor, the processes look like:</p>
<ol>
<li>Stuff <code>L-1</code> zeros between each input sample.</li>
<li>Apply your anti-alias filter. This fills in those zeros from step 1.</li>
<li>Keep one out of <code>M</code> samples from step 2, and throw away the rest.</li>
</ol>
<p>Let's design a naive rational resampler:</p>
<div class="highlight highlight-source-julia"><pre><span class="pl-k">function</span> <span class="pl-en">naiveresampler</span><span class="pl-c1">{T}</span>( h<span class="pl-k">::</span><span class="pl-c1">Vector{T}</span>, x<span class="pl-k">::</span><span class="pl-c1">Vector{T}</span>, ratio<span class="pl-k">::</span><span class="pl-c1">Rational{Int}</span> )
    upfactor   <span class="pl-k">=</span> <span class="pl-c1">num</span>( ratio )
    downfactor <span class="pl-k">=</span> <span class="pl-c1">den</span>( ratio )
    xStuffed   <span class="pl-k">=</span> <span class="pl-c1">zeros</span>( T, <span class="pl-c1">length</span>(x) <span class="pl-k">*</span> upfactor )

    <span class="pl-k">for</span> n <span class="pl-k">in</span> <span class="pl-c1">0</span><span class="pl-k">:</span><span class="pl-c1">length</span>(x)<span class="pl-k">-</span><span class="pl-c1">1</span>
        xStuffed[ n<span class="pl-k">*</span>upfactor<span class="pl-k">+</span><span class="pl-c1">1</span> ] <span class="pl-k">=</span> x[ n<span class="pl-k">+</span><span class="pl-c1">1</span> ]
    <span class="pl-k">end</span>

    yInterpolated <span class="pl-k">=</span> DSP<span class="pl-k">.</span><span class="pl-c1">firfilt</span>( h, xStuffed )
    y <span class="pl-k">=</span> [ yInterpolated[n] <span class="pl-k">for</span> n <span class="pl-k">=</span> <span class="pl-c1">1</span><span class="pl-k">:</span>downfactor<span class="pl-k">:</span><span class="pl-c1">length</span>( yInterpolated ) ]
<span class="pl-k">end</span></pre></div>
<p>and define some input samples:</p>
<div class="highlight highlight-source-julia"><pre><span class="pl-c"><span class="pl-c">#</span> resample from 48 kHz to 44.1 kHz</span>
sampleRate    <span class="pl-k">=</span> <span class="pl-c1">48000</span>
interpolation <span class="pl-k">=</span> <span class="pl-c1">147</span>
decimation    <span class="pl-k">=</span> <span class="pl-c1">160</span>
ratio         <span class="pl-k">=</span> interpolation<span class="pl-k">//</span>decimation
numTaps       <span class="pl-k">=</span> <span class="pl-c1">24</span><span class="pl-k">*</span>interpolation           <span class="pl-c"><span class="pl-c">#</span> that's a lot of taps</span>
x             <span class="pl-k">=</span> <span class="pl-c1">rand</span>( Float32, <span class="pl-c1">1_000_000</span> )
h             <span class="pl-k">=</span> Multirate<span class="pl-k">.</span><span class="pl-c1">firdes</span>( numTaps, <span class="pl-c1">0.5</span><span class="pl-k">/</span>interpolation, Multirate<span class="pl-k">.</span>kaiser, beta <span class="pl-k">=</span> <span class="pl-c1">7.8562</span>  )</pre></div>
<p>Now let's time <code>naiveresampler</code> and <code>Multirate.filt</code>:</p>
<pre lang="jlcon"><code>julia&gt; @time y = naiveresampler( h, x, 147//160 );
elapsed time: **3.713267311 seconds** (**2361632424** bytes allocated, 5.05% gc time)
</code></pre>
<pre lang="jlcon"><code>julia&gt; resampler = Multirate.FIRFilter( h, ratio );

julia&gt; @time y = Multirate.filt( resampler, x );
elapsed time: **0.056938961 seconds** (**7350144** bytes allocated)
</code></pre>
<p>The difference in speed is huge, but not necessarily surprising. Perhaps the most surprising part of this demo is the memory allocation. <code>naiveresampler</code> allocated almost 2.4 GB vs <code>Multirate.filt</code>'s 7.4 MB.</p>
<p><code>Multirate</code> is not heavily optimized numerically speaking, and has much room for improvement, but by using polyphase FIR implementation it is able to only do math on the samples that are kept. It doesn't waste cycles multiplying taps by stuffed zeros, and it does not do any multiplications on samples that are thrown out in the decimation process. Despite this, it produces results identical to the naive approach.</p>
<p>One way the code is optimized is that the filter-taps/polyphase-filter-banks are flipped in memory. Many filter routines found on the internet use the literal definition of discrete convolution by moving forward in memory through the input samples, and backwards in memory through the filter taps, or vice versa. Having the coefficients flipped is more friendly to the compiler. In some cases the compiler will emit <a href="http://en.wikipedia.org/wiki/SIMD" rel="nofollow">SIMD</a> instructions to perform multiple multiplications at a time.</p>
