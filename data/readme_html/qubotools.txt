<h2 dir="auto"><a id="user-content-introduction" class="anchor" aria-hidden="true" href="#introduction"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Introduction</h2>
<p dir="auto">The <code>QUBOTools.jl</code> package implements codecs for QUBO (<em>Quadratic Unconstrained Binary Optimization</em>) instances.
Its purpose is to provide fast and reliable conversion between common formats used to represent such problems.
This allows for rapid leverage of many emergent computing architectures whose job is to solve this kind of optimization problem.</p>
<p dir="auto">The term QUBO is widely used when referring to <em>boolean</em> problems of the form</p>
<p dir="auto"><math-renderer class="js-display-math" style="display: block" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$$\begin{array}{rl}
\min &amp; \mathbf{x}'\ Q\ \mathbf{x} \\
\text{s.t.} &amp; \mathbf{x} \in \mathbb{B}^{n}
\end{array}$$</math-renderer></p>
<p dir="auto">with symmetric <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$Q \in \mathbb{R}^{n \times n}$</math-renderer>. Nevertheless, this package also fully supports <em>Ising Models</em>, given by</p>
<p dir="auto"><math-renderer class="js-display-math" style="display: block" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$$\begin{array}{rl}
\min &amp; \mathbf{s}'\ J\ \mathbf{s} + \mathbf{h}'\ \mathbf{s} \\
\text{s.t.} &amp; \mathbf{s} \in \left\lbrace-1, 1\right\rbrace^{n}
\end{array}$$</math-renderer></p>
<p dir="auto">where <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$J \in \mathbb{R}^{n \times n}$</math-renderer> is triangular and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$\mathbf{h} \in \mathbb{R}^{n}$</math-renderer>.</p>

</h3>
<p dir="auto">The <a href="https://bqpjson.readthedocs.io" rel="nofollow">BQPJSON</a> format was designed at <a href="https://github.com/lanl-ansi">LANL-ANSI</a> to represent Binary Quadratic Programs in a platform-independet fashion.
This is accomplished by using <code>.json</code> files validated using a well-defined <a href="/src/models/bqpjson.schema.json">JSON Schema</a>.</p>

<h2 dir="auto">
<a id="user-content-backend" class="anchor" aria-hidden="true" href="#backend"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Backend</h2>
<p dir="auto">The <code>AbstractModel{D}</code> abstract type is defined, where <code>D &lt;: Domain</code>.
Available variable domains are <code>BoolDomain</code> and <code>SpinDomain</code>, respectively, <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$x \in \mathbb{B} = \lbrace 0, 1 \rbrace$</math-renderer> and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$s \in \lbrace -1, 1 \rbrace$</math-renderer>.
Conversion between domains follows the identity</p>
<p dir="auto"><math-renderer class="js-display-math" style="display: block" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$$s = 2x - 1$$</math-renderer></p>
<p dir="auto"><strong>QUBOTools.jl</strong> also exports the <code>StandardQUBOModel{S, U, T, D} &lt;: AbstractModel{D}</code> type, designed to work as a powerful standard backend for all other models.
Here, <code>S &lt;: Any</code> plays the role of variable indexing type and usually defaults to <code>Int</code>.
It is followed by <code>U &lt;: Integer</code>, used to store sampled states of type <code>Vector{U}</code>.</p>
<p dir="auto">When <code>D &lt;: SpinDomain</code>, it is necessary that <code>U &lt;: Signed</code>.
<code>T &lt;: Real</code> is the type used to represent all coefficients.
It is also the choice for the energy values corresponding to each solution.
It's commonly set as <code>Float64</code>.</p>
<p dir="auto">This package's mathematical formulation was inspired by <strong>BQPJSON</strong>'s, and is given by</p>
<p dir="auto"><math-renderer class="js-display-math" style="display: block" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$$\min f(\mathbf{x}) = \alpha \left[{ \sum_{i &lt; j} q_{i, j},x_{i},x_{j} +\sum_{i} l_{i},x_{i} + \beta }\right]$$</math-renderer></p>
<p dir="auto">where <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$\alpha$</math-renderer> is a scaling factor, <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$\beta$</math-renderer> a constant offset, <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$l_{i},x_{i}$</math-renderer> are the linear terms and <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$q_{i, j},x_{i},x_{j}$</math-renderer> the quadratic ones.</p>
<p dir="auto">We defined our problems to follow a minimization sense by default.
The scaling factor <math-renderer class="js-inline-math" style="display: inline" data-static-url="https://github.githubassets.com/static" data-run-id="90b50d6870caaa16f4022708d9038313">$\alpha$</math-renderer> is strictly positive.</p>
<h3 dir="auto">
<a id="user-content-jump-integration" class="anchor" aria-hidden="true" href="#jump-integration"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a><a href="https://jump.dev" rel="nofollow">JuMP</a> Integration</h3>
<p dir="auto">One of the main ideas was to make JuMP / MathOptInterface integration easy and, in fact, the implemented backend does a lot of the the data crunching.
When <code>S</code> is set to <code>MOI.VariableIndex</code> and <code>T</code> matches <code>Optimzer{T}</code>, we can say that most of the hard work is done.</p>
