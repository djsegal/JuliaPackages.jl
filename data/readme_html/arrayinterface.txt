<p>Julia has only recently reached v1.0 and the AbstractArray interface is still
quite new. The purpose of this library is to solidify extensions to the current
AbstractArray interface which are put to use in package ecosystems like
DifferentialEquations.jl. Since these libraries are live, this package will
serve as a staging ground for ideas before they merged into Base Julia. For this
reason, no functionality is exported so that way if such functions are added
and exported in a future Base Julia there will be no issues with the upgrade.</p>

<h2><a id="user-content-zeromatrixu" class="anchor" aria-hidden="true" href="#zeromatrixu"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>zeromatrix(u)</h2>
<p>Creates the zero'd matrix version of <code>u</code>. Note that this is unique because
<code>similar(u,length(u),length(u))</code> returns a mutable type, so is not type-matching,
while <code>fill(zero(eltype(u)),length(u),length(u))</code> doesn't match the array type,
i.e. you'll get a CPU array from a GPU array. The generic fallback is
<code>u .* u' .* false</code> which works on a surprising number of types, but can be broken
with weird (recursive) broadcast overloads. For higher order tensors, this
returns the matrix linear operator type which acts on the <code>vec</code> of the array.</p>
<h2><a id="user-content-restructurexy" class="anchor" aria-hidden="true" href="#restructurexy"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>restructure(x,y)</h2>
<p>Restructures the object <code>y</code> into a shape of <code>x</code>, keeping its values intact. For
simple objects like an <code>Array</code>, this simply amounts to a reshape. However, for
more complex objects such as an <code>ArrayPartition</code>, not all of the structural
information is adequately contained in the type for standard tools to work. In
these cases, <code>restructure</code> gives a way to convert for example an <code>Array</code> into
a matching <code>ArrayPartition</code>.</p>

<p>2.0: Changed the default of <code>ismutable(array::AbstractArray) = true</code>. We previously defaulted to
<code>Base.@pure ismutable(array::AbstractArray) = typeof(array).mutable</code>, but there are a lot of cases
where this tends to not work out in a way one would expect. For example, if you put a normal array
into an immutable struct that adds more information to it, this is considered immutable, even if
all of the <code>setindex!</code> methods work (by forwarding to the mutable array). Thus it seems safer to just
always assume mutability is standard for an array, and allow arrays to opt-out.</p>
