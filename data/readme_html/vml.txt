<p dir="auto">This package provides bindings to the Intel MKL <a href="https://www.intel.com/content/www/us/en/develop/documentation/onemkl-developer-reference-c/top/vector-mathematical-functions.html" rel="nofollow">Vector Mathematics Functions</a>.
This is often substantially faster than broadcasting Julia's built-in functions, especially when applying a transcendental function over a large array.
Until Julia 0.6 the package was registered as <code>VML.jl</code>.</p>
<p dir="auto">Similar packages are <a href="https://github.com/JuliaMath/Yeppp.jl">Yeppp.jl</a>, which wraps the open-source Yeppp library, and <a href="https://github.com/JuliaMath/AppleAccelerate.jl">AppleAccelerate.jl</a>, which provides access to macOS's Accelerate framework.</p>

<p dir="auto">There is currently <a href="https://github.com/JuliaPackaging/BinaryBuilder.jl/issues/700" data-hovercard-type="issue" data-hovercard-url="/JuliaPackaging/BinaryBuilder.jl/issues/700/hovercard">the following</a> issue between the <code>CompilerSupportLibraries_jll</code> artifact, which is used for example by <code>SpecialFunctions.jl</code>, and <code>MKL_jll</code>. Unless <code>MKL_jll</code> is loaded first, there might be wrong results coming from a small number of function for particular input array lengths. If you are unsure which, if any, your used packages might load this artifact, loading <code>IntelVectorMath</code> as the very first package should be fine.</p>

<p dir="auto">To install IntelVectorMath.jl run</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="julia&gt; ] add IntelVectorMath"><pre>julia<span class="pl-k">&gt;</span> ] add IntelVectorMath</pre></div>
<p dir="auto">Since version 0.4 <code>IntelVectorMath</code> uses the <code>MKL_jll</code> artifact, which is shared with other packages uses MKL, removing several other dependencies. This has the side effect that from version 0.4 onwards this package requires at least Julia 1.3.</p>
<p dir="auto">For older versions of Julia <code>IntelVectorMath v0.3</code> downloads its own version of MKL and keeps only the required files in its own directory. As such installing MKL.jl or MKL via intel are no longer required, and may mean some duplicate files if they are present. However, this package will adopt the new artifact system in the next minor version update and fix this issue.
In the event that MKL was not installed properly you will get an error when first <code>using</code> it. Please try running</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="julia&gt; ] build IntelVectorMath"><pre>julia<span class="pl-k">&gt;</span> ] build IntelVectorMath</pre></div>
<p dir="auto">If this does not work, please open an issue and include the output of <code>&lt;packagedir&gt;/deps/build.log</code>.</p>

<p dir="auto">After loading <code>IntelVectorMath</code>, you have the supported function listed below, for example <code>IntelVectorMath.sin(rand(100))</code>. These should provide a significant speed-up over broadcasting the Base functions.
As the package name is quite long, the alias <code>IVM</code> is also exported to allow <code>IVM.sin(rand(100))</code> after <code>using</code> the package.
If you <code>import</code> the package, you can add this alias via <code>const IVM = IntelVectorMath</code>. Equally, you can replace <code>IVM</code> with another alias of your choice.</p>

<p dir="auto">By default, IntelVectorMath uses <code>VML_HA</code> mode, which corresponds to an accuracy of
&lt;1 ulp, matching the accuracy of Julia's built-in openlibm
implementation, although the exact results may be different. To specify
low accuracy, use <code>vml_set_accuracy(VML_LA)</code>. To specify enhanced
performance, use <code>vml_set_accuracy(VML_EP)</code>. More documentation
regarding these options is available on
<a href="https://www.intel.com/content/www/us/en/develop/documentation/onemkl-developer-reference-c/top/vector-mathematical-functions.html" rel="nofollow">Intel's website</a>.</p>
<h3 dir="auto"><a id="user-content-denormalized-numbers" class="anchor" aria-hidden="true" href="#denormalized-numbers"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Denormalized numbers</h3>
<p dir="auto">On some CPU, operations on denormalized numbers are extremely slow. You case use <code>vml_set_denormalmode(VML_DENORMAL_FAST)</code>
to handle denormalized numbers as zero. See the <code>?VML_DENORMAL_FAST</code> for more information. You can get the
current mode by <code>vml_get_denormalmode()</code>. The default is <code>VML_DENORMAL_ACCURATE</code>.</p>

<p dir="auto">By default, IntelVectorMath uses multithreading. The maximum number of threads that a call may use
is given by <code>vml_get_max_threads()</code>. On most environment this will default to the number of physical
cores available to IntelVectorMath. This behavior can be changed using <code>vml_set_num_threads(numthreads)</code>.</p>
