<p>Testing AWSClusterManagers can be performed on your local system using:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="Pkg.test(&quot;AWSClusterManagers&quot;)
"><pre>Pkg<span class="pl-k">.</span><span class="pl-c1">test</span>(<span class="pl-s"><span class="pl-pds">"</span>AWSClusterManagers<span class="pl-pds">"</span></span>)</pre></div>
<p>Adjustments can be made to the tests with the environmental variables <code>ONLINE</code> and
<code>STACK_NAME</code>:</p>
<ul>
<li><code>ONLINE</code>: Should contain a comma separated list which contain elements from the set
"docker" and/or "batch".  Including "docker" will run the online Docker tests (requires
<a href="https://www.docker.com/community-edition" rel="nofollow">Docker</a> to be installed) and "batch" will run
AWS Batch tests (see <code>STACK_NAME</code> for details).</li>
<li><code>STACK_NAME</code>: Set the AWS Batch tests to use the stack specified. It is expected that
the stack already exists in the current AWS profile. Note that <code>STACK_NAME</code> is only
used if <code>ONLINE</code> contains "batch".</li>
</ul>
<h3><a id="user-content-online-docker-tests" class="anchor" aria-hidden="true" href="#online-docker-tests"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Online Docker tests</h3>
<p>To run the online Docker tests you'll need to have <a href="https://www.docker.com/community-edition" rel="nofollow">Docker</a>
installed. Additionally you'll also need access to pull down the image
"468665244580.dkr.ecr.us-east-1.amazonaws.com/julia-baked" using your current AWS profile.
If your current profile doesn't have access then ask <code>@sudo</code> in <a href="https://invenia.slack.com/messages/C02A3K084/" rel="nofollow">#techsupport</a>
to "Please grant account ID &lt;ACCOUNT_ID&gt; permissions to the <a href="https://console.aws.amazon.com/ecs/home?region=us-east-1#/repositories/julia-baked#permissions" rel="nofollow"><code>julia-baked</code></a> repo".
Make sure to replace <code>&lt;ACCOUNT_ID&gt;</code> with the results of <code>aws sts get-caller-identity --query Account</code>.</p>
<h3><a id="user-content-online-aws-batch-tests" class="anchor" aria-hidden="true" href="#online-aws-batch-tests"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Online AWS Batch tests</h3>
<p>To run the online AWS Batch tests you need all of the requirements as specified in <a href="#online-docker-tests">Online Docker tests</a>,
the current AWS profile should have an aws-batch-manager-test stack running and
<code>STACK_NAME</code> needs to be set.</p>
<p>To make an aws-batch-manager-test compatible stack you can use the included CloudFormation
template <a href="test/batch.yml">batch.yml</a>. Alternatively you should be able to use your own
custom stack but it will be required to have, at a minimum, the named outputs as shown in the
included template.</p>
<h2><a id="user-content-sample-project-architecture" class="anchor" aria-hidden="true" href="#sample-project-architecture"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Sample Project Architecture</h2>
<p>The details of how the AWSECSManager &amp; AWSBatchManager will be described in more detail shortly, but we'll briefly summarizes a real world application archtecture using the AWSBatchManager.</p>
<p><a target="_blank" rel="noopener noreferrer" href="docs/src/assets/figures/batch_project.svg"><img src="docs/src/assets/figures/batch_project.svg" alt="Batch Project" style="max-width:100%;"></a></p>
<p>The client machines on the left (e.g., your laptop) begin by pushing a docker image to ECR, registering a job definition, and submitting a cluster manager batch job.
The cluster manager job (JobID: 9086737) begins executing <code>julia demo.jl</code> which immediately submits 4 more batch jobs (JobIDs: 4636723, 3957289, 8650218 and 7931648) to function as its workers.
The manager then waits for the worker jobs to become available and register themselves with the manager.
Once the workers are available the remainder of the script sees them as ordinary julia worker processes (identified by the integer pid values shown in parentheses).
Finally, the batch manager exits, releasing all batch resources, and writing all STDOUT &amp; STDERR to CloudWatch logs for the clients to view or download and saving an program results to S3.
The clients may then choose to view or download the logs/results at a later time.</p>
