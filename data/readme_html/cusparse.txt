<p dir="auto">CUSPARSE.jl proves bindings to a subset of the CUSPARSE library. It extends the amazing <a href="https://github.com/JuliaGPU/CUDArt.jl">CUDArt.jl</a> library to provide four new sparse matrix classes:</p>
<ul dir="auto">
<li>
<p dir="auto"><code>CudaSparseMatrixCSC</code></p>
</li>
<li>
<p dir="auto"><code>CudaSparseMatrixCSR</code></p>
</li>
<li>
<p dir="auto"><code>CudaSparseMatrixBSR</code></p>
</li>
<li>
<p dir="auto"><code>CudaSparseMatrixHYB</code></p>
</li>
</ul>
<p dir="auto">which implement compressed sparse row/column storage, block CSR, and NVIDIA hybrid (<code>HYB</code>) <code>COO</code>-<code>ELL</code> format on the GPU. Since the native sparse type in Julia is <code>CSC</code>, and in CUSPARSE is <code>CSR</code>, automatic format conversion is provided, so that when you write</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="A = sprand(10,8,0.2)
d_A = CudaSparseMatrixCSR(A)"><pre>A <span class="pl-k">=</span> <span class="pl-c1">sprand</span>(<span class="pl-c1">10</span>,<span class="pl-c1">8</span>,<span class="pl-c1">0.2</span>)
d_A <span class="pl-k">=</span> <span class="pl-c1">CudaSparseMatrixCSR</span>(A)</pre></div>
<p dir="auto"><code>A</code> is transformed into <code>CSC</code> format moved to the GPU, then auto-converted to <code>CSR</code> format for you. Thus, <code>d_A</code> is <em>not</em> a transpose of <code>A</code>! Similarly, if you have a matrix in dense format on the GPU (in a <code>CudaArray</code>), you can simply call <code>sparse</code> to turn it into a sparse representation. Right now <code>sparse</code> by default turns the matrix it is given into <code>CSR</code> format. It takes an optional argument that lets you select <code>CSC</code> or <code>HYB</code>:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="d_A = CudaArray(rand(10,20))
d_A = sparse(d_A) #now in CSR format

d_B = CudaArray(rand(10,20))
d_B = sparse(d_B,'C') #now in CSC format

d_C = CudaArray(rand(10,20))
d_C = sparse(d_C,'H') #now in HYB format

d_D = CudaArray(rand(10,20))
d_D = sparse(d_C,'B') #now in BSR format"><pre>d_A <span class="pl-k">=</span> <span class="pl-c1">CudaArray</span>(<span class="pl-c1">rand</span>(<span class="pl-c1">10</span>,<span class="pl-c1">20</span>))
d_A <span class="pl-k">=</span> <span class="pl-c1">sparse</span>(d_A) <span class="pl-c"><span class="pl-c">#</span>now in CSR format</span>

d_B <span class="pl-k">=</span> <span class="pl-c1">CudaArray</span>(<span class="pl-c1">rand</span>(<span class="pl-c1">10</span>,<span class="pl-c1">20</span>))
d_B <span class="pl-k">=</span> <span class="pl-c1">sparse</span>(d_B,<span class="pl-s"><span class="pl-pds">'</span>C<span class="pl-pds">'</span></span>) <span class="pl-c"><span class="pl-c">#</span>now in CSC format</span>

d_C <span class="pl-k">=</span> <span class="pl-c1">CudaArray</span>(<span class="pl-c1">rand</span>(<span class="pl-c1">10</span>,<span class="pl-c1">20</span>))
d_C <span class="pl-k">=</span> <span class="pl-c1">sparse</span>(d_C,<span class="pl-s"><span class="pl-pds">'</span>H<span class="pl-pds">'</span></span>) <span class="pl-c"><span class="pl-c">#</span>now in HYB format</span>

<h2 dir="auto"><a id="user-content-a-note-about-factorizations" class="anchor" aria-hidden="true" href="#a-note-about-factorizations"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>A note about factorizations</h2>
<p dir="auto">CUSPARSE provides <strong>incomplete</strong> LU and Cholesky factorization. Often, for a sparse matrix, the full LU or Cholesky factorization is much less sparse than the original matrix.
This is a problem if the sparse matrix is very large, since GPU memory is limited. CUSPARSE provides <strong>incomplete</strong> versions of these factorizations, such that <code>A</code> is
<strong>approximatily</strong> equal to <code>L * U</code> or <code>L* L</code>. In particular, the incomplete factorizations have the same sparsity pattern as <code>A</code>, so that they occupy the same amount of GPU
memory. They are preconditioners - we can solve the problem <code>y = A \ x</code> by applying them iteratively. You should not expect <code>ilu0</code> or <code>ic0</code> in CUSPARSE to have matrix elements
equal to those from Julia <code>lufact</code> or <code>cholfact</code>, because the Julia factorizations are complete.</p>

<p dir="auto">Some questions you might have:</p>
<ul dir="auto">
<li>What are the three <code>'O'</code>s for?
<ul dir="auto">
<li>CUSPARSE allows us to use one- or zero-based indexing. Julia uses one-based indexing for arrays, but many other libraries (for instance, C-based libraries) use zero-based. The <code>'O'</code>s tell CUSPARSE that our matrices are one-based. If you had a zero-based matrix from an external library, you can tell CUSPARSE using <code>'Z'</code>.</li>
</ul>
</li>
<li>Should we move <code>alpha</code> and <code>beta</code> to the GPU?
<ul dir="auto">
<li>We do not have to. CUSPARSE can read in scalar parameters like <code>alpha</code> and <code>beta</code> from the host (CPU) memory. You can just pass them to the function and CUSPARSE.jl handles telling the CUDA functions where they are for you. If you have an array, like <code>A</code> and <code>B</code>, you do need to move it to the GPU before CUSPARSE can work on it. Similarly, to see results, if they are in array form, you will need to move them back to the CPU with <code>to_host</code>.</li>
</ul>
</li>
<li>Since <code>d_C</code> is in <code>CSR</code> format, is <code>C</code> the transpose of what we want?
<ul dir="auto">
<li>No. CUSPARSE.jl handles the conversion internally so that the final result is in <code>CSC</code> format for Julia, and <em>not</em> the transpose of the correct answer.</li>
</ul>
</li>
</ul>

<p dir="auto">Moving data between the CPU and GPU memory is very time-intensive. In general, if you only do one operation on the GPU (e.g. one matrix-vector multiplication), the computation is dominated by the time spent copying data. However, if you do many operations with the data you have on the GPU, like doing twenty matrix-vector multiplications, then the GPU can easily beat the CPU. Below you can see some timing tests for the CPU vs the GPU for 20 operations:
<a target="_blank" rel="noopener noreferrer" href="/test/mm.png"><img src="/test/mm.png" alt="matrix matrix multiplication" style="max-width: 100%;"></a>
<a target="_blank" rel="noopener noreferrer" href="/test/mv.png"><img src="/test/mv.png" alt="matrix vector multiplication" style="max-width: 100%;"></a>
<a target="_blank" rel="noopener noreferrer" href="/test/sv.png"><img src="/test/sv.png" alt="matrix vector solve" style="max-width: 100%;"></a></p>
<p dir="auto">The GPU does very well in these tests, but if we only did one operation, the GPU would do as well as or worse than the CPU. It is not worth it to use the GPU if most of your time will be spent copying data around!</p>
