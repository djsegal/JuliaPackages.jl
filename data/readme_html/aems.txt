<p dir="auto">Implements anytime error minimization search (AEMS) solver for POMDPs. This algorithm was originally described in</p>
<blockquote>
<p dir="auto">Ross, St√©phane, and Brahim Chaib-Draa. "AEMS: An Anytime Online Search Algorithm for Approximate Policy Refinement in Large POMDPs." IJCAI. 2007.</p>
</blockquote>
<p dir="auto">Specifically, this solver is AEMS2, which outperforms AEMS1 in nearly all published experiments.</p>

<h1 dir="auto"><a id="user-content-solver-options" class="anchor" aria-hidden="true" href="#solver-options"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Solver Options</h1>
<p dir="auto">The following keyword options are available; for examle, <code>AEMSSolver(max_iterations = 100, max_time = 0.1)</code>.</p>
<ul dir="auto">
<li><code>max_iterations</code> Maximum number of fringe expansions during one action. Defaults to 1000.</li>
<li><code>max_time</code> Maximum time (in seconds) to spend on one action. Defaults to 1 second.</li>
<li><code>updater</code> The updater used to propagate beliefs in the tree. Defaults to a discrete updater.</li>
<li><code>lower_bound</code> Defaults to a fixed-action policy.</li>
<li><code>upper_bound</code> Defaults to a policy generated by FIB.</li>
<li><code>root_manager</code> Determines how the root changes once an action is taken and an observation is received. This process is further described below. Allowed values are <code>:clear</code>, <code>:belief</code>, <code>:user</code>. Defaults to <code>:clear</code>.</li>
<li><code>action_selector</code> Determines if action with best upper or lower bound should be selected after tree expansion. Defaults to <code>:U</code> to select action with best upper bound; use <code>:L</code> to select action with best lower bound.</li>
</ul>
<h1 dir="auto"><a id="user-content-bounds" class="anchor" aria-hidden="true" href="#bounds"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Bounds</h1>
<p dir="auto">The upper and lower bounds must be subtypes of the <code>Policy</code> type and have the <code>value</code> function implemented.
This function is then used to estimate the bound values at new beliefs.</p>
<h1 dir="auto"><a id="user-content-root-management" class="anchor" aria-hidden="true" href="#root-management"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Root Management</h1>
<p dir="auto">Once an action is taken and an observation is received, the root of the search tree becomes the updated belief. However, due to the structure of <code>POMDPs.jl</code>, the planner is not made aware of the resulting observation. Therefore we provide three different ways to manage the root.</p>
<h4 dir="auto"><a id="user-content-1-clearing-the-tree-after-each-action" class="anchor" aria-hidden="true" href="#1-clearing-the-tree-after-each-action"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>1. <strong>Clearing the tree after each action</strong></h4>
<p dir="auto">This solves the root issue by making a new tree with the current belief as the node, but it's probably less efficient to throw away the tree and start from scratch for each call to <code>action</code>.
Tree clearing is the default behavior, but if you want to be explicit you can call <code>AEMSSolver</code> with the option <code>root_manager = :clear</code>.</p>
<h4 dir="auto"><a id="user-content-2-searching-through-child-beliefs-of-previous-root" class="anchor" aria-hidden="true" href="#2-searching-through-child-beliefs-of-previous-root"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>2. <strong>Searching through child beliefs of previous root</strong></h4>
<p dir="auto">The updated belief should be equal to one of the child beliefs of the previous root.
We can set the root to whichever of the children match the updated belief.
However, this requires <code>==</code> to be defined for the belief type you are using.
Further, the beliefs must <em>exactly</em> match (no tolerance for slight numerical differences).
If none of the child nodes match the belief provided to <code>action</code>, an error is thrown (perhaps it should just clear the tree and start over).
Note that multiple child beliefs can be identical.
In this case, the root is set to the first matching belief (perhaps it should select the one which has the tightest bounds).
To use this method, call <code>AEMSSolver</code> with the option <code>root_manager = :belief</code>.</p>
<h4 dir="auto"><a id="user-content-3-the-user-can-provide-the-planner-with-the-action-taken-and-observation-received" class="anchor" aria-hidden="true" href="#3-the-user-can-provide-the-planner-with-the-action-taken-and-observation-received"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>3. <strong>The user can provide the planner with the action taken and observation received.</strong></h4>
<p dir="auto">This option is probably the most efficient, because the tree can be reused and no searching over beliefs must be performed. The planner can then follow the action and observation down the tree to the next belief node.
To provide this information, the user can call <code>update_root(planner, a, o)</code>, where <code>a</code> and <code>o</code> are the action and observtion.
To use this method, call <code>AEMSSolver</code> with the option <code>root_manager = :user</code>.
Note that <code>update_root</code> will throw an error if the <code>root_manager</code> is not set to <code>:user</code>; don't mess with the root unless you specify that it's the user's job.</p>
<p dir="auto">Unfortunately, the user can't personally call <code>update_root</code> after each action/observation pair during simulation.
Nor do existing simulators take care to update the planner (only the belief).
Therefore, <code>AEMS.jl</code> has re-implemented some of the simulators from <code>POMDPToolbox.jl</code> for the <code>AEMSPlanner</code> type (so far only <code>TestSimulator</code>).
The only changes are the following three lines of code after generating a new observation:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="if planner.root_manager == :user
    update_root(planner, a, o)
end"><pre><span class="pl-k">if</span> planner<span class="pl-k">.</span>root_manager <span class="pl-k">==</span> <span class="pl-c1">:user</span>
    <span class="pl-c1">update_root</span>(planner, a, o)
<span class="pl-k">end</span></pre></div>
<p dir="auto">Obviously, this solution is horrifying.
Any simulation that wants to take advantage of the <code>update_root</code> function must be specifically made for the <code>AEMSPlanner</code> type.
While the other root management methods, like belief-searching, don't require any changes to simulations, I'm not sure they are the optimal solution.
It can be expensive if there are many beliefs and each belief is very large.
There is also the issue of beliefs that should be the same but have small numerical differences; these will not be recognized as the same belief and the root will go un-updated.
In the long-term, I think <code>POMDPs.jl</code> should implement a <code>update(policy, a, o)</code> function that defaults to doing nothing for all policies.</p>
