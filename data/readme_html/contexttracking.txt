<h1 dir="auto"><a id="user-content-contexttrackingjl" class="anchor" aria-hidden="true" href="#contexttrackingjl"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>ContextTracking.jl</h1>
<p dir="auto">ContextTracking is used to keep track of execution context.  The context data is kept in a stack data structure.  When a function is called, the context is saved.  When the function exits, the context is restored.  User can make changes to the context during execution, and the data is visible to both the current and deeper stack frames.</p>
<p dir="auto">The usage is embarassingly simple:</p>
<ol dir="auto">
<li>Annotate functions with <code>@ctx</code> macro</li>
<li>Attach context data using the <code>@memo</code> macro</li>
<li>Access context data anywhere using the <code>context</code> function</li>
</ol>

<p dir="auto">Suppose that we are <a href="images/web_service_example.png">processing a web request</a>.  We may want to create a <a href="https://blog.rapid7.com/2016/12/23/the-value-of-correlation-ids/" rel="nofollow">correlation id</a> to keep track of the request and include the request id whenever we write anything to the log file during any part of the processing of that request.</p>
<p dir="auto">It may seems somewhat redundant to log the same data multiple times but it is invaluable in debugging production problems.  Imagine that two users are hitting the same web service at the same time.  If we look at the log file, everything could be interleaving and it would be quite confusing without the context.</p>
<p dir="auto">As context data is stored in a stack structure, you naturally gain more "knowledge" when going deeper into the execution stack. Then, you naturally "forget" about those details when the execution stack unwinds.  With this design, you can just memoize the most valuable knowledge needed in the log file.</p>

<p dir="auto">The <code>context</code> function returns a <code>Context</code> object with the following properties:</p>
<ul dir="auto">
<li><code>id</code>: context id, which is unique per task/thread</li>
<li><code>data</code>: the data being tracked by the context.  By default, it is a <code>Dict</code>.</li>
<li><code>path</code>: the call path, an array of function names as recorded by <code>@ctx</code></li>
<li><code>generations</code>: number of context levels in the stack</li>
<li><code>hex_id</code>: same as <code>id</code>, represented as a hexadecimal string</li>
</ul>

<p dir="auto">By annotating a function with <code>@ctx</code> macro, the function body is wrapped by code that saves and restores context.  Consider the following example:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="@ctx function foo()
    @info &quot;Inside Foo&quot;
end"><pre><span class="pl-c1">@ctx</span> <span class="pl-k">function</span> <span class="pl-en">foo</span>()
    <span class="pl-c1">@info</span> <span class="pl-s"><span class="pl-pds">"</span>Inside Foo<span class="pl-pds">"</span></span>
<span class="pl-k">end</span></pre></div>
<p dir="auto">It would be translated to something like this:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="function foo()
    try
        # &lt;&lt; inserted code to save context &gt;&gt;
        @info &quot;Your code inside Foo&quot;
    finally
        # &lt;&lt; inserted code to restore context &gt;&gt;
    end
end"><pre><span class="pl-k">function</span> <span class="pl-en">foo</span>()
    <span class="pl-k">try</span>
        <span class="pl-c"><span class="pl-c">#</span> &lt;&lt; inserted code to save context &gt;&gt;</span>
        <span class="pl-c1">@info</span> <span class="pl-s"><span class="pl-pds">"</span>Your code inside Foo<span class="pl-pds">"</span></span>
    <span class="pl-k">finally</span>
        <span class="pl-c"><span class="pl-c">#</span> &lt;&lt; inserted code to restore context &gt;&gt;</span>
    <span class="pl-k">end</span>
<span class="pl-k">end</span></pre></div>
<p dir="auto">The purpose of the save/restore operation is to guarantee that context data is visible only during the current execution chain - inside the current function or any subsequent functions being called from here.</p>

<p dir="auto">The <code>@memo</code> macro is used to assign data to the current context.  Consider the following example:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="@memo x = 1"><pre><span class="pl-c1">@memo</span> x <span class="pl-k">=</span> <span class="pl-c1">1</span></pre></div>
<p dir="auto">It would be translated to something like:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="val = (x = 1)
push!(ContextTracking.context(), :x =&gt; val)"><pre>val <span class="pl-k">=</span> (x <span class="pl-k">=</span> <span class="pl-c1">1</span>)
<span class="pl-c1">push!</span>(ContextTracking<span class="pl-k">.</span><span class="pl-c1">context</span>(), <span class="pl-c1">:x</span> <span class="pl-k">=&gt;</span> val)</pre></div>
<p dir="auto">It is highly recommended that you only use <code>@memo</code> in functions that are annotated with <code>@ctx</code> macro.  Failing to do so would leak your data to the parent function's context, which is <em>usually</em> not a desirable effect.</p>
<h2 dir="auto"><a id="user-content-is-it-thread-safe" class="anchor" aria-hidden="true" href="#is-it-thread-safe"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Is it thread-safe?</h2>
<p dir="auto">The <code>context()</code> function always returns a <code>Context</code> object that is unique by thread / async task.
Therefore, the context data is managed properly even when you run your program using multiple
threads or with <code>@async</code>.</p>
<p dir="auto">For example, if you run the program with 4 threads, then <code>context()</code> would return a separate
context when it is called from the individual threads.  Likewise for async tasks.</p>

different.  The only requirement is that the container type must implement the following functions:</p>
