<h2><a id="user-content-the-design" class="anchor" aria-hidden="true" href="#the-design"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>The Design</h2>
<p>This package provides <code>TimeDate</code> to hold the date and time of day given in nanoseconds (or more coarsely).  And provides<code>TimeDateZone</code> to holds the the date and time of day in nanoseconds with a timezone. This work relies heavily on <code>Dates</code> and <code>TimeZones</code>; most of the attention to detail plays through.</p>
<p><code>Dates</code> has a <code>Time</code> type that has nanosecond resolution; it is not well supported, even within <code>Dates</code>.  This <code>Time</code> type recognizes strings only if they are limited to millisecond resolution. Only millisecond resolved times are relevant to <code>DateTime</code>s.  While at present limited by this millisecond barrier, <code>TimeZones</code> is a laudable package with active support.  I expect an eventual melding of what's best.</p>
<p>Here, the inner dynamics rely upon the <code>Period</code> types (<code>Year</code> .. <code>Day</code>, <code>Hour</code>, .., <code>Nanosecond</code>) and <code>CompoundPeriod</code> all provided by <code>Dates</code>.  We distinguish <em>slowtime</em>, which is millisecond resolved, from a nanosecond resolved <em>fasttime</em>.</p>
<p>The general approach is separate the date, slowtime, fasttime, and timezone (if appropriate), then use the date, slowtime and timezone (if appropriate) to obtain a coarse result using the facilities provided by the <code>Date</code> and <code>TimeZones</code> packages.  We refine the coarse result by adding or subtracting the fasttime, as appropriate.</p>
