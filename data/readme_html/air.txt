<p dir="auto">Air is an <a href="https://en.wikipedia.org/wiki/Persistent_data_structure" rel="nofollow">immutable data structure</a>
and <a href="https://en.wikipedia.org/wiki/Software_transactional_memory" rel="nofollow">software transactional memory</a>
tool for Julia.  It provides several persistent data structures including
dictionaries, sets, N-dimensional arrays, heaps, weighted dictionaries, and
weighted sets, and it includes a transaction system that allows one to compose
atomic operations over such data.</p>
<p dir="auto">Air is currently under development but includes substantial testing and is
generally stable.  Inspiration for Air's design is derived largely from
paradigms in <a href="https://en.wikipedia.org/wiki/Clojure" rel="nofollow">Clojure</a>.</p>

<h2 dir="auto"><a id="user-content-plans" class="anchor" aria-hidden="true" href="#plans"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Plans</h2>
<p dir="auto">Note that many of the core components for Air already have working
implementations. Others are currently undergoing testing. In particular, the
existing persistent data structures are fairly well tested and have a
performance comparable to their Clojure counterparts. Additionally, initial
tests of the thread-safe transaction system using the transaction block macro,
<code>Actor</code>s and <code>Volatile</code>s appear to work fine. However, as the author is not an
expert on testing multi-threaded code, some caution is advisable.</p>
<ul dir="auto">
<li>Completed plans:
<ul dir="auto">
<li>Persistent data structures:
<ul dir="auto">
<li><code>PArray</code>, a persistent array type that mimics Julia's native <code>Array</code></li>
<li><code>PDict</code>, a persistent dictionary type that mimics Julia's native <code>Dict</code></li>
<li><code>PSet</code>, a persistent set type.</li>
<li><code>PWSet</code>, a weighted persistent set type. Each element of a <code>PWSet</code> has a
weight, and the set itself acts both as a piority queue (<code>first(pwset)</code>
always yields the element with the highest weight in <code>O(1)</code> time) and as
a distribution (<code>rand(pwset)</code> yields a random element of the set where the
probability of each element is proportional to its weight).</li>
<li><code>PWDict</code>, a persistent weighted dictionary type is like the <code>PWSet</code> type,
but instead of elements with weights, the dictionary contains key–value
pairs with weights.</li>
<li><code>LazyDict</code>, a persistent lazy dictionary type.</li>
</ul>
</li>
<li>Composable multi-threading utilities, inspired by Clojure:
<ul dir="auto">
<li>A <code>Volatile</code> type that operates within transaction blocks to ensure
that all updates to references within a synchronized block are performed
atomically.</li>
<li>An <code>Actor</code> type for sending asynchronous jobs to independent threads which
also respects the atomic requirements of transactional blocks.</li>
<li>Thread-local <code>Var</code> type.</li>
<li>Thread-safe <code>Delay</code> type.</li>
</ul>
</li>
</ul>
</li>
<li>Plans with incomplete testing:
<ul dir="auto">
<li>Thread-safe <code>Promise</code> types.</li>
</ul>
</li>
<li>Plans that are not yet implemented:
<ul dir="auto">
<li><strong>Improved Persistent Array Methods.</strong> Currently, most methods of persistent
arrays yield mutable Julia arrays instead of persistent arrays. This is
largely because the <code>PArray</code> class does not explicitly overload these
functions or the broadcasting functions in order to ensure that it creates
<code>PArray</code> objects. For example, <code>PArray([1,2,3]) .+ 1</code> should yield a
<code>PArray</code>; currently it yields an <code>Array</code>.</li>
<li><strong>Forms.</strong> Arbitrarily-deep nested persistent dictionaries and vectors are
a common data organization paradigm for persistent data. With <code>Volatile</code>
objects embedded in such a data-structure, multi-threaded operations can be
made almost transparent. The <code>Form</code> trait will be a subsystem for such
structures, using the various persistent types.</li>
<li>Better query/build/update API tools for the data structures:
<ul dir="auto">
<li>The <code>setindex</code> and <code>push</code> functions are great, but it would be nice if
nested persistent data structures had single coherent way of updating and
querying them.</li>
</ul>
</li>
<li><strong>Persistent Record Types.</strong> Tool for defining data structures that
support lazy reification of members and updates to their properties in the
style of persistent data structure.</li>
<li><strong>Inheritable Method Parameters.</strong> A common problem in many languages is
that an API function that calls another lower-level function often needs to
support some subset of the argument preprocessing that occurs in the
lower-level function. The preprocessing and documentation of the shared
arguments shouldn't need to be duplicated in multiple places; rather a
simple macro should make it easy for common parameters to occur in both
places and be documented identically in both.</li>
</ul>
</li>
<li>Problems that needs to be worked out:
<ul dir="auto">
<li>It's not clear to me that Julia has entirely settled on or documented the
difference between the <code>@async</code> and the <code>@spawn</code> macros. My suspicion is
that the <code>Actor</code> constructor may need an additional parameter <code>async=false</code>
that can be set to true in order to force threads to be created using
<code>@async</code> instead of <code>@spawn</code>. In Clojure there's a similar distinction
between the <code>send</code> and the <code>send-off</code> functions.</li>
</ul>
</li>
</ul>
