<p dir="auto">This package contains tools for visualizing and interacting with profiling data collected
with <a href="http://julialang.org" title="Julia" rel="nofollow">Julia's</a> built-in sampling
<a href="https://docs.julialang.org/en/v1/manual/profile/" rel="nofollow">profiler</a>. It
can be helpful for getting a big-picture overview of the major
bottlenecks in your code, and optionally highlights lines that trigger
garbage collection as potential candidates for optimization.</p>
<p dir="auto">This type of plot is known as a <a href="https://github.com/brendangregg/FlameGraph">flame
graph</a>.
The main logic is handled by the <a href="https://github.com/timholy/FlameGraphs.jl">FlameGraphs</a> package; this package is just a visualization front-end.</p>
<p dir="auto">Compared to other flamegraph viewers, ProfileView adds interactivity features, such as:</p>
<ul dir="auto">
<li>zoom, pan for exploring large flamegraphs</li>
<li>right-clicking to take you to the source code for a particular statement</li>
<li>analyzing inference problems via <code>code_warntype</code> for specific, user-selected calls</li>
</ul>
<p dir="auto">These features are described in detail below.</p>

<p dir="auto"><code>@profview f(args...)</code> is just shorthand for <code>Profile.clear(); @profile f(args...); ProfileView.view()</code>.
(These commands require that you first say <code>using Profile</code>, the Julia profiling standard library.)</p>
<blockquote>
<p dir="auto">If you use ProfileView from VSCode you'll get an error <code>UndefVarError: @profview not defined</code>.
This is because VSCode defines its own <code>@profview</code>, which conflicts with ProfileView's.
Fix it by using <code>ProfileView.@profview</code>.</p>
</blockquote>
<p dir="auto">If you're following along, you may see something like this:</p>
<p dir="auto"><a target="_blank" rel="noopener noreferrer" href="readme_images/pv1.png"><img src="readme_images/pv1.png" alt="ProfileView" style="max-width: 100%;"></a></p>
<p dir="auto">(Note that collected profiles can vary by Julia version and from run-to-run, so don't be alarmed
if you get something different.)
This plot is a visual representation of the <em>call graph</em> of the code that you just profiled.
The "root" of the tree is at the bottom; if you move your mouse along the long horizontal
bar at the bottom, you'll see a tooltip that's something like</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="boot.jl, eval: 330"><pre class="notranslate"><code>boot.jl, eval: 330
</code></pre></div>
<p dir="auto">This refers to one of Julia's own source files, <a href="https://github.com/JuliaLang/julia/blob/2e6715c045042e1c8ae9adc7a578340649b0ad5a/base/boot.jl#L330">base/boot.jl</a>.
<code>eval</code> is the name of the function being executed, and <code>330</code> is the line number of the file.
This is the function that evaluated your <code>profile_test(10)</code> command that you typed at the REPL.
(Indeed, to reduce the amount of internal "overhead" in the flamegraph, some of these internals are truncated; see the <code>norepl</code> option of <code>FlameGraphs.flamegraph</code>.)
If you move your mouse upwards, you'll then see bars corresponding to the function(s) you ran with <code>@profview</code> (in this case, <code>profile_test</code>).
Thus, the vertical axis represents nesting depth: bars lie on top of the bars that called them.</p>
<p dir="auto">The horizontal axis represents the amount of time (more precisely, the
number of backtraces) spent at each line.  The row at which the single
long bar breaks up into multiple different-colored bars corresponds
to the execution of different lines from <code>profile_test</code>.
The fact that
they are all positioned on top of the lower peach-colored bar means that all
of these lines are called by the same "parent" function. Within a
block of code, they are sorted in order of increasing line number, to
make it easier for you to compare to the source code.</p>
<p dir="auto">From this visual representation, we can very quickly learn several
things about this function:</p>
<ul dir="auto">
<li>
<p dir="auto">On the right side, you see a stack of calls to functions in <code>sort.jl</code>.
This is because sorting is implemented using recursion (functions that call themselves).</p>
</li>
<li>
<p dir="auto"><code>mapslices(sum, A; dims=2)</code> is considerably more expensive (the corresponding bar is horizontally wider) than
<code>mapslices(sort, B; dims=1)</code>. This is because it has to process more
data.</p>
</li>
</ul>
<p dir="auto">It is also worth noting that red is (by default) a special color: it is reserved for function
calls that have to be resolved at run-time.
Because run-time dispatch (aka, dynamic dispatch, run-time method lookup, or
a virtual call) often has a significant
impact on performance, ProfileView highlights the problematic call in red. It's
worth noting that some red is unavoidable; for example, the REPL can't
predict in advance the return types from what users type at the
prompt, and so the bottom <code>eval</code> call is red.
Red bars are problematic only when they account for a sizable
fraction of the top of a call stack, as only in such cases are they likely to be
the source of a significant performance bottleneck.
In the image above, can see that <code>mapslices</code> relied on run-time dispatch;
from the absence of pastel-colored bars above much of the red, we
might guess that this made a substantial
contribution to its total run time.
(Your version of Julia may show different results.)
See <a href="solving-type-inference-problems">Solving type-inference problems</a> below
for tips on how to efficiently diagnose the nature of the problem.</p>
<p dir="auto">Yellow is also a special color: it indicates a site of garbage collection, which can be
triggered at a site of memory allocation. You may find that such bars lead you to lines
whose performance can be improved by reducing the amount of temporary memory allocated
by your program. One common example is to consider using <code>@views(A[:, i] .* v)</code> instead
of <code>A[:, i] .* v</code>; the latter creates a new column-vector from <code>A</code>, whereas the former
just creates a reference to it.
Julia's <a href="https://docs.julialang.org/en/v1/stdlib/Profile/#Memory-profiling" rel="nofollow">memory profiler</a>
may provide much more information about the usage of memory in your program.</p>

<p dir="auto">Some default settings can be changed and retained across settings through a
<code>LocalPreferences.toml</code> file that is added to the active environment.</p>
<ul dir="auto">
<li>
<p dir="auto">Default color theme: The default is <code>:light</code>.
Alternatively <code>:dark</code> can be set.
Use <code>ProfileView.set_theme!(:dark)</code> to change the default.</p>
</li>
<li>
<p dir="auto">Default graph type: The default is <code>:flame</code> which displays from the bottom up.
Alternatively <code>:icicle</code> displays from the top down.
Use <code>ProfileView.set_graphtype!(:icicle)</code> to change the default.</p>
</li>
</ul>
<h3 dir="auto"><a id="user-content-gtk-interface" class="anchor" aria-hidden="true" href="#gtk-interface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Gtk Interface</h3>
<ul dir="auto">
<li>
<p dir="auto">Ctrl-q and Ctrl-w close the window. You can also use
<code>ProfileView.closeall()</code> to close all windows opened by ProfileView.</p>
</li>
<li>
<p dir="auto">Left-clicking on a bar will cause information about this line to be
printed in the REPL. This can be a convenient way to "mark" lines
for later investigation.</p>
</li>
<li>
<p dir="auto">Right-clicking on a bar calls the <code>edit()</code> function to open the line
in an editor.  (On a trackpad, use a 2-fingered tap.)</p>
</li>
<li>
<p dir="auto">CTRL-clicking and dragging will zoom in on a specific region of the image.  You
can also control the zoom level with CTRL-scroll (or CTRL-swipe up/down).</p>
<p dir="auto">CTRL-double-click to restore the full view.</p>
</li>
<li>
<p dir="auto">You can pan the view by clicking and dragging, or by scrolling your
mouse/trackpad (scroll=vertical, SHIFT-scroll=horizontal).</p>
</li>
<li>
<p dir="auto">The toolbar at the top contains two icons to load and save profile
data, respectively.  Clicking the save icon will prompt you for a
filename; you should use extension <code>*.jlprof</code> for any file you save.
Launching <code>ProfileView.view(nothing)</code> opens a blank
window, which you can populate with saved data by clicking on the
"open" icon.</p>
</li>
<li>
<p dir="auto">After clicking on a bar, you can type <code>warntype_last</code> and see the
result of <code>code_warntype</code> for the call represented by that bar.</p>
</li>
<li>
<p dir="auto"><code>ProfileView.view(windowname="method1")</code> allows you to name your window,
which can help avoid confusion when opening several ProfileView windows
simultaneously.</p>
</li>
<li>
<p dir="auto">On Julia 1.8 ProfileView.view(expand_tasks=true) creates one tab per task.
Expanding by thread is on by default and can be disabled with <code>expand_threads=false</code>.</p>
</li>
</ul>
<p dir="auto"><strong>NOTE</strong>: ProfileView does not support the old JLD-based <code>*.jlprof</code> files anymore.
Use the format provided by FlameGraphs v0.2 and higher.</p>

<p dir="auto">The <code>view</code> command has the following syntax:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="function view([fcolor,] data = Profile.fetch(); lidict = nothing, C = false, fontsize = 14, kwargs...)"><pre class="notranslate"><code>function view([fcolor,] data = Profile.fetch(); lidict = nothing, C = false, fontsize = 14, kwargs...)
</code></pre></div>
<p dir="auto">Here is the meaning of the different arguments:</p>
<ul dir="auto">
<li>
<p dir="auto"><code>fcolor</code> optionally allows you to control the scheme used to select
bar color. This can be quite extensively customized; see <a href="https://timholy.github.io/FlameGraphs.jl/stable/" rel="nofollow">FlameGraphs</a> for details.</p>
</li>
<li>
<p dir="auto"><code>data</code> is the vector containing backtraces. You can use <code>using Profile; data1 = copy(Profile.fetch()); Profile.clear()</code> to store and examine results
from multiple profile runs simultaneously.</p>
</li>
<li>
<p dir="auto"><code>lidict</code> is a dictionary containing "line information." This is obtained together with
<code>data</code> from <code>using Profile; data, lidict = Profile.retrieve()</code>. Computing <code>lidict</code> is
the slow step in displaying profile data, so calling <code>retrieve</code> can speed up repeated
visualizations of the same data.</p>
</li>
<li>
<p dir="auto"><code>C</code> is a flag controlling whether lines corresponding to C and Fortran
code are displayed. (Internally, ProfileView uses the information
from C backtraces to learn about garbage-collection and to
disambiguate the call graph).</p>
</li>
<li>
<p dir="auto"><code>fontsize</code> controls the size of the font displayed as a tooltip.</p>
</li>
<li>
<p dir="auto"><code>expand_threads</code> controls whether a page is created for each thread (requires julia 1.8, enabled by default)</p>
</li>
<li>
<p dir="auto"><code>expand_tasks</code> controls whether a page is shown for each task (requires julia 1.8, off by default)</p>
</li>
<li>
<p dir="auto"><code>graphtype::Symbol = :default</code> controls how the graph is shown. <code>:flame</code> displays from the bottom up, <code>:icicle</code>
displays from the top down. The default is <code>:flame</code> which can be changed via e.g. <code>ProfileView.set_graphtype!(:icicle)</code>.</p>
</li>
</ul>
<p dir="auto">These are the main options, but there are others; see FlameGraphs for more details.</p>

<p dir="auto">Profiling and <a href="https://github.com/timholy/Revise.jl">Revise</a> are natural partners,
as together they allow you to iteratively improve the performance of your code.
If you use Revise and are tracking the source files (either as a package or with <code>includet</code>),
the source locations (file and line number) reported by ProfileView
will match the current code at the time the window is created.</p>
