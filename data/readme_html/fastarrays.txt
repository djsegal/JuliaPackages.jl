<p dir="auto">Fast multi-dimensional arrays, with arbitrary lower and upper bounds that can be fixed at compile time to improve efficiency</p>
<h2 dir="auto"><a id="user-content-background" class="anchor" aria-hidden="true" href="#background"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Background</h2>
<p dir="auto">Sometimes you really want arrays where the lower index bound is different from 1. This is not a question of performance, but of convenience -- for example, if you have quantum numbers ranging from <code>0</code> to <code>k</code>, then adding <code>1</code> every time you index an array looks tedious. This really should be part of the type declaration.</p>
<p dir="auto">And sometimes, you know the size of a particular array dimension ahead of time. This is now a question of efficiency -- indexing into a multi-dimensional array is significantly more efficient if the sizes of the dimensions are known ahead of time.</p>
<p dir="auto">This is just what this package <code>FastArrays</code> provides: A way to define multi-dimensional array types where both lower and upper index bounds can be chosen freely, and which generates more efficient code if these bounds are known ahead of time.</p>
<p dir="auto">Here is an example:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="using FastArrays

<p dir="auto">Fast arrays are accessed like regular arrays, using the <code>[]</code> notation or the <code>getindex</code> and <code>setindex!</code> functions.</p>
<p dir="auto">You will have noticed the slightly unusual notation for fast arrays. Implementation-wise, the set of all bounds that are kept fixed determine the (parameterized) type of the array; different choices for fixed array bounds correspond to different types. <code>FastArray</code> is a function that returns the respective type, creating this type if necessary.</p>
<p dir="auto">Currently, fast arrays do not yet support resizing, reshaping, or subarrays; adding this would be straightforward.</p>
<p dir="auto">I designed FastArrays for the two reasons above -- I needed to model quantum numbers that have a range of <code>0:k</code>, and I noticed that the C++ version of the respecived code became significantly faster when setting array sizes at compile time. In Julia, I confirmed that the generated machine code is also much simpler in this case. Of course, this provides a benefit only if array accessing is actually a bottleneck in the code.</p>
<h2 dir="auto"><a id="user-content-manual" class="anchor" aria-hidden="true" href="#manual"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Manual</h2>
<p dir="auto">Each array dimension has a lower and an upper bound. Both can either be fixed or flexible. Fixed bounds are the same for all objects of this type, flexible bounds have to be chosen when the array is allocated. Julia's <code>Array</code> types correspond to <code>FastArray</code> types where all lower bounds are fixed to <code>1</code>, and all upper bounds are flexible.</p>
<p dir="auto">Internally, the fixed bounds are represented as a tuple of either nothing or integers: <code>DimSpec = NTuple{2, Union{Void, Int}}</code>.</p>
<p dir="auto">For each dimension, the fixed bounds are set via:</p>
<ul dir="auto">
<li>A range <code>lb:ub</code> to define both bounds</li>
<li>An integer <code>lb</code> to define the lower bound, leaving the upper bound flexible</li>
<li>A colon <code>:</code> to indicate that both lower and upper bounds are flexible
Instead of the above, you can also define the fixed bounds via a tuple of type <code>DimSpec</code>.</li>
</ul>
<p dir="auto">When an array is allocated, the flexible bounds are set conversely:</p>
<ul dir="auto">
<li>A colon <code>:</code> indicates no flexible bounds (if both bounds are fixed)</li>
<li>An integer <code>ub</code> defines the upper bound (if the lower bound is fixed)</li>
<li>A range <code>lb:ub</code> defines both lower and upper bounds</li>
<li>A one-element integer tuple <code>(lb,)</code> defines the lower bound (if the upper bound is fixed)</li>
</ul>
<p dir="auto">Each fast array type is subtype of <code>AbstractFastArray{T,N}</code>, where <code>T</code> is the element type and <code>N</code> is the rank. This abstract is a subtype of <code>DenseArray{T,N}</code>.</p>
