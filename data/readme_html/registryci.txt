<p>RegistryCI provides continuous integration (CI) tools, including automated testing and automatic merging (automerge) of pull requests.</p>
<p>The <a href="https://github.com/JuliaRegistries/General">General</a> registry uses RegistryCI.</p>
<p>You can also use RegistryCI for your own Julia package registry.</p>
<h2><a id="user-content-automatic-merging-guidelines" class="anchor" aria-hidden="true" href="#automatic-merging-guidelines"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Automatic merging guidelines</h2>
<p>These guidelines are intended not as requirements for packages but as very conservative guidelines, which, if your new package or new version of a package meets them, it may be automatically merged.</p>
<p>Note that commenting on a pull request will automatically disable automerging on that pull request. Therefore, if you want to leave a comment on a pull request but you still want that pull request to be automerged, please include the text <code>[noblock]</code> in your comment.</p>
<h3><a id="user-content-new-package" class="anchor" aria-hidden="true" href="#new-package"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>New package</h3>
<ol>
<li>
<p>Normal capitalization</p>
<p>The package name should match <code>r"^[A-Z]\w*[a-z]\w*[0-9]?$"</code>, i.e. start with a capital letter, contain ASCII alphanumerics only, contain at 1 lowercase letter.</p>
</li>
<li>
<p>Not too short</p>
<p>At least five letters. <em>You can register names shorter than this, but doing so requires someone to approve.</em></p>
</li>
<li>
<p>Standard initial version number: one of <code>0.0.1</code>, <code>0.1.0</code>, <code>1.0.0</code>.</p>
</li>
<li>
<p>Repo URL ends with <code>/$name.jl.git</code> where <code>name</code> is the package name.</p>
</li>
</ol>
<h3><a id="user-content-new-version-of-an-existing-package" class="anchor" aria-hidden="true" href="#new-version-of-an-existing-package"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>New version of an existing package</h3>
<ol>
<li>
<p>Sequential version number</p>
<p>If the last version was <code>1.2.3</code> then the next can be <code>1.2.4</code>, <code>1.3.0</code> or <code>2.0.0</code>.</p>
</li>
<li>
<p><a href="https://julialang.github.io/Pkg.jl/v1/compatibility/" rel="nofollow">Compat entries</a> for all dependencies.</p>
<ul>
<li>all <code>[deps]</code> should also have <code>[compat]</code> entries (and Julia itself)</li>
<li><code>[compat]</code> entries should have upper bounds</li>
</ul>
<p>Compat entries are not required for standard libraries. For the time being, compat entries are not required for JLL dependencies because they often have non-standard version numbering schemes; however, this may change in the future.</p>
</li>
<li>
<p>Version can be installed</p>
<p>Given the proposed changes to the registry, can we resolve and install the new version of the package?</p>
</li>
<li>
<p>Version can be loaded</p>
<p>Once it's been installed (and built?), can we load the code?</p>
</li>
</ol>
