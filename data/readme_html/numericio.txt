<p dir="auto">Improved support for formatting numeric data.</p>
<ul dir="auto">
<li>Includes facilities to display values using SI prefixes (<code>Y</code>, <code>Z</code>, <code>E</code>, <code>P</code>, <code>T</code>, <code>G</code>, <code>M</code>, <code>k</code>, <code>m</code>, μ, <code>n</code>, <code>p</code>, <code>f</code>, <code>a</code>, <code>z</code>, <code>y</code>)</li>
<li>Makes it easy to control the number of significant digits to display.</li>
</ul>

<p dir="auto">NumericIO.jl tries to provide the convenience of c++ <code>ios_base</code> configurability (ex: setting <code>ios_base::precision</code>) <em>without</em> modifying the output format of the base streaming object. Instead, NumericIO.jl uses the <code>FormattedIO</code> wrapper object to print data with the desired output format.  This appears to be somewhat similar (but not as flexible) as the <code>Base.IOContext</code> object used in core Julia.</p>
<p dir="auto">To obtain a string representation of a <code>Real</code> value using SI prefixes, one can use the <code>formatted</code> function:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="formatted(3.14159e-8, :SI, ndigits=3) # =&gt; &quot;31.14n&quot;"><pre class="notranslate"><code>formatted(3.14159e-8, :SI, ndigits=3) # =&gt; "31.14n"
</code></pre></div>
<p dir="auto">Similarly, one can generate a string using scientific notation with the following:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="formatted(3.14159e-8, :SCI, ndigits=3) # =&gt; &quot;3.14×10⁻⁸&quot;"><pre class="notranslate"><code>formatted(3.14159e-8, :SCI, ndigits=3) # =&gt; "3.14×10⁻⁸"
</code></pre></div>
<p dir="auto">Or using engineering notation (limiting to powers divisible by 3) with:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="formatted(3.14159e-8, :ENG, ndigits=3) # =&gt; &quot;31.4×10⁻⁹&quot;"><pre class="notranslate"><code>formatted(3.14159e-8, :ENG, ndigits=3) # =&gt; "31.4×10⁻⁹"
</code></pre></div>
<p dir="auto">To limit results to <code>ASCII</code> output, specify the <code>charset</code> keyword:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="formatted(3.14159e-8, :ENG, ndigits=3, charset=:ASCII) # =&gt; &quot;31.4E-9&quot;"><pre class="notranslate"><code>formatted(3.14159e-8, :ENG, ndigits=3, charset=:ASCII) # =&gt; "31.4E-9"
</code></pre></div>
<p dir="auto">One might also prefer to create a convenience formatting function:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="SI(x) = formatted(x, :SI, ndigits=4)

<p dir="auto">To print out multiple values, it is preferable (more efficient) to directly create a FormattedIO wrapper object:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="fio = formatted(STDOUT, :SI, ndigits=4) # =&gt; FormattedIO

..."><pre class="notranslate"><code>fio = formatted(STDOUT, :SI, ndigits=4) # =&gt; FormattedIO

<p dir="auto">Lower-level structures of NumericIO can be used to fine-tune numeric output even further, if desired.  The following shows an example that approximates engineering notation using the <code>ASCII</code> characterset only:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="asciiexponentfmt = NumericIO.IOFormattingExpNum(
	&quot;x10^&quot;, false, '+', '-', NumericIO.ASCII_SUPERSCRIPT_NUMERALS
)
fmt = NumericIO.IOFormattingReal(asciiexponentfmt,
	ndigits=4, decpos=0, decfloating=true, eng=true, minus='-', inf=&quot;Inf&quot;
)
fio = formatted(STDOUT, fmt) # =&gt; FormattedIO
println(fio, 3.14159e-8) # =&gt; 31.42x10^-9"><pre class="notranslate"><code>asciiexponentfmt = NumericIO.IOFormattingExpNum(
	"x10^", false, '+', '-', NumericIO.ASCII_SUPERSCRIPT_NUMERALS
)
fmt = NumericIO.IOFormattingReal(asciiexponentfmt,
	ndigits=4, decpos=0, decfloating=true, eng=true, minus='-', inf="Inf"
)
fio = formatted(STDOUT, fmt) # =&gt; FormattedIO
println(fio, 3.14159e-8) # =&gt; 31.42x10^-9
</code></pre></div>
<p dir="auto">It is also possible to generate the mantissa &amp; exponent portions of a number separately.  This could be useful when displaying a plot's tick labels when using a common axis multiplier.  See implementation of <code>NumericIO.print_formatted(..., showexp=false)</code> and <code>NumericIO.print_formatted_exp(...)</code> for details.</p>

<p dir="auto">Users can make REPL outputs easier to read, simply by adding a few lines to their <code>~/.julia/config/startup.jl</code> file:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="import REPL

#Change how Float32/64s are displayed on `REPLDisplay`:
Base.display(r::REPL.REPLDisplay, v::Union{Float32,Float64}) = print(formatted(REPL.outstream(r.repl), :SI, ndigits=4), v)

#Change how `Vectors` are displayed on `REPLDisplay`:
function Base.display(r::REPL.REPLDisplay, v::Vector)
	fio = formatted(REPL.outstream(r.repl), :SI, ndigits=4)
	N = length(v)
	println(REPL.outstream(r.repl), N, &quot;-element &quot;, typeof(v), &quot;:&quot;)
	for v_i in v
		println(fio, v_i) #Use formatted output
	end
end"><pre class="notranslate"><code>import REPL
using NumericIO

<p dir="auto">This solution should be fairly safe: few programmers writing to REPL displays would later read back values from said display (failing the subsequent read operation).  The most likely issue with this application is that programmers expecting carefully formatted output would now have suboptimal-looking output.</p>

<p dir="auto">Note that defining the above <code>display</code> method does not alter how <code>Array{Float}</code> arrays are displayed.  In order to change this particular output, one would have to define something like:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="Base.show{T&lt;:Base.AbstractIOBuffer}(ctx::IOContext{T}, v::Union{Float32,Float64}) = print(formatted(ctx.io, :SI, ndigits=4), v)"><pre class="notranslate"><code>Base.show{T&lt;:Base.AbstractIOBuffer}(ctx::IOContext{T}, v::Union{Float32,Float64}) = print(formatted(ctx.io, :SI, ndigits=4), v)
</code></pre></div>
<p dir="auto">... because currently (Julia v0.5.0) arrays generate their outputs by writing to temporary <code>IOBuffer</code>s.</p>
<p dir="auto"><strong>This solution is not recommended</strong>: it involves altering the behaviour of relatively low-level functions.  It is likely to cause issues.</p>

<p dir="auto">Again, the <code>display</code> function described above does not work in conjuction with SIUnits.jl - because the displayed object is of type <code>SIQuantity</code> (not <code>Float32/64</code>).  One could, instead define the following function:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="Base.show(io::IOContext{Base.Terminals.TTYTerminal}, v::Union{Float32,Float64}) = print(formatted(io, :SI, ndigits=4), v)"><pre class="notranslate"><code>Base.show(io::IOContext{Base.Terminals.TTYTerminal}, v::Union{Float32,Float64}) = print(formatted(io, :SI, ndigits=4), v)
</code></pre></div>
<p dir="auto">Which would work for formatting outputs of <code>SIQuantity</code> values, in addition to many (but not all) other values written to terminal.</p>
<p dir="auto">Note however, that <code>show()</code> is a lower level function, and this definition is more likely to cause undesired behaviours.</p>

<ul dir="auto">
<li>Support for SI notation is limited between <code>y (1e-24)</code> and <code>Y (1e24)</code>.  Values beyond this range default to scientific notation.</li>
<li>SI notation displays <code>{1e9, 10e9, 100e9}</code> as <code>{1G, 10G, 100G}</code>.  It would be possible to reconfigure NumericIO to arbitrarily display <code>{1G, 10G, 0.1T}</code>, or <code>{1G, 0.01T, 0.1G}</code>, or ...</li>
<li>Algorithms are likely be a bit more complicated than absolutely necessary.  It would be nice to simplify/optimize the code as much as possible.</li>
<li>Does not support arbitrary bases (ex: <code>1x2^8</code>).</li>
</ul>
