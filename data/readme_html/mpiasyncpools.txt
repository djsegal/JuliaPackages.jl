<p>This package is for implementing straggler-resilient distributed algorithms and methods, e.g., <a href="https://en.wikipedia.org/wiki/Stochastic_gradient_descent" rel="nofollow">stochastic gradient descent</a>, in systems composed of a coordinator process and multiple worker processes. Stragglers are workers that experience bursts of high latency, e.g., due to network congestion. These workers may, e.g., significantly reduce the rate of convergence of distributed optimization methods, unless the implementation is resilient against stragglers.</p>
<p>To initiate communication, one creates an <code>MPIAsyncPool</code> at the coordinator, which handles connecting to the workers via <code>MPI.Irecv!</code> and <code>MPI.Isend</code>. For example:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="pool = MPIAsyncPool([1, 2, 4]) # pool consisting of nodes with MPI ranks 1, 2, 4
pool = MPIAsyncPool(4)         # pool consisting of nodes with MPI ranks 1, 2, 3, 4
"><pre>pool <span class="pl-k">=</span> <span class="pl-c1">MPIAsyncPool</span>([<span class="pl-c1">1</span>, <span class="pl-c1">2</span>, <span class="pl-c1">4</span>]) <span class="pl-c"><span class="pl-c">#</span> pool consisting of nodes with MPI ranks 1, 2, 4</span>
pool <span class="pl-k">=</span> <span class="pl-c1">MPIAsyncPool</span>(<span class="pl-c1">4</span>)         <span class="pl-c"><span class="pl-c">#</span> pool consisting of nodes with MPI ranks 1, 2, 3, 4</span></pre></div>
<p>Next, the <code>asyncmap!</code> function is used to broadcast data, e.g., an iterate (in the case of gradient descent), to the workers and to collect responses. <code>asyncmap!</code> returns once results have been received from the <code>nwait</code> fastest workers. Alternatively, one can define a custom condition, e.g., to always wait for worker 1.</p>
<p>The workers communicate using regular <code>MPI.Irecv!</code> and <code>MPI.Isend</code>; see the <a href="https://github.com/JuliaParallel/MPI.jl">MPI.jl</a> documentation, and the <a href="./examples/">examples directory</a> for a complete example of how to use <code>asyncmap!</code>.</p>
<p>The docstring of <code>asyncmap!</code> is:</p>
<blockquote>
<p><code>asyncmap!(pool::MPIAsyncPool, sendbuf::AbstractArray, recvbuf::AbstractArray, isendbuf::AbstractArray, irecvbuf::AbstractArray, comm::MPI.Comm; nwait::Union{&lt;:Integer,Function}=pool.nwait, epoch::Integer=pool.epoch+1, tag::Integer=0)</code></p>
<p>Send the data in <code>sendbuf</code> asynchronously (via <code>MPI.Isend</code>) to all workers and wait for some of them to respond (via a corresponding <code>MPI.Isend</code>). If <code>nwait</code> is an integer, this function returns  when at least <code>nwait</code> workers have responded. On the other hand, if <code>nwait</code> is a function, this function returns once <code>nwait(epoch, repochs)</code> evaluates to <code>true</code>. The second argument <code>repochs</code> is a vector of length equal to the number of workers in the pool, where the <code>i</code>-th element is the <code>epoch</code> that transmission to the corresponding worker was initiated at.</p>
<p>Returns the <code>repochs</code> vector. Similarly to the <code>MPI.Gather!</code> function, the elements of <code>recvbuf</code> are partitioned into a number of chunks equal to the number of workers (hence, the length of <code>recvbuf</code> must be divisible by the number of workers) and the data recieved from the <code>j</code>-th worker is stored in the <code>j</code>-th partition.</p>
<p>The buffers <code>isendbuf</code> and <code>irecvbuf</code> are for internal use by this function only and should never be changed or accessed outside of it. The length of <code>isendbuf</code> must be equal to the length of <code>sendbuf</code> multiplied by the number of workers, and <code>irecvbuf</code> must have length equal to that of <code>recvbuf</code>.</p>
</blockquote>
