<div id="readme" class="rst" data-path="README.rst"><article class="markdown-body entry-content container-lg" itemprop="text"><h1><a id="user-content-unroll-macro" class="anchor" aria-hidden="true" href="#unroll-macro"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Unroll macro</h1>
<p>This package provides the <code>unroll</code> and <code>tuplegen</code> macros.
The <code>unroll</code> macro in Julia unrolls simple for-loops.  For example,
the following code:</p>
<pre>@unroll for i = 1 : 2
   x += a[i]
end
</pre>
<p>will macro-expand as:</p>
<pre>x += a[1]
x += a[2]
</pre>
<p>For this to be possible, the loop bounds must be known at the time
of macro-expansion.  A common case is that they are literal
constant values as in
the above example.
The loop bounds may include symbolic constants that are global
within the module:</p>
<pre>const LOOPBOUND = 2
function myfunct()
   @unroll for i = 1 : LOOPBOUND
      &lt;etc&gt;
   end
end
</pre>
<p>The <code>unroll</code> macro can be nested.</p>
<p>Finally, the <code>unroll</code> macro will search for conditionals that
depend on the loop counter and unroll these as well.  For example,
the call:</p>
<pre>@unroll for i = 1 : 4
    if mod(i,2) == 1
        a += b[i]
    else
        a += 2*b[i]
    end
 end
</pre>
<p>will macro-expand to:</p>
<pre>a += b[1]
a += 2*b[2]
a += b[3]
a += 2*b[4]
</pre>

<h2><a id="user-content-the-tuplegen-macro" class="anchor" aria-hidden="true" href="#the-tuplegen-macro"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>The tuplegen macro</h2>
<p>The <code>tuplegen</code> macro generates fixed-length tuples using comprehension-like
syntax.  For example:</p>
<pre>v = @tuplegen [(i==2)? i * 6 : i for i = 1 : 4]
</pre>
<p>macro-expands to:</p>
<pre>v = (1, 2*6, 3, 4)
</pre>
<p>and therefore generates the tuple <code>(1,12,3,4)</code>.
Without the <code>@tuplegen</code> call, this
same statement would generate the array <code>[1,12,3,4]</code>.  It is possible
generate tuples from comprehensions
via the following standard
statement:</p>
<pre>v = tuple([(i==2)? i * 6 : i for i = 1 : 4]...)
</pre>
<p>but this statement is less efficient because it creates a heap-allocated
array as a temporary.</p>
<p>Here is a more complicated example of <code>@tuplegen</code>.  Suppose 2-by-2
matrices are represented as 2-tuples of 2-tuples, e.g., <code>((1,2),(5,7))</code>
stands for:</p>
<pre>1  2
5  7
</pre>
<p>Then 2-by-2 matrix multiplication may be defined by:</p>
<pre>mtxmult(a,b) = @tuplegen [@tuplegen [a[i][1]*b[1][j] + a[i][2]*b[2][j]
                                     for j = 1 : 2]
                          for i = 1 : 2]
</pre>
<p>This definition generates unrolls into four expressions on the right-hand side and
works as expected:</p>
<pre>julia&gt; mtxmult(((1,2),(5,7)),((4,1),(2,8)))
((8,17),(34,61))
</pre>
<p>As with the <code>@unroll</code>
macro, the loop bounds for <code>@tuplegen</code>
must be known at macro-expansion time.
In particular, the following plausible attempt to define a generic function for
addition of arbitrary
fixed-length tuples (so that <code>(1,7)+(-2,3)</code> yields <code>(-1,10)</code>) does not work:</p>
<pre>+{N}(a::NTuple{N}, b::NTuple{N}) = @tuplegen [a[i]+b[i] for i=1:N]
</pre>
<p>because the type parameter <code>N</code> is not known at the time of macro expansion; instead
it is determined later by the dispatch mechanism.  If someone knows how to fix
this, please create an issue or pull request.  (It is possible to write generic
addition for tuples via the more complicated generated-function mechanism.)</p>

<h2><a id="user-content-cautionary-notes" class="anchor" aria-hidden="true" href="#cautionary-notes"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Cautionary notes</h2>
<blockquote>
<ul>
<li>The
loop index must be a simple variable (e.g., the loop cannot be
of the form <code>for (k,v) in mydict</code> or something similar).</li>
<li>The loop index is matched symbolically by the macro.  This
means that the same symbol may not be used in the loop body
with a different meaning (e.g., qualified by a different module name).</li>
<li>If the loop index is somehow concealed inside the loop body,
say with an eval/parse statement, then the macro will fail.</li>
<li>The macro calls <code>eval</code> to obtain the loop bounds and also
to check whether <code>if</code> conditions are satisfied.  This means
that the loop should not include statements with side effects
(like <code>print</code>) in either the loop bounds or in conditionals,
since these statements may get unexpectedly executed during
the macro expansion phase.</li>
</ul>
</blockquote>
