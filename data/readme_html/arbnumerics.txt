<p dir="auto">For multiprecision numerical computing using values with 25..2,500 digits. With arithmetic and higher level mathematics, this package offers you the best balance of performance and accuracy.</p>
<p dir="auto">This package uses the <a href="http://arblib.org/index.html" rel="nofollow">Arb C Library</a>, and adapts some C library interface work from <a href="https://github.com/wbhart/Nemo.jl">Nemo</a> (see <a href="https://github.com/JeffreySarnoff/ArbNumerics.jl/blob/master/README.md#acknowledgements"><em>below</em></a>). Here is a presentation by the designer and architect of the <a href="https://fredrikj.net/math/oxford2019.pdf" rel="nofollow">Arb C library</a>.</p>

<p dir="auto">ArbNumerics exports three types: <code>ArbFloat</code>, <code>ArbReal</code>, <code>ArbComplex</code>.  <code>ArbFloat</code> is an extended precision floating point type. Math using <code>ArbFloat</code> is expected to be very near the veridical value, and often is the closest value for the precision in use. <code>ArbReal</code> is an interval-valued quantity formed of an <code>ArbFloat</code> (the midpoint) and a <code>radius</code>.  Math functions with <code>ArbReal</code> are assured to enclose the veridical value.  This assurance extends to multiple function applications.  <code>ArbComplex</code> is an <code>ArbReal</code> pair (real, imaginary).  The same enclosure assurance applies.</p>
<p dir="auto">While the bounds of an <code>ArbReal</code> or <code>ArbComplex</code> are available, the default is to show these values as digit sequences which almost assuredly are accurate, in a round to nearest sense, to the precision displayed.  Math with <code>ArbFloat</code> does not provide the assurance one gets using <code>ArbReal</code>, as an <code>ArbFloat</code> is a point value.  While some effort has been taken to provide you with more reliable results from math with <code>ArbFloat</code> values than would be the case using the underlying library itself, <code>ArbReal</code> or <code>ArbComplex</code> are suggested for work that is important to you.  <code>ArbFloat</code> is appropriate when exactness is not required during development, or with applications that are approximating something at increasing precisions.</p>

<p dir="auto">If you want to work with bit-level precision, first do <code>setextrabits(0)</code>.</p>
<p dir="auto">Otherwise, some extra bits are used to assist with printing values rounded to the last digit displayed.  You can find out how many extra bits are used with <code>extrabits()</code>.  If you want to change the number of extra bits used, call <code>setextrabits</code> with the desired number of extra bits.</p>
<p dir="auto">You can set the internal working precision (which is the same as the displayed precision with <code>setextrabits(0)</code>) to a given number of bits or a given number of decimal digits:</p>
<p dir="auto"><code>setworkingprecision(ArbFloat, bits=250)</code>, <code>setworkingprecision(ArbReal, digits=100)</code></p>
<p dir="auto">The type can be any of <code>ArbFloat</code>, <code>ArbReal</code>, <code>ArbComplex</code>.  All types share the same precision so interconversion makes sense.</p>
<p dir="auto">You can set the external displayed precision (which is the the same as the internal precision with <code>setextrabits(0)</code>) to a given
number of bits or a given number of decimal digits:</p>
<p dir="auto"><code>setprecision(ArbFloat, bits=250)</code>, <code>setworkingprecision(ArbReal, digits=100)</code></p>
<p dir="auto">The type can be any of <code>ArbFloat</code>, <code>ArbReal</code>, <code>ArbComplex</code>.  All types share the same precision so interconversion makes sense.</p>

<p dir="auto">Initially, the default precision is set to 106 bits.  All ArbNumeric types use the same default precision. You can change this to e.g. 750 bits: <code>setprecision(ArbFloat, 750)</code> or <code>setprecision(ArbReal, 750)</code> or <code>setprecision(ArbComplex, 750)</code>. Change one default the others follow automatically.  This is done to preserve internal consistency.  Sometimes more than one type is used within a function.  The minimal precision allowed is 24 bits. There is no maximum. The underlying C library calculates more rapidly than BigFloat at any precision.</p>
<p dir="auto">The precision in use may be set globally, as with BigFloats, or it may be given with the constructor.  For most purposes, you should work with a type at one, two, or three precisions.  It is helps clarity to convert precisions explicitly, however, it is not necessary.</p>

<p dir="auto">The radii are kept using an Arb C library internal structure that has a 30 bit unsigned significand and a power-of-2 exponent that is, essentially, a BigInt.  All radii are nonnegative.  From the Arb documentation:</p>
<blockquote>
<p dir="auto">The mag_t type holds an unsigned floating-point number with a fixed-precision mantissa (30 bits) and an arbitrary-precision exponent ..., suited for representing magnitude bounds. The special values zero and positive infinity are supported, but not NaN.   Operations that involve rounding will always produce a valid bound, For performance reasons, no attempt is made to compute the best possible bounds: in general, a bound may be several ulps larger/smaller than the optimal bound.</p>
</blockquote>
<p dir="auto">When constructing intervals , you should scale the radius to be as small as possible while preserving enclosure.</p>

<li>

<p dir="auto">For a numeric types like <code>Float64</code> and <code>ComplexF64</code> with about twice their precision, <a href="https://github.com/JuliaMath/Quadmath.jl">Quadmath.jl</a> exports <code>Float128</code> and <code>ComplexF128</code>.  For almost as much precision with better performance, <a href="https://github.com/JuliaMath/DoubleFloats.jl">DoubleFloats.jl</a> exports <code>Double64</code> and <code>ComplexDF64</code>. ValidatedNumerics.jl and other packages available at <a href="https://github.com/JuliaIntervals">JuliaIntervals</a> provide an alternative approach to developing correctly contained results.
