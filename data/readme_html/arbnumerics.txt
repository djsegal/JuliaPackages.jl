<p>For multiprecision numerical computing using values with 25..2,500 digits. With arithmetic and higher level mathematics, this package offers you the best balance of performance and accuracy.</p>
<p>This package uses the <a href="http://arblib.org/index.html" rel="nofollow">Arb C Library</a>, and adapts some C library interface work from <a href="https://github.com/wbhart/Nemo.jl">Nemo</a> (see <a href="https://github.com/JeffreySarnoff/ArbNumerics.jl/blob/master/README.md#acknowledgements"><em>below</em></a>).</p>

<h2><a id="user-content-introduction" class="anchor" aria-hidden="true" href="#introduction"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Introduction</h2>
<p>ArbNumerics exports three types: <code>ArbFloat</code>, <code>ArbReal</code>, <code>ArbComplex</code>.  <code>ArbFloat</code> is an extended precision floating point type. Math using <code>ArbFloat</code> is expected to be very near the veridical value, and often is the closest value for the precision in use. <code>ArbReal</code> is an interval-valued quantity formed of an <code>ArbFloat</code> (the midpoint) and a <code>radius</code>.  Math functions with <code>ArbReal</code> are assured to enclose the veridical value.  This assurance extends to multiple function applications.  <code>ArbComplex</code> is an <code>ArbReal</code> pair (real, imaginary).  The same enclosure assurance applies.</p>
<p>While the bounds of an <code>ArbReal</code> or <code>ArbComplex</code> are available, the default is to show these values as digit sequences which almost assuredly are accurate, in a round to nearest sense, to the precision displayed.  Math with <code>ArbFloat</code> does not provide the assurance one gets using <code>ArbReal</code>, as an <code>ArbFloat</code> is a point value.  While some effort has been taken to provide you with more reliable results from math with <code>ArbFloat</code> values than would be the case using the underlying library itself, <code>ArbReal</code> or <code>ArbComplex</code> are suggested for work that is important to you.  <code>ArbFloat</code> is appropriate when exactness is not required during development, or with applications that are approximating something at increasing precisions.</p>

<p>If you want to work with bit-level precision, first do <code>setextrabits(0)</code>.</p>
<p>Otherwise, some extra bits are used to assist with printing values rounded to the last digit displayed.  You can find out how many extra bits are used with <code>extrabits()</code>.  If you want to change the number of extra bits used, call <code>setextrabits</code> with the desired number of extra bits.</p>
<p>You can set the internal working precision (which is the same as the displayed precision with <code>setextrabits(0)</code>) to a given number of bits or a given number of decimal digits:</p>
<p><code>setworkingprecision(ArbFloat, bits=250)</code>, <code>setworkingprecision(ArbReal, digits=100)</code></p>
<p>The type can be any of <code>ArbFloat</code>, <code>ArbReal</code>, <code>ArbComplex</code>.  All types share the same precision so interconversion makes sense.</p>
<p>You can set the external displayed precision (which is the the same as the internal precision with <code>setextrabits(0)</code>) to a given
number of bits or a given number of decimal digits:</p>
<p><code>setprecision(ArbFloat, bits=250)</code>, <code>setworkingprecision(ArbReal, digits=100)</code></p>
<p>The type can be any of <code>ArbFloat</code>, <code>ArbReal</code>, <code>ArbComplex</code>.  All types share the same precision so interconversion makes sense.</p>

<p>Initially, the default precision is set to 106 bits.  All ArbNumeric types use the same default precision. You can change this to e.g. 750 bits: <code>setprecision(ArbFloat, 750)</code> or <code>setprecision(ArbReal, 750)</code> or <code>setprecision(ArbComplex, 750)</code>. Change one default the others follow automatically.  This is done to preserve internal consistency.  Sometimes more than one type is used within a function.  The minimal precision allowed is 24 bits. There is no maximum. The underlying C library calculates more rapidly than BigFloat at any precision.</p>
<p>The precision in use may be set globally, as with BigFloats, or it may be given with the constructor.  For most purposes, you should work with a type at one, two, or three precisions.  It is helps clarity to convert precisions explicitly, however, it is not necessary.</p>

<h3><a id="user-content-working-with-intervals" class="anchor" aria-hidden="true" href="#working-with-intervals"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>working with intervals</h3>
<p>The radii are kept using an Arb C library internal structure that has a 30 bit unsigned significand and a power-of-2 exponent that is, essentially, a BigInt.  All radii are nonnegative.  From the Arb documentation:</p>
<blockquote>
<p>The mag_t type holds an unsigned floating-point number with a fixed-precision mantissa (30 bits) and an arbitrary-precision exponent ..., suited for representing magnitude bounds. The special values zero and positive infinity are supported, but not NaN.   Operations that involve rounding will always produce a valid bound, For performance reasons, no attempt is made to compute the best possible bounds: in general, a bound may be several ulps larger/smaller than the optimal bound.</p>
</blockquote>
<p>When constructing intervals , you should scale the radius to be as small as possible while preserving enclosure.</p>

<ul>
<li>
<p><a href="https://arblib.org" rel="nofollow">Arb</a> is a C library for rigorous real and complex arithmetic with arbitrary precision.
Fredrik Johansson is Arb's designer and primary author., with contributions from others.</p>
</li>
<li>
<p>Arb tracks numerical errors automatically using the midpoint-radius representation of an interval.</p>
</li>
<li>
<p>Arb is designed to provide evaluands that contain the veridical numerical result.</p>
</li>
<li>
<p>Arb uses algorithms with provable error bounds foe multiprecision mathematical functions.</p>
</li>
<li>
<p>The code is thread-safe, portable, and extensively tested. The library outperforms others.</p>
</li>
</ul>

<p>For a numeric types like <code>Float64</code> and <code>ComplexF64</code> with about twice their precision, <a href="https://github.com/JuliaMath/Quadmath.jl">Quadmath.jl</a> exports <code>Float128</code> and <code>ComplexF128</code>.  For almost as much precision with better performance, <a href="https://github.com/JuliaMath/DoubleFloats.jl">DoubleFloats.jl</a> exports <code>Double64</code> and <code>ComplexDF64</code>. ValidatedNumerics.jl and other packages available at <a href="https://github.com/JuliaIntervals">JuliaIntervals</a> provide an alternative approach to developing correctly contained results.  Those packages are very good and worthwhile when you do not require multiprecision numerics.</p>
