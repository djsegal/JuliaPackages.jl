<p dir="auto">This <a href="https://julialang.org" rel="nofollow">Julia</a> package allows for playing and analyzing <a href="https://en.wikipedia.org/wiki/Wordle" rel="nofollow">Wordle</a> and related games, such as <a href="https://cojofra.github.io/primel/" rel="nofollow">Primel</a>.</p>
<p dir="auto">A game is represented by a <code>GamePool</code> object containing potential guesses, a subset of which are valid targets, and some game play status information.
By default the game is played as in the "Hard Mode" setting on the Wordle app and web site, which means that the only guesses allowed at each turn are those in the current target pool.
As a consequence, the initial pool of potential guesses is the same as the initial target pool.</p>

<p dir="auto">This creates a <code>GamePool</code> from the Wordle targets, a list of 2315 5-letter English words.
The <code>playgame!</code> and <code>showgame!</code> methods can play a Wordle game, selecting each guess according to a criterion.
By default the guess is chosen to maximize the <a href="https://en.wikipedia.org/wiki/Entropy_(information_theory)" rel="nofollow">entropy</a> of the distribution of scores from the current target pool, as explained below.</p>
<p dir="auto">For example, suppose the target is <code>"super"</code>.
It takes 4 guesses to isolate this target using this strategy.</p>

<p dir="auto">The size of the initial target pool is 2315.
The first guess, <code>"raise"</code>, will reduce the size of target pool after it has been scored.
It is not known what the score will be but the set of scores from all possible targets can be calculated.
Assuming the possible targets are equally likely, this gives a distribution of scores, and also a distribution of pool sizes after the guess is scored.
Informally, the entropy of the distribution of scores is a measure of how uniformly they are distributed over the set of the possible scores.
Choosing the guess with the greatest entropy will likely result in a large reduction in the size of the target pool after the guess is scored.</p>
<p dir="auto">The expected size of the target pool, after this guess is scored, is a little over 61.
The actual score in this game, represented as <code>ğŸŸ¨ğŸŸ«ğŸŸ«ğŸŸ¨ğŸŸ¨</code> in colored tiles or <code>[1,0,0,1,1]</code> as digits, indicates that  <code>r</code>, <code>s</code> and <code>e</code> are in the target but not in the guessed positions and <code>a</code> and <code>i</code> do not occur in the target.  (The <code>sc</code> value in that row, 85, is the decimal value of <code>10011</code> in base-3.)</p>
<p dir="auto">(This package uses the Unicode character <code>U+F7EB</code>, the <code>:large_brown_square:</code> emoji, <code>ğŸŸ«</code>, instead of a gray square for the "didn't match" tile - a kind of "traffic lights" motif.
But the real reason for this choice is that it is surprisingly difficult to get a consistent-width black or gray square symbol in many fonts.)</p>
<p dir="auto">There are only 18 of the 2315 possible targets that would have given this score.
Of these 18 targets the guess that will do the best job of spreading out the distribution of scores is <code>"sheer"</code>.
The actual score for this guess is <code>ğŸŸ©ğŸŸ«ğŸŸ«ğŸŸ©ğŸŸ©</code>, meaning that the <code>s</code>, the second <code>e</code> and the <code>r</code> are in the correct positions, the <code>h</code> is not in the target and there isn't a second <code>e</code>.</p>
<p dir="auto">(When a character is repeated in a guess but occurs only once in the target, "correct position" takes precedence over "in the target", as in this case.
If none of the guesses are in the correct position then the leftmost position in the guess takes precedence.)</p>
<p dir="auto">The size of the target pool is reduced to 4, which is larger than the expected size of 2.11, and the game continues with another guess (<code>"sober"</code>) and another score (<code>ğŸŸ©ğŸŸ«ğŸŸ«ğŸŸ©ğŸŸ©</code>) until the target, <code>"super"</code> is matched.</p>
<p dir="auto">If no target is specified in a call to <code>showgame!</code> or <code>playgame!</code> one is chosen at random from the set of possible targets.</p>

<p dir="auto">Each turn in a Wordle-like game can be regarded as submitting a guess to an "oracle" which returns a score that is used to update the information on the play.
Initially the target can be any element of the target pool.
Each guess/score combination reduces the size of the target pool, as shown in the game summaries above.</p>
<p dir="auto">In a <code>GamePool</code> object the actual pool of potential targets and guesses is not modified.
Instead there is a <code>BitVector</code> field, <code>active</code>, that is used to keep track of the active target pool.
The size of the current target pool is the sum of <code>active</code>.</p>
<p dir="auto">The score for a particular guess is known to the oracle but not to the player.
However, the scores for any potential guess and a member of the target pool can be evaluated.
The number of possible scores is finite (<code>3^N</code> where <code>N</code> is the number of tiles in the score).</p>
<p dir="auto">For example the first guess chosen in the Wordle games shown about is <code>"raise"</code>, which is at position 1535 in <code>wordle.targetpool</code>.</p>

<p dir="auto">Assuming the targets are equally likely, which apparently is the case in the online games, the probability of each score is the count for that score divided by the size of the active target pool.
The expected pool size is the sum of the <code>counts</code> multiplied by the probabilities or, equivalently, the sum of the squared counts divided by the sum of the counts.</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="julia&gt; sum(abs2, wordle.counts) / sum(wordle.counts)  # abs2(x) returns x * x

<p dir="auto">Measured in bits, the entropy of the probabilities is <code>- Î£áµ¢ páµ¢ logâ‚‚(páµ¢)</code>.
Entropy measures how the probability is dispersed among the possible scores.
The best case is for each of the <code>n</code> possible scores to have probability <code>1/n</code> of occurring.
In that case, whichever score is returned, there will only be a small number of targets with that score.
It is not possible to get uniform pool sizes from a starting guess but, sometimes when the target pool is small, a particular guess may be able to split the remaining <code>k</code> targets into <code>k</code> distinct scores.</p>
<p dir="auto">In particular, this always occurs when there are only two targets left.</p>
<p dir="auto">The guesses can be chosen to minimize the expected pool size but this strategy is not as effective as is maximizing the entropy.</p>

The guess is "raise" with a pool size of 2315, an entropy of 5.88 and an expected pool size of 61.00 after scoring.</p>
<p dir="auto">If the score for "raise" is <code>ğŸŸ¨ğŸŸ«ğŸŸ«ğŸŸ«ğŸŸ¨</code>, the next guess will be "deter", with the characteristics shown.
If the score is <code>ğŸŸ«ğŸŸ«ğŸŸ«ğŸŸ«ğŸŸ«</code>, which is the most likely score for the first guess, the next guess is "mulch", and so on.</p>
<p dir="auto">Note that in the tree the score is associated with the guess that it will produce next, whereas in the summary of the game the score is associated with the guess that produced it.</p>
<p dir="auto">The reason that this tree is not very interesting is that it simply reproduces the game summaries, with the minor changes that the root node is common to all the games and the score tiles refer to the score that has been observed, not the score that will be observed.</p>
