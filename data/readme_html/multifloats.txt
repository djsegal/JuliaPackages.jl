<p><strong>MultiFloats.jl</strong> is a Julia package for extended-precision arithmetic using 100 – 400 bits (≈ 30 – 120 digits). In this range, it is the fastest extended-precision library that I am aware of. At 100-bit precision, <strong>MultiFloats.jl</strong> is roughly <strong>40x faster than <a href="https://docs.julialang.org/en/v1/manual/integers-and-floating-point-numbers/#Arbitrary-Precision-Arithmetic" rel="nofollow"><code>BigFloat</code></a></strong> and <strong>2x faster than <a href="https://github.com/JuliaMath/DoubleFloats.jl">DoubleFloats.jl</a></strong>.</p>
<p><strong>MultiFloats.jl</strong> achieves speed by using native, vectorizable <code>Float64</code> operations and immutable data structures that do not dynamically allocate memory. In many cases, <code>MultiFloat</code> arithmetic can be performed entirely in CPU registers, eliminating memory access altogether. In contrast, <a href="https://docs.julialang.org/en/v1/manual/integers-and-floating-point-numbers/#Arbitrary-Precision-Arithmetic" rel="nofollow"><code>BigFloat</code></a> allocates memory with every single arithmetic operation, requiring frequent pauses for garbage collection.</p>
<p><strong>MultiFloats.jl</strong> provides basic arithmetic operations (<code>+</code>, <code>-</code>, <code>*</code>, <code>/</code>, <code>sqrt</code>), comparison operators (<code>==</code>, <code>!=</code>, <code>&lt;</code>, <code>&gt;</code>, <code>&lt;=</code>, <code>&gt;=</code>), <code>exp</code>, <code>log</code>, and floating-point introspection methods (<code>isfinite</code>, <code>eps</code>, <code>minfloat</code>, etc.). Work on trigonometric and hyperbolic functions is currently in progress.</p>
<p><strong>MultiFloats.jl</strong> stores extended-precision numbers in a generalized form of <a href="https://en.wikipedia.org/wiki/Quadruple-precision_floating-point_format#Double-double_arithmetic" rel="nofollow">double-double representation</a> which supports arbitrary number of components. This idea takes inspiration from Jonathan Shewchuk's work on <a href="http://dx.doi.org/10.1007/pl00009321" rel="nofollow">adaptive-precision floating-point arithmetic</a> and Yozo Hida, Xiaoye Li, and David Bailey's <a href="https://doi.org/10.1109/ARITH.2001.930115" rel="nofollow">algorithms for quad-double arithmetic</a>, combined in a novel fashion with Julia's unique JIT architecture and metaprogramming capabilities.</p>

<h2><a id="user-content-usage" class="anchor" aria-hidden="true" href="#usage"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Usage</h2>
<p><strong>MultiFloats.jl</strong> provides the types <code>Float64x2</code>, <code>Float64x3</code>, ..., <code>Float64x8</code>, which represent extended-precision numbers with 2x, 3x, ..., 8x the precision of <code>Float64</code>. These are all subtypes of the parametric type <code>MultiFloat{T,N}</code>, where <code>T = Float64</code> and <code>N = 2, 3, ..., 8</code>.</p>
<p>Instances of <code>Float64x2</code>, <code>Float64x3</code>, ..., <code>Float64x8</code> are convertible to and from <code>Float64</code> and <code>BigFloat</code>, as shown in the following example.</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; using MultiFloats

julia&gt; x = Float64x4(2.0)

julia&gt; y = sqrt(x)

julia&gt; y * y - x
-1.1566582006914837e-66

julia<span class="pl-k">&gt;</span> x <span class="pl-k">=</span> <span class="pl-c1">Float64x4</span>(<span class="pl-c1">2.0</span>)

<p>A comparison with <code>sqrt(BigFloat(2))</code> reveals that all displayed digits are correct in this example.</p>
<p><sup>Note: <strong>MultiFloats.jl</strong> also provides a <code>Float64x1</code> type that has the same precision as <code>Float64</code>, but behaves like <code>Float64x2</code>–<code>Float64x8</code> in terms of supported operations. This is occasionally useful for testing, since any code that works for <code>Float64x1</code> should also work for <code>Float64x2</code>–<code>Float64x8</code> and vice versa.</sup></p>
<h2><a id="user-content-precision-modes" class="anchor" aria-hidden="true" href="#precision-modes"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Precision Modes</h2>
<p><strong>MultiFloats.jl</strong> provides three user-selectable levels of precision. The default mode is <strong>standard mode</strong>, which aims for a sweet-spot between performance and precision. <strong>Clean mode</strong> does a bunch of extra work to get the last few bits of the answer right, while <strong>sloppy mode</strong> throws some safety guarantees out the window in pursuit of reckless speed.</p>
<p><strong>When in doubt, stick to standard mode.</strong> If you come across a numerical bug, then switch to clean mode. The use of sloppy mode should be limited to highly specialized cases with strong assumptions about the input data. Sloppy mode can exhibit bizarre failure modes related to (the lack of) renormalization that are difficult to reproduce.</p>
<p>To switch between precision modes, call any of the following three functions:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="MultiFloats.use_clean_multifloat_arithmetic()
MultiFloats.use_standard_multifloat_arithmetic() [default]
MultiFloats.use_sloppy_multifloat_arithmetic()
"><pre><code>MultiFloats.use_clean_multifloat_arithmetic()
MultiFloats.use_standard_multifloat_arithmetic() [default]
MultiFloats.use_sloppy_multifloat_arithmetic()
</code></pre></div>
<p>Note that switching between precision modes is a very expensive operation that is not thread-safe. Calling any of these functions triggers recompilation of all <code>MultiFloat</code> arithmetic operations, so this should never be performed in the middle of a calculation.</p>
<p>Each of these functions takes an optional <code>Int</code> parameter that dictates the <code>MultiFloat</code> sizes to generate code for. For example, if you want to use <code>Float64x9</code> (which is not provided by default), you can call <code>MultiFloats.use_standard_multifloat_arithmetic(9)</code> to define the necessary arithmetic operators. Note that this will not define the <em>name</em> <code>Float64x9</code>; you will have to refer to this type as <code>MultiFloat{Float64,9}</code> or <code>Float64x{9}</code>.</p>

<h2><a id="user-content-caveats" class="anchor" aria-hidden="true" href="#caveats"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Caveats</h2>
<p><strong>MultiFloats.jl</strong> requires an underlying implementation of <code>Float64</code> with IEEE round-to-nearest semantics. It works out-of-the-box on x86 and ARM, but may fail on more exotic architectures.</p>
<p><strong>MultiFloats.jl</strong> does not attempt to propagate IEEE <code>Inf</code> and <code>NaN</code> values through arithmetic operations, as this <a href="https://github.com/dzhang314/MultiFloats.jl/issues/12#issuecomment-751151737">could cause significant performance losses</a>. You can pass these values through the <code>Float64x{N}</code> container types, and introspection functions (<code>isinf</code>, <code>isnan</code>, etc.) will work, but arithmetic operations will typically produce <code>NaN</code> on all non-finite inputs.</p>
