<h2><a id="user-content-basic-api" class="anchor" aria-hidden="true" href="#basic-api"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Basic API</h2>
<p>The <code>AbstractWallets</code> API defines the following basic structures:</p>
<table>
<thead>
<tr>
<th>Type</th>
<th>Description</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>AbstractAddress</code></td>
<td>Abstraction over digital addresses.</td>
</tr>
<tr>
<td><code>AbstractPortfolio</code></td>
<td>A container of assets with methods for evaluating their value relative to other assets.</td>
</tr>
<tr>
<td><code>AbstractWallet</code></td>
<td>A wallet is a portfolio with a digital address to identify it.</td>
</tr>
</tbody>
</table>
<h3><a id="user-content-transactions" class="anchor" aria-hidden="true" href="#transactions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Transactions</h3>
<p>Transactions between digital wallets or from wallets to addresses is mediated by the <code>AbstractTransaction</code> type. Transactions can take the two generic forms</p>
<table>
<thead>
<tr>
<th>Form</th>
<th>Description</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>AbstractSendTransaction</code></td>
<td>Results in the movement of assets from one party to another.</td>
</tr>
<tr>
<td><code>AbstractDestroyTransaction</code></td>
<td>Results in the net destruction of assets.</td>
</tr>
</tbody>
</table>
<p>which can be mediated by the two generic transaction functions:</p>
<table>
<thead>
<tr>
<th>Action</th>
<th>Constructor</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>send!</code></td>
<td><code>build_send_transaction</code></td>
</tr>
<tr>
<td><code>destroy!</code></td>
<td><code>build_destroy_transaction</code></td>
</tr>
</tbody>
</table>
<p>To use more advanced transaction types we have the following states:</p>
<table>
<thead>
<tr>
<th>State</th>
<th>Description</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>is_pending</code></td>
<td>Transaction is waiting to complete.</td>
</tr>
<tr>
<td><code>has_executed</code></td>
<td>Transaction was executed but not necessarily completed. It is at least pending.</td>
</tr>
<tr>
<td><code>was_completed</code></td>
<td>Transaction was executed and completed.</td>
</tr>
<tr>
<td><code>was_cancelled</code></td>
<td>Transaction was cancelled before completion.</td>
</tr>
<tr>
<td><code>failed</code></td>
<td>Transaction failed to complete, but was not cancelled.</td>
</tr>
</tbody>
</table>
<p>and the allowed actions:</p>
<table>
<thead>
<tr>
<th>Action</th>
<th>Description</th>
</tr>
</thead>
<tbody>
<tr>
<td><code>sign!</code></td>
<td>Signatures on a transaction verify that it should be executed.</td>
</tr>
<tr>
<td><code>execute!</code></td>
<td>Execution of a transaction is the attempt to perform the appropriate transfer of assets defined in the transaction.</td>
</tr>
<tr>
<td><code>cancel!</code></td>
<td>Stop a pending transaction.</td>
</tr>
<tr>
<td><code>rollback!</code></td>
<td>Reset a completed transaction to return the state of the associated portfolios to before the transaction occured.</td>
</tr>
<tr>
<td><code>cancelback!</code></td>
<td>Cancel and then roll back the transaction.</td>
</tr>
</tbody>
</table>

<h3><a id="user-content-private-keys" class="anchor" aria-hidden="true" href="#private-keys"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Private Keys</h3>
<p>For cryptocurrency wallets and other wallets protected by asymmetric-cryptography, public-private key pairs are the method with which wallets can identify themselves. To access the keys, use either the <code>keypairs(wallet)</code> function or the <code>privatekey(wallet, publickey)</code>.</p>
<h3><a id="user-content-deterministic-wallets" class="anchor" aria-hidden="true" href="#deterministic-wallets"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Deterministic Wallets</h3>
<p>A deterministic wallet can generate new public-private key pairs from one root private key. Such wallets are represented by the <code>AbstractDeterministicWallet</code> type. Use <code>rootkey</code>, <code>mnemonic</code>, or <code>seed</code> to access the root private key, and <code>next_keypair!</code> or <code>next_address!</code> to generate the next key.</p>
