<p dir="auto">Scanf scans UTF8-encoded input streams or strings and creates output data according to a format string.
It mimics the behaviour of the C-function with the same name.</p>

<h2 dir="auto"><a id="user-content-features" class="anchor" aria-hidden="true" href="#features"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Features</h2>
<p dir="auto"><code>Scanf</code> provides the macro <code>r, a,... = @scanf([io, ] "format_string", args...)</code>
and the function <code>r, a,... = scanf(io, f::Scanf.Format, args...)</code>.</p>
<p dir="auto">The format string must be a string literal, which is evaluated once at macro expansion time.</p>
<p dir="auto">Alternatively <code>f = Scanf.format"format_string"</code> or <code>f = Scanf.Format(::AbstractString)</code> create a format object, which can be
used in the function call.</p>
<p dir="auto">The arguments are default values of types <code>Real</code>, <code>AbstractChar</code>, <code>AbstractString</code>, <code>Ptr</code>, <code>AbstractVector{Char}</code>.</p>
<p dir="auto">They may also be concrete subtypes of <code>Real</code>, <code>AbstractChar</code>, <code>AbstractString</code>, <code>Ptr</code>.</p>
<p dir="auto">Numeric format specifiers are compatible with numeric arguments and <code>String</code>. Conversion errors may happen (float =&gt; int).</p>
<p dir="auto">If the numeric arg type is not wide enough for the value, boundary values with the correct sign are stored (e.g. <code>Inf</code>, <code>-0.0</code>).</p>
<p dir="auto">Format specifiers <code>c</code>, <code>s</code>, <code>[</code> are all compatible with arguments <code>Char</code>, <code>Char[]</code>, and <code>String</code>.
In case of <code>Char</code> the first character of a string is taken.</p>
<p dir="auto">All output data are returned as a tuple including the number of assigned values as the first element.
If a value cannot be parsed,
the default value is assigned. In the case of no value is in the corresponding element of <code>arg</code>, the default value is derived form <code>T</code>.</p>
<p dir="auto">If the default argument is a <code>Vector</code> object, the output values are additionally stored in it.</p>
<p dir="auto">The format strings follow the definition of C++-scanf <a href="https://en.cppreference.com/w/c/io/fscanf" rel="nofollow">C++ reference</a>
with some adaptations:</p>
<ul dir="auto">
<li>
<p dir="auto">All unicode characters are supported in format strings and input data.</p>
</li>
<li>
<p dir="auto">in format strings, whitespace specifiers are only the ASCII space characters in <code>" \n\r\t\f\v"</code>.</p>
</li>
<li>
<p dir="auto">in input data, all characters <code>x</code> with <code>isspace(x) == true</code> are skipped by any whitespace specifier in the format string.</p>
</li>
<li>
<p dir="auto">The <code>%n$...</code> form of format specifications is not supported.</p>
</li>
<li>
<p dir="auto">Optional type modifiers like <code>h</code>, <code>l</code>, <code>L</code> etc. are ignored; the target type is derived from the corresponding default argument, instead.</p>
</li>
<li>
<p dir="auto">for type specifier <code>%c</code>, without <code>width</code> specified, the corresponding argument must have type <code>Char</code> or <code>String</code>.</p>
</li>
<li>
<p dir="auto">for type specifier <code>%Nc</code>, with a width field <code>N</code>, the argument must have type <code>String</code> or <code>Vector{Char}</code>.
This vector is re-used and resized to the number of characters actually read.</p>
</li>
<li>
<p dir="auto">the type specifier <code>%n</code>, returns an integer value, which is the character offset in the input data, consumed by this scanf so far.</p>
</li>
<li>
<p dir="auto">the type specifier <code>%p</code> requires a <code>Ptr</code> default argument.</p>
</li>
<li>
<p dir="auto">The return value of both calls is the amount of output arguments, followed by all output data, the trailing ones maybe default values.
In contrast to C and C++, also the arguments for <code>%n</code> are counted.</p>
</li>
</ul>
<h3 dir="auto"><a id="user-content-implementation" class="anchor" aria-hidden="true" href="#implementation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Implementation</h3>
<p dir="auto">If the input stream is exhausted before a character is read, the EOF-indicator <code>-1</code> is returned in place of the number of assigned values.</p>
<p dir="auto">For format specifiers "Whitespace", any number of characters (also zero) <code>x</code> with <code>isspace(x) == true</code> are consumed from the input stream.</p>
<p dir="auto">For a format specifier literal character, the next character is read and compared with the literal character. If it is equal, it is consumed,
otherwise the process fails.</p>
<p dir="auto">If format specifier "Character" <code>%c</code> is processed, at least one character is read and assigned to the output argument. If no character is available, the process fails.</p>
<p dir="auto">If format specifier <code>%n</code> is processed, no data are consumed (and no EOF returned), but the current read position is returned.</p>
<h2 dir="auto"><a id="user-content-description" class="anchor" aria-hidden="true" href="#description"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Description</h2>
<p dir="auto">As derived from <a href="https://en.cppreference.com/w/c/io/fscanf" rel="nofollow">C++ reference</a></p>
<p dir="auto">The <code>scanf</code> function reads input from the stream pointed to by <code>io</code>, under control of the string <code>format</code> that specifies the admissible
input sequences and how they are to be interpreted, using subsequent arguments
to define the type of the converted input. The number of arguments must match the number of format specifiers required by the format.</p>
<p dir="auto">The format is composed of zero or more directives: one or more (ASCII) white-space characters, an ordinary (UTF8) character (neither <code>'%'</code> nor a
(ASCII) white-space character), or a conversion specification. Each conversion specification is introduced by the character <code>'%'</code>. After the
<code>'%'</code>, the following appear in sequence:</p>
<ul dir="auto">
<li>An optional assignment-suppressing character <code>'*'</code>.</li>
<li>An optional decimal integer greater than zero that specifies the maximum field width (in characters).</li>
<li>An optional length modifier that is not used by this implementation.</li>
<li>A conversion specifier character that specifies the type of conversion to be applied.</li>
</ul>
<p dir="auto">The <code>scanf</code> function executes each directive of the format in turn. When all directives have been executed, or if a directive fails (as
detailed  below), the function returns. Failures are described as input failures (due to the occurrence of an encoding error or the unavailability of input characters), or matching failures (due to inappropriate input).</p>
<p dir="auto">A directive composed of white-space character(s) is executed by reading input up to the first non-white-space character (which
remains unread), or until no more characters can be read. The directive never fails.</p>
<p dir="auto">A directive that is an ordinary character is executed by reading the next character of the stream. If that character differs from the  directive, the directive fails and the differing and subsequent characters remain unread. Similarly, if end-of-file, an encoding error, or a read error prevents a character from being read, the directive fails.</p>
<p dir="auto">A directive that is a conversion specification defines a set of matching input sequences, as described below for each specifier. A conversion
specification is executed in the following steps:</p>
<p dir="auto">Input white-space characters (as specified by the <code>isspace</code> function) are skipped, unless the specification includes a <code>'['</code>, <code>'c'</code>,
or <code>'n'</code> specifier. These white-space characters are not counted against a specified field width.</p>
<p dir="auto">An input item is read from the stream, unless the specification includes an <code>'n'</code> specifier.  An input item is defined as the longest sequence
of input characters which does not exceed any specified field width and which is, or is a prefix of, a matching input sequence.</p>
<p dir="auto">The first character, if any, after the input item remains unread. If the length of the input item is zero, the execution of the directive
fails; this condition is a matching failure unless end-of-file, an encoding error, or a read error prevented input from the stream, in which
case it is an input failure.</p>
<p dir="auto">Except in the case of a <code>'%'</code> specifier, the input item (or, in the case of a <code>'%n'</code> directive, the count of input characters) is converted
to a type appropriate to the conversion specifier and corresponding argument. If the input item is not a matching sequence, the execution of the
directive fails: this condition is a matching failure. Unless assignment suppression was indicated by a <code>*</code>, the result of the conversion
is  pushed to the ouput tuple.</p>
<p dir="auto">If the result of the conversion cannot be represented in the output type, a conversion error is thrown.</p>
<p dir="auto">Optional length modifiers <code>l</code>, <code>ll</code>, <code>h</code>, <code>hh</code>, <code>L</code>, <code>j</code>, <code>z</code>, <code>t</code> are accepted before all type specifier characters, but otherwise ignored.</p>
<p dir="auto">The conversion specifiers and their meanings are:</p>
<ul dir="auto">
<li>
<p dir="auto"><code>d</code> Matches an optionally signed decimal integer, whose format is the same as expected for the subject sequence of the <code>parse(T, _, base=10)</code>
function, where <code>T</code> is the integer type of the argument.</p>
</li>
<li>
<p dir="auto"><code>i</code> Matches an optionally signed integer, whose format is the same as expected for the subject sequence of the <code>parse(T, _, base=nothing)</code>
function.</p>
</li>
<li>
<p dir="auto"><code>o</code> Matches an optionally signed octal integer, whose format is the same as expected for the subject sequence of the <code>parse(T, _, base=8)</code>
function.</p>
</li>
<li>
<p dir="auto"><code>u</code> Matches an optionally signed decimal integer, whose format is the same as expected for the subject sequence of the <code>parse(T, _, base=10)</code>
function.</p>
</li>
<li>
<p dir="auto"><code>x</code> Matches an optionally signed hexadecimal integer, whose format is the same as expected for the subject sequence of thew <code>parse(T, _, base=16)</code> function.</p>
</li>
<li>
<p dir="auto"><code>a</code>,<code>e</code>,<code>f</code>,<code>g</code> Matches an optionally signed floating-point number, <code>infinity</code>, or <code>NaN</code>, whose format is the same as expected for the<br>
subject sequence of the <code>parse(T, _)</code> function, where <code>T</code> is a floating point type.</p>
</li>
<li>
<p dir="auto"><code>c</code> Matches a sequence of characters of exactly the number specified by the field width (<code>1</code> if no field width is present in the directive).
The argument type must be <code>String</code>, <code>Char</code>, or <code>Vector{Char}</code>, If the field width is greater than <code>1</code> and a <code>Char</code> type is given, only the
first character is stored.</p>
</li>
<li>
<p dir="auto"><code>s</code> Matches a (non-empty) sequence of non-white-space characters. The argument types are as for <code>c</code>.</p>
</li>
<li>
<p dir="auto"><code>[</code> Matches a nonempty sequence of characters from a set of expected characters (the scanset). The argument types are as for <code>c</code>.
The conversion specifier includes all subsequent characters in the format string, up to and including the matching right bracket (<code>]</code>).
The characters between the brackets (the scanlist) compose the scanset, unless the character after the left bracket is a circumflex (<code>^</code>),
in which case the scanset contains all characters that do not appear in the scanlist between the circumflex and the right bracket.
If the conversion specifier begins with <code>[]</code> or <code>[^]</code>, the right bracket character is in the scanlist and the next following right bracket
character is the matching right bracket that ends the specification; otherwise the first following right bracket character is the one that ends the specification. If a <code>-</code> character is in the scanlist and is not the first, nor the second where the first character is a <code>^</code>,
nor the last character, it defines the range between the character left of <code>-</code> and the character right of <code>-</code>. The order defining
the range is the integer range of unicode codepoints of the characters. Empty ranges (like <code>b-a</code>) are ignored.</p>
</li>
<li>
<p dir="auto"><code>p</code> Matches an set of sequences, which are the same as the set of sequences that may be produced by
the <code>%p</code> conversion of the <code>printf</code>-function. The corresponding argument must be of type <code>Ptr{T}</code> where T may be any type.
The  input  item  is  converted  to  a  pointer  value  in  an implementation-defined manner.  If the input item is a value converted earlier during  the  same  program  execution,  the  pointer  that  results  shall  compare equal to that value; otherwise the behavior of the <code>%p</code> conversion is undefined.</p>
</li>
<li>
<p dir="auto"><code>n</code> No input is consumed. The corresponding argument must be an integer type, into which is converted the number of characters read from the input
stream so far by this call to the <code>scanf</code> function. Execution of a <code>%n</code> directive does as well increment the assignment count returned at
the completion of execution of the <code>scanf</code> function. If the conversion specification includes an assignment-suppressing character no argument
is consumed. An optional width field is ignored.</p>
</li>
<li>
<p dir="auto"><code>%</code> Matches a single <code>'%'</code> character; no conversion or assignment occurs. The complete conversion specification is <code>%%</code>.
(with other words, <code>%%</code> is treated like a single ordinary character <code>%</code>).</p>
</li>
</ul>
<p dir="auto">If a conversion specification is invalid, the behavior is undefined.</p>
<p dir="auto">The conversion specifiers <code>A</code>,<code>E</code>,<code>F</code>,<code>G</code>, and <code>X</code> are also valid and behave the same as, respectively, <code>a</code>,<code>e</code>,<code>f</code>,<code>g</code>, and <code>x</code>.</p>
<p dir="auto">Trailing white space (including new-line characters) is left unread unless matched by a directive. The success of literal matches and
suppressed assignments is not directly determinable other than via the <code>%n</code> directive.</p>
