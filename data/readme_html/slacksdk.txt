<p dir="auto">Access to the Slack api is provided via the <code>WebClient</code> struct. Construct it by passing a <a href="https://api.slack.com/authentication/token-types" rel="nofollow">token</a> directly. Alternatively, set the token as an environment variable called <code>SLACK_API_TOKEN</code> and then use the no-arg constructor. You will most likely want to use a <em>bot</em> token here.</p>
<p dir="auto">The list of api methods are documented by slack at <a href="https://api.slack.com/methods" rel="nofollow">https://api.slack.com/methods</a>. It will be helpful to familiarise yourself with the structure of the api before using this package.</p>
<p dir="auto">The Slack api methods are generally of the form <code>group[.subgroup].methodname</code>. Replace the dots with underscores, and use that name to call the member function on the <code>WebClient</code> object you created above. For example, the <code>chat.postMessage(...)</code> API method is called using <code>client.chat_postMessage</code>, while the <code>admin.analytics.getFile</code> method is called using <code>client.admin_analytics_getFile(...)</code>. Required arguments should be passed as positional arguments to this function, and optional arguments are passed as keyword arguments.</p>
<blockquote>
<p dir="auto">There are some <a href="https://github.com/slackapi/slack-api-specs/issues/70" data-hovercard-type="issue" data-hovercard-url="/slackapi/slack-api-specs/issues/70/hovercard">discrepancies</a> between the online documentation and the api specs about which arguments are required. In case of doubt, please refer to spec file stored in this repo, or view the docs. Search the method name to find its details. Required parameters will be explicitly marked as such.</p>
</blockquote>

<p dir="auto">The files in the <code>web/</code> subdirectory of this package were generated from the <a href="https://github.com/slackapi/slack-api-specs">Slack API Specs</a>. The commit <code>bc08db49625630e3585bf2f1322128ea04f2a7f3</code> was used, which last had substantive changes in October 2020. The code generation used the <code>Swagger.jl@v0.3.5</code> julia package, invoked via <code>/home/me/.julia/packages/Swagger/LiWeJ/plugin/generate.sh -i specs/slack_web_openapi_v2_without_examples.json -o src/web -c specs/config.json</code>.</p>
<p dir="auto">The token parameter is present in all methods in this schema, but is not meant to be a query parameter. It is meant to be injected into the headers as a <em>Bearer</em> token, and is unclear how the spec expects this to work. Swagger.jl currently does not have automatic support for this mechanism.</p>
<p dir="auto">The following regex was therefore used to remove all <code>token</code> parameters from the API calls.  <code>/\{\n.*\n.*\n\s*"name": "token",\n.*\n(.*\n)?\s*\},?\n\s*//</code> . One manual intervention was required for the <code>/reactions.add</code> method.</p>
<p dir="auto">The header with the bearer token is then injected by a higher level wrapper within this package.</p>

<ul dir="auto">
<li>Only the Slack <a href="https://api.slack.com/web" rel="nofollow">Web API</a> is implemented. The <a href="https://api.slack.com/apis/connections/events-api" rel="nofollow">Event API</a> is not yet implemented.</li>
<li>Not all the methods are tested. Please open issues if you find problems calling any method.</li>
<li>There are no integration tests. Maybe a test workspace could be created exclusively for use by this package, and automated tests run against that? Contributions on these lines are most welcome.</li>
<li>The interface is somewhat non-julian. It's admittedly more pythonic. However, in this case, it seemed to be simplest and most convinient implementation. It allows us to hide a lot of the complexity of calling the API in a generic fashion.</li>
</ul>

<p dir="auto">There are a <a href="https://juliahub.com/ui/Search?q=Slack&amp;type=packages" rel="nofollow">few other packages</a> in the Julia ecosystem that interacts with the Slack API: <a href="https://juliahub.com/ui/Packages/Slack/vfmTp/0.1.2" rel="nofollow">Slack</a>, <a href="https://juliahub.com/ui/Packages/Slacker/zliBU/1.0.0" rel="nofollow">Slacker</a> and <a href="https://juliahub.com/ui/Packages/SlackThreads/irQKT/0.1.4" rel="nofollow">SlackThreads</a> Most of them seem limited to a small set of functionality, implementing only a couple of API methods. This package on the other hand is mostly auto-generated from the Slack API specs, and thus covers almost the entirety of the Slack API surface.</p>
