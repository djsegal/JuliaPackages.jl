<p>Strs.jl is container for a number of different packages from <a href="https://juliastring.org" rel="nofollow">JuliaString.org</a></p>
<p>It brings together the following:</p>
<ol>
<li>
<p>A better type of string literal, using <a href="https://github.com/JuliaString/StrLiterals.jl">StrLiterals</a>, <a href="https://github.com/JuliaString/StrFormat.jl">StrFormat</a>, and <a href="https:://github.com/JUliaString/StrEntities.jl">StrEntities</a></p>
<p>This is of the form <code>f"..."</code> or <code>F"..."</code>.</p>
<p>This uses Swift-style <code>\</code> escape sequences, such as <code>\u{xxxx}</code> for Unicode constants, instead of <code>\uXXXX</code> and <code>\UXXXXXXXX</code>, which have the advantage of not having to worry about some digit or letter A-F or a-f occurring after the last hex digit of the Unicode constant.</p>
<p>It also means that <code>$</code>, a very common character for LaTeX strings or output of currencies, does not need to be in a string quoted as '$'</p>
<p>It uses <code>\(expr)</code> for interpolation like Swift, instead of <code>$name</code> or <code>$(expr)</code>, which also has the advantage of not having to worry about the next character in the string someday being allowed in a name.</p>
<p>It allows for embedding Unicode characters using a variety of easy to remember names, instead of hex codes:
<code>\:emojiname:</code>
<code>\&lt;latexname&gt;</code>
<code>\N{unicodename}</code>
<code>\&amp;htmlname;</code>
Examples of this are:
<code>f"\&lt;dagger&gt; \&amp;yen; \N{ACCOUNT OF} \:snake:"</code>, which returns the string: <code>"‚Ä† ¬• ‚ÑÄ üêç   "</code></p>
<p>Default formatting based on the type of the argument: i.e. <code>f"\(1.23)"</code> returns <code>"1.23"</code>, but <code>f"\%(1.23)"</code> returns <code>"1.230000"</code> based on the default format set up for AbstractFloat types.
See <a href="https://github.com/JuliaString/Format.jl">Format</a> for more information on how to set up defaults for your own types, or to change the defaults for floats, strings, etc.</p>
<p>It also supports C-style formatting, but without having to count the position of the argument: i.e. instead of <code>@sprintf("$name has \$%10.8f", 1.23)"</code>, <code>f"\(name) has $\%10.8f(1.23)"</code></p>
</li>
<li>
<p>Faster and more flexible set of string and character types,
such as <code>ASCIIStr</code>, <code>Latin1Str</code>, <code>UCS2Str</code>, <code>UTF32Str</code>, that are indexed by character,
which can be much easier to use than to have deal with <code>nextind</code> and <code>prevind</code>, etc.
This alleviates a common source of bugs in Julia, where people are unfamiliar with the difference between indexing by the byte or codeunit offset, instead of by the codepoints (characters).
Using these types in your code can help speed things up.</p>
</li>
<li>
<p>Faster and VALIDATED <code>UTF8Str</code> type.
Julia's built-in <code>String</code> type allows storing invalid sequences.
(Strs provides a <code>Text1Str</code> type for holding strings that might be invalid UTF-8, or might be
some other encoding, such as Microsoft's CP1252)
This is especially a problem because the built-in Regex support in Julia incorrectly passes a flag saying that the argument has already been checked and is a valid UTF-8 sequence.  Skipping that check in PCRE2 does make regex matching much faster, however it leaves you open to attacks if your are using regex on unchecked string input.</p>
</li>
<li>
<p>Types for Binary strings, as well as strings that are known to be text strings, but whose encoding is not known (might be UTF-8 with certain commonly accepted but invalid sequence, such as representing characters &gt; uFFFF in 6 bytes, as two 16 bit surrogate characters, or encoding a null byte as <code>\0xc0\0x80</code> instead of <code>\0</code>, or S-JIS, CP1252, etc.)</p>
</li>
<li>
<p>Highly optimized string functions, operating on 2, 4, or 8 characters at a time
(I do intend to optimize these further, by using vector instructions on Intel, ARM, and POWER architectures, to process up to 64 characters at a time).</p>
</li>
<li>
<p>Thread-safe Regex support (it was not thread-safe in the LTS (long term support) version of Julia, currently v1.05, but that has been fixed as of the v1.3 release)</p>
</li>
<li>
<p>Regex support that doesn't assume that <code>String</code> values are valid UTF-8, so that it can't be used as a way of attacking programs written in Julia by passing certain unvalidated strings to the PCRE2 library. For speed, one can use the <code>UTF8Str</code> type instead of <code>String</code> using <code>R"..."</code> instead of the <code>r"..."</code>.</p>
</li>
</ol>
<p>I would very much appreciate any constructive criticism, help implementing some of the ideas, ideas on how to make it perform better, bikeshedding on names and API, etc.
Also, I'd love contributions of benchmark code and/or samples for different use cases of strings,
or pointers to such (such as a way to get lots of tweets, to test mixed text and emojis, for example).</p>

<p>The new package <a href="https://github.com/JuliaString/ModuleInterfaceTools.jl">ModuleInterfaceTools</a> is used to set up a consistent and easy to use API for most of the cooperating packages, without having to worry too much about imports, exports, using, and what functions are part of a public API, and which ones are part of the internal development API for other packages to extend.</p>
<h2><a id="user-content-architecture-and-operations" class="anchor" aria-hidden="true" href="#architecture-and-operations"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Architecture and Operations</h2>
<p>The general philosophy of the architecture is as follows: have a single easy to use type that can replace <code>String</code> that conforms to the recommendations of the Unicode Organization (which internally uses 4 types and is implemented currently as a Julia Union, and has O(1) indexing to characters, not just code units), as well as types to represent binary strings, raw unvalidated strings (made up of 1, 2, or 4 byte codepoints), as well as types for validated ASCII, Latin1, UCS2 (16-bit, BMP [Basic Multilingual Plane]), UTF-8, UTF-16, and UTF-32 encoded strings.</p>
<p>Optimizations for multi code unit encodings such as UTF-8 &amp; UTF-16 will be moved to <code>StrUTF8</code> and <code>StrUTF16</code> packages (before splitting them out, I'll make sure that the functionality still works, only with slower generic methods, so that you only take up the extra space if you need the faster speed).
Extensions such as converting to and from non-Unicode encodings, such as Windows CP-1252 or China's official character set, GB18030,  will be done in another package, <code>StrEncodings</code>.</p>
<p>Subtypes that directly support things like substrings, caching hash values, and caching one or more versions of the string (such as the originally unmodified byte, 16-bit or 32-bit word stream, in the case where the input was not valid, or a valid UTF-8 (similar to the way Python can cache a UTF-8 version of a string) and/or UTF-16 encoded version, for better performance when interoperating with other languages such as JavaScript, Swift, Java, or OS APIs like Windows that expect UTF-16).</p>
<p>Since things like <code>AbstractChar</code> and <code>CodeUnits</code> which I originally implemented for this package have now been added to <code>Base</code> (in master), I have moved support for those to a file to provide them for v0.6.2.
There is a <code>codepoints</code> iterator, which iterates over the unsigned integer codepoints in a string
(for strings with the <code>CodeUnitSingle</code> trait, it is basically the same as the <code>codeunits</code>
iterator).</p>
<p>Also in the works is using the new ability to add properties, in order to allow for different "views" of a string, no matter how it is stored internally, for example a <code>mystring.utf8</code> view, or a <code>mystring.utf16</code> view (that can use the internal cached copy if available, as an optimization).</p>
<h2><a id="user-content-types" class="anchor" aria-hidden="true" href="#types"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Types</h2>
<p>Currently, there are the following types:</p>
<ul>
<li>
<p><code>Str</code>, which is the general, parameterized string type, and</p>
</li>
<li>
<p><code>Chr</code>, the general, parameterized character type.</p>
</li>
<li>
<p><code>BinaryStr</code> for storing non-textual data as a sequence of bytes.</p>
</li>
<li>
<p><code>ASCIIStr</code>   an ASCII string, composed of <code>ASCIIChr</code> 1-byte codepoints</p>
</li>
<li>
<p><code>LatinStr</code>   a string using the Latin1 subset of Unicode, composed of <code>LatinChr</code> 1-byte codepoints.</p>
</li>
<li>
<p><code>UCS2Str</code>    a string composed of characters (<code>UCS2Chr</code>s) only in the Unicode BMP, stored as 2 byte code units (that each store a single codepoint)</p>
</li>
<li>
<p><code>UTF32Str</code>   a string with only valid Unicode characters, 0-0xd7ff, 0xe000-0x10ffff, stored as 4 byte code units.</p>
</li>
<li>
<p><code>UTF8Str</code>    a string with only valid Unicode characters, the same as <code>UTF32Str</code>, however encoded using UTF-8, conforming to the Unicode Organization's standard, i.e. no long encodings, surrogates, or invalid bytes.</p>
</li>
<li>
<p><code>UTF16Str</code>   a string similar to `UTF8Str, encoded via UTF-16, also conforming to the Unicode standard, i.e. no out of order or isolated surrogates.</p>
</li>
<li>
<p><code>Text1Str</code>   a text string that may contain any sequence of bytes</p>
</li>
<li>
<p><code>Text2Str</code>   a text string that may contain any sequence of 16-bit words</p>
</li>
<li>
<p><code>Text4Str</code>   a text string that may contain any sequence of 32-bit words</p>
</li>
<li>
<p><code>UniStr</code>     a Union type, that can be one of the following 4 types, <code>ASCIIStr</code>, and 3 internal types:</p>
</li>
<li>
<p><code>_LatinStr</code>  a byte string that must contain at least one character &gt; 0x7f</p>
</li>
<li>
<p><code>_UCS2Str</code>   a word string that must contain at least one character &gt; 0xff</p>
</li>
<li>
<p><code>_UTF32Str</code>  a 32-bit word string that must contain at least one character &gt; 0xffff</p>
</li>
</ul>
<p>The only real difference in handling <code>LatinStr</code> and <code>_LatinStr</code>, is that uppercasing the characters <code>'¬µ': (Unicode U+00b5 (category Ll: Letter, lowercase)</code> and <code>'√ø': Unicode U+00ff (category Ll: Letter, lowercase)</code> produces the BMP characters <code>'Œú': Unicode U+039c (category Lu: Letter, uppercase)</code> and <code>'≈∏': Unicode U+0178 (category Lu: Letter, uppercase)</code> respectively in <code>_LatinStr</code>, because it is just an optimization for storing the full Unicode character set, not the ANSI/ISO 8859-1 character set that ise used for the first 256 code points of Unicode.
Those three internal types should never be used directly, as indicated by the leading <code>_</code> in the name.</p>
<p>For all of the built-in <code>Str</code> types, there is a corresponding built-in character set encoding, i.e. <code>BinaryCSE</code>, <code>LatinCSE</code>, <code>UTF8CSE</code>, etc.
There are also a number of similar built-in character sets defined (<code>*CharSet</code>), and character types (<code>*Chr</code>).
The <code>cse</code> function returns the character set encoding for a string type or a string.
<code>charset</code> returns the character set, and <code>encoding</code> returns the encoding.
For example, <code>cse(UTF8Str)</code> returns <code>UTF8CSE</code>, <code>charset(UTF8Str)</code> returns <code>CharSet{UTF32}</code>,
<code>encoding(UTF8Str)</code> return <code>Encoding{UTF8}()</code></p>

<p>There is a new API that I am working on for indexing and searching, (however there is a problem on v0.7 due to the deprecation for <code>find</code> being overbroad, and causing a type of type piracy, deprecating methods of types not in Base):</p>
<ul>
<li>
<p><code>find(First, ...)</code> replaces <code>findfirst</code></p>
</li>
<li>
<p><code>find(Last, ...)</code>  replaces <code>findlast</code></p>
</li>
<li>
<p><code>find(Next, ...)</code>  replaces <code>findnext</code></p>
</li>
<li>
<p><code>find(Prev, ...)</code>  replaces <code>findprev</code></p>
</li>
<li>
<p><code>index(str, pos)</code>  replaces <code>thisind(str, pos)</code></p>
</li>
<li>
<p><code>index(Next, ...</code>  replaces <code>nextind(...)</code></p>
</li>
<li>
<p><code>index(Prev, ...</code>  replaces <code>prevind(...)</code></p>
</li>
</ul>
<p>Also there are more readable function names that always separate words with <code>_</code>, and avoid hard to understand abbreviations:</p>
<ul>
<li>
<p><code>is*</code>              -&gt; <code>is_*</code> (for <code>ascii</code>, <code>digit</code>, <code>space</code>, <code>numeric</code>, <code>valid</code>,
<code>defined</code>, <code>empty</code>, <code>assigned</code>)</p>
</li>
<li>
<p><code>iscntrl</code>          -&gt; <code>is_control</code></p>
</li>
<li>
<p><code>isgraph</code>          -&gt; <code>is_graphic</code></p>
</li>
<li>
<p><code>isprint</code>          -&gt; <code>is_printable</code></p>
</li>
<li>
<p><code>ispunct</code>          -&gt; <code>is_punctuation</code></p>
</li>
<li>
<p><code>isalpha</code>          -&gt; <code>is_letter</code></p>
</li>
<li>
<p><code>isalnum</code>          -&gt; <code>is_alphanumeric</code></p>
</li>
<li>
<p><code>isgraphemebreak</code>  -&gt; <code>is_grapheme_break</code></p>
</li>
<li>
<p><code>isgraphemebreak!</code> -&gt; <code>is_grapheme_break!</code></p>
</li>
<li>
<p><code>occursin</code>         -&gt; <code>occurs_in</code></p>
</li>
<li>
<p><code>textwidth</code>        -&gt; <code>text_width</code></p>
</li>
<li>
<p><code>lowercasefirst</code>   -&gt; <code>lowercase_first</code></p>
</li>
<li>
<p><code>uppercasefirst</code>   -&gt; <code>uppercase_first</code></p>
</li>
<li>
<p><code>startswith</code>       -&gt; <code>starts_with</code></p>
</li>
<li>
<p><code>endswith</code>         -&gt; <code>ends_with</code></p>
</li>
<li>
<p>In addition, I've added <code>is_alphabetic</code></p>
</li>
