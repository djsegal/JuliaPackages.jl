<p>It represents an attempt to give Julia better string handling than possible with Base <code>String</code> and <code>Char</code>.</p>
<p>I am now trying to make sure that all of the functionality in String and Char is implemented for
Str and Chr, and to start optimizing the functions (although they are already substantially faster)</p>
<p>Strs.jl is now a container for a number of different packages from <a href="http://juliastring.org" rel="nofollow">JuliaString.org</a></p>

<p>The new package <a href="https://github.com/JuliaString/ModuleInterfaceTools.jl">ModuleInterfaceTools</a> is used to set up a consistent and easy to use API for most of the cooperating packages, without having to worry too much about imports, exports, using, and what functions are part of a public API, and which ones are part of the internal development API for other packages to extend.</p>
<p>I would very much appreciate any constructive criticism, help implementing some of the ideas, ideas on how to make it perform better, bikeshedding on names and API, etc.
Also, I'd love contributions of benchmark code and/or samples for different use cases of strings,
or pointers to such (such as a way to get lots of tweets, to test mixed text and emojis, for example).</p>
<h2><a id="user-content-architecture-and-operations" class="anchor" aria-hidden="true" href="#architecture-and-operations"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Architecture and Operations</h2>
<p>The general philosophy of the architecture is as follows: have a single easy to use type that can replace <code>String</code> that conforms to the recommendations of the Unicode Organization (which internally uses 4 types and is implemented currently as a Julia Union, and has O(1) indexing to characters, not just code units), as well as types to represent binary strings, raw unvalidated strings (made up of 1, 2, or 4 byte codepoints), as well as types for validated ASCII, Latin1, UCS2 (16-bit, BMP [Basic Multilingual Plane]), UTF-8, UTF-16, and UTF-32 encoded strings.</p>
<p>Optimizations for multi code unit encodings such as UTF-8 &amp; UTF-16 will be moved to <code>StrUTF8</code> and <code>StrUTF16</code> packages (before splitting them out, I'll make sure that the functionality still works, only with slower generic methods, so that you only take up the extra space if you need the faster speed).
Extensions such as converting to and from non-Unicode encodings, such as Windows CP-1252 or China's official character set, GB18030,  will be done in another package, <code>StrEncodings</code>.</p>
<p>Subtypes that directly support things like substrings, caching hash values, and caching one or more versions of the string (such as the originally unmodified byte, 16-bit or 32-bit word stream, in the case where the input was not valid, or a valid UTF-8 (similar to the way Python can cache a UTF-8 version of a string) and/or UTF-16 encoded version, for better performance when interoperating with other languages such as JavaScript, Swift, Java, or OS APIs like Windows that expect UTF-16).</p>
<p>Since things like <code>AbstractChar</code> and <code>CodeUnits</code> which I originally implemented for this package have now been added to <code>Base</code> (in master), I have moved support for those to a file to provide them for v0.6.2.
There is a <code>codepoints</code> iterator, which iterates over the unsigned integer codepoints in a string
(for strings with the <code>CodeUnitSingle</code> trait, it is basically the same as the <code>codeunits</code>
iterator).</p>
<p>Also in the works is using the new ability to add properties, in order to allow for different "views" of a string, no matter how it is stored internally, for example a <code>mystring.utf8</code> view, or a <code>mystring.utf16</code> view (that can use the internal cached copy if available, as an optimization).</p>
<h2><a id="user-content-types" class="anchor" aria-hidden="true" href="#types"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Types</h2>
<p>Currently, there are the following types:</p>
<ul>
<li>
<p><code>Str</code>, which is the general, parameterized string type, and</p>
</li>
<li>
<p><code>Chr</code>, the general, parameterized character type.</p>
</li>
<li>
<p><code>BinaryStr</code> for storing non-textual data as a sequence of bytes.</p>
</li>
<li>
<p><code>ASCIIStr</code>   an ASCII string, composed of <code>ASCIIChr</code> 1-byte codepoints</p>
</li>
<li>
<p><code>LatinStr</code>   a string using the Latin1 subset of Unicode, composed of <code>LatinChr</code> 1-byte codepoints.</p>
</li>
<li>
<p><code>UCS2Str</code>    a string composed of characters (<code>UCS2Chr</code>s) only in the Unicode BMP, stored as 2 byte code units (that each store a single codepoint)</p>
</li>
<li>
<p><code>UTF32Str</code>   a string with only valid Unicode characters, 0-0xd7ff, 0xe000-0x10ffff, stored as 4 byte code units.</p>
</li>
<li>
<p><code>UTF8Str</code>    a string with only valid Unicode characters, the same as <code>UTF32Str</code>, however encoded using UTF-8, conforming to the Unicode Organization's standard, i.e. no long encodings, surrogates, or invalid bytes.</p>
</li>
<li>
<p><code>UTF16Str</code>   a string similar to `UTF8Str, encoded via UTF-16, also conforming to the Unicode standard, i.e. no out of order or isolated surrogates.</p>
</li>
<li>
<p><code>Text1Str</code>   a text string that may contain any sequence of bytes</p>
</li>
<li>
<p><code>Text2Str</code>   a text string that may contain any sequence of 16-bit words</p>
</li>
<li>
<p><code>Text4Str</code>   a text string that may contain any sequence of 32-bit words</p>
</li>
<li>
<p><code>UniStr</code>     a Union type, that can be one of the following 4 types, <code>ASCIIStr</code>, and 3 internal types:</p>
</li>
<li>
<p><code>_LatinStr</code>  a byte string that must contain at least one character &gt; 0x7f</p>
</li>
<li>
<p><code>_UCS2Str</code>   a word string that must contain at least one character &gt; 0xff</p>
</li>
<li>
<p><code>_UTF32Str</code>  a 32-bit word string that must contain at least one character &gt; 0xffff</p>
</li>
</ul>
<p>The only real difference in handling <code>LatinStr</code> and <code>_LatinStr</code>, is that uppercasing the characters <code>'µ': (Unicode U+00b5 (category Ll: Letter, lowercase)</code> and <code>'ÿ': Unicode U+00ff (category Ll: Letter, lowercase)</code> produces the BMP characters <code>'Μ': Unicode U+039c (category Lu: Letter, uppercase)</code> and <code>'Ÿ': Unicode U+0178 (category Lu: Letter, uppercase)</code> respectively in <code>_LatinStr</code>, because it is just an optimization for storing the full Unicode character set, not the ANSI/ISO 8859-1 character set that ise used for the first 256 code points of Unicode.
Those three internal types should never be used directly, as indicated by the leading <code>_</code> in the name.</p>
<p>For all of the built-in <code>Str</code> types, there is a corresponding built-in character set encoding, i.e. <code>BinaryCSE</code>, <code>LatinCSE</code>, <code>UTF8CSE</code>, etc.
There are also a number of similar built-in character sets defined (<code>*CharSet</code>), and character types (<code>*Chr</code>).
The <code>cse</code> function returns the character set encoding for a string type or a string.
<code>charset</code> returns the character set, and <code>encoding</code> returns the encoding.
For example, <code>cse(UTF8Str)</code> returns <code>UTF8CSE</code>, <code>charset(UTF8Str)</code> returns <code>CharSet{UTF32}</code>,
<code>encoding(UTF8Str)</code> return <code>Encoding{UTF8}()</code></p>
<h2><a id="user-content-functions" class="anchor" aria-hidden="true" href="#functions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Functions</h2>
<p>There is a new API that I am working on for indexing and searching, (however there is a problem on v0.7 due to the deprecation for <code>find</code> being overbroad, and causing a type of type piracy, deprecating methods of types not in Base):</p>
<ul>
<li>
<p><code>find(First, ...)</code> replaces <code>findfirst</code></p>
</li>
<li>
<p><code>find(Last, ...)</code>  replaces <code>findlast</code></p>
</li>
<li>
<p><code>find(Next, ...)</code>  replaces <code>findnext</code></p>
</li>
<li>
<p><code>find(Prev, ...)</code>  replaces <code>findprev</code></p>
</li>
<li>
<p><code>index(str, pos)</code>  replaces <code>thisind(str, pos)</code></p>
</li>
<li>
<p><code>index(Next, ...</code>  replaces <code>nextind(...)</code></p>
</li>
<li>
<p><code>index(Prev, ...</code>  replaces <code>prevind(...)</code></p>
</li>
</ul>
<p>Also there are more readable function names that always separate words with <code>_</code>, and avoid hard to understand abbreviations:</p>
<ul>
