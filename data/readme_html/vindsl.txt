<h2><a id="user-content-a-variational-inference-domain-specific-language" class="anchor" aria-hidden="true" href="#a-variational-inference-domain-specific-language"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>A Variational Inference Domain-Specific Language</h2>
<p><a href="https://en.wikipedia.org/wiki/Variational_Bayesian_methods" rel="nofollow">Variational inference</a> is an approximate method of statistical inference based on optimization. Unlike conventional Bayesian methods based on <a href="https://en.wikipedia.org/wiki/Markov_chain_Monte_Carlo" rel="nofollow">Markov Chain Monte Carlo</a> (MCMC), it scales well to large and streaming datasets, making it a competitive technique for machine learning applications.</p>
<p>However, coding variational inference models by hand traditionally involves lots of tedious algebra and careful index accounting. New techniques like <a href="http://www.cs.columbia.edu/~blei/papers/RanganathGerrishBlei2014.pdf" rel="nofollow">black-box variational inference</a> and <a href="http://papers.nips.cc/paper/5678-local-expectation-gradients-for-black-box-variational-inference" rel="nofollow">local expectation gradients</a> allow much of this to be avoided, and implementations for some models are already possible in <a href="http://mc-stan.org/" rel="nofollow">Stan</a>, but no current framework allows these rapidly developing techniques to be mixed and matched by researchers.</p>
<p>The goal of VinDsl.jl is to provide a set of data abstractions and macros that take the pain out of coding variational inference models. In particular, because the syntactic sugar for defining models is implemented in the same language as the underlying building blocks, the entire framework is easily extensible and hackable.</p>
<h3><a id="user-content-things-we-have" class="anchor" aria-hidden="true" href="#things-we-have"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Things we have:</h3>
<ul>
<li>Intelligent index handling: you define the model structure, VinDsl handles the sum over indices automatically</li>
<li>A set of macros for coding conjugate models and updates</li>
<li>Limited support for automatic expectation-taking</li>
<li>built-in support for Hidden Markov Models</li>
<li>preliminary support for <a href="http://arxiv.org/abs/1603.00788" rel="nofollow">ADVI</a></li>
</ul>
