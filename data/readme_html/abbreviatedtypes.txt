<p dir="auto"><strong>AbbreviatedTypes</strong> defines compact type aliases of frequently used types with long names, e.g., <code>AbsVecFloat</code> for <code>AbstractVector{Float64}</code>.  The "abbreviated" type names are useful in making function signatures more compact.</p>
<h2 dir="auto"><a id="user-content-change-since-version-03" class="anchor" aria-hidden="true" href="#change-since-version-03"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Change since Version 0.3</h2>
<p dir="auto">Before Version 0.3, AbbreviatedTypes defined the type aliases internally and exported them so that they can be used in the package importing AbbreviatedTypes.  Since Version 0.3, AbbreviatedTypes does not define the type aliases internally; instead, it defines the type aliases directly in the packages importing AbbreviatedTypes.  This allows the users to tailor the behavior of AbbreviatedTypes for individual packages importing it.  Specifically, the users can define <code>Float</code> as any of the concrete subtypes of <code>AbstractFloat</code>, i.e., <code>Float16</code>, <code>Float32</code>, or <code>Float64</code>.  This is useful when you want to use <code>AbsVecFloat</code> as an abstract vector of <code>Float32</code> in one package (e.g., written for GPU) and of <code>Float64</code> in another package.  The different definitions of <code>Float</code> in different packages do not conflict with one another because they live in different namespaces.</p>

<p dir="auto">After <code>using AbbreviatedTypes</code>, call the macro <code>@define_types_with()</code> with the concrete subtype of <code>AbstractFloat</code>.  For example,</p>

<p dir="auto">The technique of using a macro to inject definitions in a calling module is discussed in <a href="https://discourse.julialang.org/t/get-the-name-of-the-invoking-module/22685" rel="nofollow">this discourse thread</a>.</p>
<p dir="auto">For the definitions of available type aliases, see the code <a href="https://github.com/wsshin/AbbreviatedTypes.jl/blob/main/src/AbbreviatedTypes.jl#L29-L159">here</a>.  The usage of the type aliases should be trivial, but I would like to note the special usage of the aliases of <a href="https://github.com/JuliaArrays/StaticArrays.jl"><code>StaticArray</code></a> types as constructors.  For example, <code>SInt{K}</code> is an alias of <code>SVector{K,Int}</code>, but <code>SInt</code> can be also used as a constructor without specifying the length <code>K</code>:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="x1 = SInt{3}(1, 2, 3)  # [1, 2, 3]

<p dir="auto">In double-precision floating point arithmetic, <code>eps(Float64)</code> and <code>Base.rtoldefault(Float64)</code> are commonly used as absolute and relative error tolerances.  In addition to type aliases, AbbreviatedTypes defines <code>τₐ₀</code> and <code>τᵣ₀</code> as <code>eps(Float)</code> and <code>Base.rtoldefault(Float)</code>, so that they can be used as the absolute and relative error tolerances for the user-defined <code>Float</code> type through <code>@define_types_with()</code>.</p>
