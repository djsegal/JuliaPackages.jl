<p><a href="https://github.com/JuliaOpt/JuMP.jl">JuMP</a> extensions for constraint programming.</p>
<p>These extensions rely on <a href="https://github.com/dourouc05/ConstraintProgrammingExtensions.jl">ConstraintProgrammingExtensions</a>, an extension for <a href="https://github.com/JuliaOpt/MathOptInterface.jl">MathOptInterface.jl</a> providing several constraint-programming-oriented sets. There is already one solver providing some of these new sets: <a href="https://github.com/dourouc05/CPLEXCP.jl">CPLEXCP.jl</a>.</p>
<p>For now, the new syntax can only be used with a patch applied on top of JuMP: <a href="https://github.com/JuliaOpt/JuMP.jl/pull/2241">PR 2241</a>.</p>
<h2><a id="user-content-design-considerations" class="anchor" aria-hidden="true" href="#design-considerations"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Design considerations</h2>
<p>The goal of these series of packages is to provide access to constraint programming (CP) within Julia, considering that CP is barely unavailable in the ecosystem, the exception being <a href="https://github.com/Wikunia/ConstraintSolver.jl">ConstraintSolver.jl</a>.</p>
<p>The CP environment is very different from the mathematical programming one (and it is not related to the lack of dual variables in CP solvers): the functionalities provided by the solvers do not easily match (some provide reified constraints, others equivalence constraints; some solvers provide specific expressions like <code>count(…)</code>, that may be then used in the objective function, others only support constraints like <code>x &lt;= count(…)</code> or <code>x == count(…)</code>). You can express the same constraints with all solvers, but not always in the same way. Moreover, solvers tend not to provide callable libraries, unlike optimisation solvers: they provide higher-level APIs, usually in languages such as C++ or Java. This means that, for closed-source solvers, there is no access to the solver, only to a modelling layer (like CPLEX CP Optimizer). Similarly, many solvers are written in Java, and thus require an access to the JVM to be run.</p>
<p>Due to these differences, the rationale behind the sets provided by <a href="https://github.com/dourouc05/ConstraintProgrammingExtensions.jl">ConstraintProgrammingExtensions.jl</a> is to propose all new elements as constraints, and not functions. Solvers only supporting the constraint <code>count(…)</code> in the form <code>x == count(…)</code> directly match these definitions (for instance, Gecode). Others provide expressions for this constraints (like CPLEX CP Optimizer), but they can be used in constraints to match the same semantics. In other words, the goal is to provide a set of "flat" primitives that is supported by a large number of solvers.</p>
<p>This is similar to what MiniZinc provides, except JuCP &amp; co. work on the code level. <a href="https://www.minizinc.org/" rel="nofollow">MiniZinc</a> is a common descriptive language for CP that is independently implemented by several solvers (like <a href="https://www.gecode.org/flatzinc.html" rel="nofollow">Gecode</a>, <a href="https://github.com/radsz/jacop">JaCoP</a>, <a href="https://github.com/chocoteam/choco-solver">Choco</a>). Under the hood, MiniZinc transforms the input file in FlatZinc, a lower-level programming language (a kind of intermediate representation), which the solver then reads. This is similar to AMPL's NL format (in time, maybe <a href="https://github.com/JuliaOpt/AmplNLWriter.jl">AmplNLWriter.jl</a> should be extended to handle CP extensions). <a href="https://savilerow.cs.st-andrews.ac.uk/index.html" rel="nofollow">Savile Row</a> and <a href="https://github.com/conjure-cp/conjure">Conjure</a> are similar in principle to MiniZinc, with the definition of a high-level language to describe the models (Essence), which is lowered by Conjure (Essence'); these two pieces of software also optimise the models, which is not, currently, a goal of JuCP &amp; co.; again, communication is mostly made through files. On the other hand, JuCP &amp; co. work by directly using the solvers' API, similarly to how other solvers work with JuMP/MOI. This approach has also been chosen by <a href="http://eclipseclp.org/" rel="nofollow">ECLiPSe</a>, for instance <a href="https://github.com/antiguru/eclipse-clp/blob/af37a9ef7506f0eb05c4dba3b862241d1f5903e3/GecodeInterface/gfd.cpp">to communicate with Gecode</a>.</p>
<p>In the specific case of CPLEX CP Optimizer, albeit the solver itself is written in C++ and available through a C++ API, this way has not been chosen. Actually, the C++ library that IBM gives is only a static library, which is a nightmare to convert into a standard dynamic library. Furthermore, this process requires many platform-specific tools to do so (mostly, the linker and many <code>.lib</code> library files for the dependencies: Clang.jl does not provide access to the linker, the dependencies require a core Visual C++ installation; these dependencies are heavier than the JVM). Another solution would have been to write a callable library on top of CPLEX Concert, which was deemed too heavy for a prototype (maybe IBM may help on this side?).</p>

<p>In approximate order of importance:</p>
<ul>
<li>Have <a href="https://github.com/JuliaOpt/JuMP.jl/pull/2051">PR 2051</a> merged into JuMP to allow more syntax extensions.</li>
<li>Complete the solver wrapper <a href="https://github.com/dourouc05/CPLEXCP.jl">CPLEXCP.jl</a>. This means (again, in approximate order of importance):
<ul>
<li>improving the test suite (see the next point)</li>
<li>adding the missing constraints</li>
<li>wrap the missing parts, especially tuning the search phase</li>
<li>study how to integrate it more with Julia, for instance <a href="https://developer.ibm.com/docloud/blog/2019/12/17/new-callback-functionality-in-cp-optimizer/" rel="nofollow">callbacks</a></li>
</ul>
</li>
<li>Decide what to do for testing the wrappers. Many tests in <a href="https://github.com/JuliaOpt/MathOptInterface.jl">MathOptInterface.jl</a> do not apply, as the solvers are severely limited when it comes to continuous problems; also, they don't provide dual values. Changing variable types is often not possible, which implies that variables must be created with their constraint (<code>add_constrained_variable</code> instead of <code>add_variable</code> plus <code>add_constraint</code>). This probably calls for simplifying methods in <a href="https://github.com/dourouc05/ConstraintProgrammingExtensions.jl">ConstraintProgrammingExtensions.jl</a> to only call the relevant tests in <a href="https://github.com/JuliaOpt/MathOptInterface.jl">MathOptInterface.jl</a>, plus providing a lot of new tests.</li>
<li>Determine how to expose more basic functionalities from solvers, like interval variables.</li>
<li>Determine how to expose more advanced functionalities from solvers, like <a href="https://www.ibm.com/support/knowledgecenter/SSSA5P_12.10.0/ilog.odms.cpo.help/CP_Optimizer/Advanced_user_manual/topics/propagator_example.html" rel="nofollow">custom propagators</a>, <a href="https://www.ibm.com/support/knowledgecenter/SSSA5P_12.10.0/ilog.odms.cpo.help/CP_Optimizer/Advanced_user_manual/topics/csts.html" rel="nofollow">custom constraints</a>, <a href="https://www.ibm.com/support/knowledgecenter/SSSA5P_12.10.0/ilog.odms.cpo.help/CP_Optimizer/Advanced_user_manual/topics/goals_overview.html" rel="nofollow">custom search techniques</a>. <a href="https://arxiv.org/abs/1203.1095" rel="nofollow">Search Combinators</a> can be a good start.</li>
<li>Write wrappers for more solvers. The major ones are:
<ul>
<li><a href="https://www.gecode.org/" rel="nofollow">Gecode</a>. In C++, models must inherit the Script class; have a look at their FlatZinc implementation, more precisely <a href="https://github.com/Gecode/gecode/blob/master/gecode/flatzinc/parser.hh#L184">ParserState</a>?</li>
<li><a href="https://github.com/radsz/jacop">JaCoP</a>. Highly similar to CPLEX CP in Java, [no need to inherit from a given class](<a href="https://github.com/radsz/jacop/blob/develop/src/main/java/org/jacop/examples/fd/ExampleFD.java">https://github.com/radsz/jacop/blob/develop/src/main/java/org/jacop/examples/fd/ExampleFD.java</a>.</li>
<li><a href="https://github.com/chocoteam/choco-solver">Choco</a>, mostly the same.</li>
</ul>
</li>
</ul>
