<h2 dir="auto"><a id="user-content-purpose" class="anchor" aria-hidden="true" href="#purpose"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Purpose</h2>
<p dir="auto">StanSample.jl wraps <code>cmdstan</code>'s <code>sample</code> method to generate draws from a Stan Language Program. It is the primary workhorse in the StanJulia ecosystem.</p>
<p dir="auto">StanSample.jl v7 supports InferenceObjects.jl as a package extension. Use <code>inferencedata(model)</code> to create an InferenceData object. See also note 1 below. An example Pluto notebook can be found <a href="https://github.com/StanJulia/StanExampleNotebooks.jl/blob/main/notebooks/DimensionalData/dimensionaldata.jl">here</a></p>

<ol dir="auto">
<li>
<p dir="auto">Use of both InferenceObjects.jl and the <code>read_samples()</code> output_format options :dimarray and :dimarrays (based on DimensionalData.jl) creates a conflict. Hence these output_format options are no longer included. See the example Pluto notebook <code>test_dimarray.jl</code>in <a href="https://github.com/StanJulia/StanExampleNotebooks.jl">StanExampleNotebooks.jl</a> for an example how to still use that option. At some point in time InferenceObjects.jl might provide an alternative way to create a stacked DataFrame and/or DimensionalData object.</p>
</li>
<li>
<p dir="auto">I've removed BridgeStan.jl from StanSample.jl. Two example Pluto notebooks,  <code>test_bridgestan.jl</code> and
<code>bridgestan_stansample_example.jl</code> in <a href="https://github.com/StanJulia/StanExampleNotebooks.jl/tree/main/notebooks/BridgeStan">StanExampleNotebooks.jl</a> demonstrate how BridgeStan can be used.</p>
</li>
</ol>

<p dir="auto">You need a working installation of <a href="https://mc-stan.org/users/interfaces/cmdstan.html" rel="nofollow">Stan's cmdstan</a>, the path of which you should specify in either <code>CMDSTAN</code> or <code>JULIA_CMDSTAN_HOME</code>, e.g. in your <code>~/.julia/config/startup.jl</code> include a line like:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="# CmdStan setup
ENV[&quot;CMDSTAN&quot;] =
     expanduser(&quot;~/.../cmdstan/&quot;) # replace with your path"><pre><span class="pl-c"><span class="pl-c">#</span> CmdStan setup</span>
<span class="pl-c1">ENV</span>[<span class="pl-s"><span class="pl-pds">"</span>CMDSTAN<span class="pl-pds">"</span></span>] <span class="pl-k">=</span>
     <span class="pl-c1">expanduser</span>(<span class="pl-s"><span class="pl-pds">"</span>~/.../cmdstan/<span class="pl-pds">"</span></span>) <span class="pl-c"><span class="pl-c">#</span> replace with your path</span></pre></div>
<p dir="auto">Or you can define and export CMDSTAN in your .profile, .bashrc, .zshrc, etc.</p>
<p dir="auto">For more details see <a href="https://github.com/StanJulia/StanSample.jl/blob/master/INSTALLING_CMDSTAN.md">this file</a>.</p>
<p dir="auto">See the <code>example/bernoulli.jl</code> for a basic example. Many more examples and test scripts are available in this package and also in Stan.jl.</p>
<h2 dir="auto"><a id="user-content-multi-threading-and-multi-chaining-behavior" class="anchor" aria-hidden="true" href="#multi-threading-and-multi-chaining-behavior"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Multi-threading and multi-chaining behavior.</h2>
<p dir="auto">From StanSample.jl v6 onwards 2 mechanisms for in paralel drawing samples for chains are supported, i.e. on C++ level (using threads) and on Julia level (by spawning a Julia process for each chain).</p>
<p dir="auto">The <code>use_cpp_chains</code> keyword argument in the call to <code>stan_sample()</code> determines if chains are executed on C++ level or on Julia level. By default, <code>use_cpp_chains = false</code>.</p>
<p dir="auto">From cmdstan-2.28.0 onwards it is possible to use C++ threads to run multiple chains by setting <code>use_cpp_chains=true</code> in the call to <code>stan_sample()</code>:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="rc = stan_sample(_your_model_; use_cpp_chains=true, [ data | init | ...])"><pre class="notranslate"><code>rc = stan_sample(_your_model_; use_cpp_chains=true, [ data | init | ...])
</code></pre></div>
<p dir="auto">To enable multithreading in <code>cmdstan</code> specify this before the build process of <code>cmdstan</code>, i.e. before running <code>make -j9 build</code>. I typically create a <code>path_to_my_cmdstan_directory/make/local</code> file containing <code>STAN_THREADS=true</code>. You can see an example in <code>.github/CI.yml</code> script.</p>
<p dir="auto">By default in either case <code>num_chains=4</code>. See <code>??stan_sample</code> for all keyword arguments. Internally, <code>num_chains</code> will be copied to either <code>num_cpp_chains</code> or <code>num_julia_chains</code>.</p>
<p dir="auto">Currently I do not suggest to use both C++ and Julia level chains. Based on the value of <code>use_cpp_chains</code> (true or false) the <code>stan_sample()</code> method will set either <code>num_cpp_chains=num_chains; num_julia_chains=1</code> or <code>num_julia_chains=num_chains;num_cpp_chain=1</code>.</p>
<p dir="auto">This default behavior can be disabled by setting the postional <code>check_num_chains</code> argument in the call to <code>stan_sample()</code> to <code>false</code>.</p>
<p dir="auto">Threads on C++ level can be used in multiple ways, e.g. to run separate chains and to speed up certain operations. By default StanSample.jl's SampleModel sets the C++ num_threads to 4.</p>
<p dir="auto">See the (updated for cmdstan-2.29.0) RedCardsStudy example <a href="https://github.com/StanJulia/Stan.jl/tree/master/Examples/RedCardsStudy/graphs">graphs</a> in Stan.jl and <a href="https://discourse.mc-stan.org/t/stan-num-threads-and-num-threads/25780/5?u=rob_j_goedman" rel="nofollow">here</a> for more details, in particular with respect to just enabling threads and including TBB or not on Intel, and also some indications of the performance on an Apple's M1/ARM processor running native (not using Rosetta and without Intel's TBB).</p>
<p dir="auto">In some cases I have seen performance advantages using both Julia threads and C++ threads but too many combined threads certainly doesn't help. Note that if you only want 1000 draws (using 1000 warmup samples for tuning), multiple chains (C++ or Julia) do not help.</p>
