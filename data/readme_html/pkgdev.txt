<h3><a id="user-content-pkgdevtagpackage_name-versionnothing-registrynothing-release_notesnothing" class="anchor" aria-hidden="true" href="#pkgdevtagpackage_name-versionnothing-registrynothing-release_notesnothing"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>PkgDev.tag(package_name, version=nothing; registry=nothing, release_notes=nothing)</h3>
<p>Tag a new release for package <code>package_name</code>. The package you want to tag must be deved in the current Julia environment. You pass the package name <code>package_name</code> as a <code>String</code>. The git commit that is the <code>HEAD</code> in the package folder will form the basis for the version to be tagged.</p>
<p>If you don't specify a <code>version</code>, then the <code>version</code> field in the <code>Project.toml</code> <em>must</em> have the format <code>x.y.z-DEV</code>, and the command will tag version <code>x.y.z</code> as the next release. Alternatively you can specify one of <code>:major</code>, <code>:minor</code> or <code>:patch</code> for the <code>version</code> parameter. In that case <code>PkgDev.tag</code> will increase that part of the version number by 1 and tag that version. Finally, you can also specify a full <code>VersionNumber</code> as the value for the <code>version</code> parameter, in which case that version will be tagged.</p>
<p>The only situation where you would specify a value for <code>registry</code> is when you want to register a new package for the first time in a registry that is not <code>General</code>. In all other situations, <code>PkgDev.tag</code> will automatically figure out in which registry your package is registered. When you do pass a value for <code>registry</code>, it should simply be the short name of a registry that is one of the registries your local system is connected with.</p>
<p>If you want to add custom release notes for <a href="https://github.com/JuliaRegistries/TagBot">TagBot</a>, do so with the <code>release_notes</code> keyword.</p>
<p><code>PkgDev.tag</code> runs through the following process when it tags a new version:</p>
<ol>
<li>Create a new release branch called <code>release-x.y.z</code>.</li>
<li>Change the version field in <code>Project.toml</code> and commit that change on the release branch.</li>
<li>Change the version field in <code>Project.toml</code> to <code>x.y.z+1-DEV</code> and commit that change also to the release branch.
4a. For packages in the General registry: add a comment that triggers <a href="https://github.com/JuliaRegistries/Registrator.jl">Registrator</a>.
4b. For packages in other registries: Open a pull request against the registry that tags the first new commit on the release branch as a new version <code>x.y.z</code>.</li>
<li>Open a pull request against the package repository to merge the release branch into <code>master</code>.</li>
</ol>
<p>If you have <a href="https://github.com/JuliaRegistries/TagBot">TagBot</a> installed for your package with the <code>branches: true</code> setting, it will automatically merge the <code>release-x.y.z</code> branch into <code>master</code> once the pull request for the registry has been merged. If you use the package butler (desribed below) it auto-configures your repository for this workflow.</p>
<h3><a id="user-content-pkgdevenable_pkgbutlerpackage_name-channelauto-templateauto" class="anchor" aria-hidden="true" href="#pkgdevenable_pkgbutlerpackage_name-channelauto-templateauto"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>PkgDev.enable_pkgbutler(package_name; channel=:auto, template=:auto)</h3>
<p>Enables the Julia Package Butler for package <code>package_name</code>. The package must be deved in the current Julia environment. The command will make various modifications to the files in the deved folder of the package. You then then need to commit these changes and push them to GitHub. The command will also add a deploy key to the GitHub repository of the package and show instructions on how to add two GitHub Actions secrets to the repository.</p>
<p>The <code>channel</code> argument can be <code>:auto</code>, <code>:stable</code> or <code>:dev</code>. There are two channels of updates: <code>stable</code> and <code>dev</code>. The <code>dev</code> channel will run the Julia Package Butler workflow every 5 minutes and it will use the <code>master</code> branch of the <a href="https://github.com/davidanthoff/PkgButlerEngine.jl">Julia Packge Butler engine</a>, i.e. it will get new features more quickly. The <code>stable</code> branch  runs the Julia Package Butler workflow every hour, and new features in the Julia Package Butler engine are only pushed out to the stable channel once they have been tested for a while on the <code>dev</code> channel. If you specify <code>:auto</code> as the argument, any existing channel choice you have previously made for the package will be retained, otherwise the package will be configure for the <code>stable</code> channel.</p>
<p>The <code>template</code> argument can be <code>:auto</code>, <code>:default</code> or <code>:bach</code>. Different templates will configure different aspects of your package. At this point everyone should use the <code>:default</code> template (or <code>:auto</code> template), everything else is considered experimental.</p>

<p>Switch the Julia Package Butler channel for package <code>package_name</code>. The package you want to tag must be deved in the current Julia environment and the Julia Package Butler must already be enabled for the package. The <code>channel</code> argument can be <code>:auto</code>, <code>:stable</code> or <code>:dev</code>, see the documentation for <code>PkgDev.enable_pkgbutler</code> for an explanation of the different channels.</p>
