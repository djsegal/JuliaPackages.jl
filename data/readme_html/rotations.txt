<p><em>3D rotations made easy in Julia</em></p>
<p><a href="https://github.com/JuliaGeometry/Rotations.jl/actions?query=workflow%3ACI"><img src="https://github.com/JuliaGeometry/Rotations.jl/workflows/CI/badge.svg" alt="Build Status" style="max-width:100%;"></a>
<a href="https://codecov.io/gh/JuliaGeometry/Rotations.jl" rel="nofollow"><img src="https://camo.githubusercontent.com/60de510bc92c6dc3207e66ce0c2f08ed66cf5c203d67f0344dca06738a11cff2/68747470733a2f2f636f6465636f762e696f2f67682f4a756c696147656f6d657472792f526f746174696f6e732e6a6c2f6272616e63682f6d61737465722f67726170682f62616467652e737667" alt="Coverage" data-canonical-src="https://codecov.io/gh/JuliaGeometry/Rotations.jl/branch/master/graph/badge.svg" style="max-width:100%;"></a></p>
<p>This package implements various 3D rotation parameterizations and defines
conversions between them. At their heart, each rotation parameterization <em>is</em>
a 3×3 unitary (orthogonal) matrix (based on the <a href="https://github.com/JuliaArrays/StaticArrays.jl">StaticArrays.jl package</a>),
and acts to rotate a 3-vector about the origin through matrix-vector multiplication.</p>
<p>While the <code>RotMatrix</code> type is a dense representation of a <code>3×3</code> matrix, we also
have sparse (or computed, rather) representations such as quaternions,
angle-axis parameterizations, and Euler angles.</p>
<p>For composing rotations about the origin with other transformations, see
the <a href="https://github.com/JuliaGeometry/CoordinateTransformations.jl">CoordinateTransformations.jl</a>
package.</p>
<h3><a id="user-content-interface" class="anchor" aria-hidden="true" href="#interface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Interface</h3>
<p>The following operations are supported by all of the implemented rotation parameterizations.</p>
<h4><a id="user-content-composition" class="anchor" aria-hidden="true" href="#composition"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Composition</h4>
<p>Any two rotations of the same type can be composed with simple multiplication:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="q3 = q2*q1
"><pre>q3 <span class="pl-k">=</span> q2<span class="pl-k">*</span>q1</pre></div>
<p>Rotations can be composed with the opposite (or inverse) rotation with the appropriate
division operation</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="q1 = q2\q3
q2 = q3/q1
"><pre>q1 <span class="pl-k">=</span> q2<span class="pl-k">\</span>q3
q2 <span class="pl-k">=</span> q3<span class="pl-k">/</span>q1</pre></div>
<h4><a id="user-content-rotation" class="anchor" aria-hidden="true" href="#rotation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Rotation</h4>
<p>Any rotation can operate on a 3D vector (represented as a <code>SVector{3}</code>), again through
simple multiplication:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="r2 = q*r
"><pre>r2 <span class="pl-k">=</span> q<span class="pl-k">*</span>r</pre></div>
<p>which also supports multiplication by the opposite rotation</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="r = q\r2
"><pre>r <span class="pl-k">=</span> q<span class="pl-k">\</span>r2</pre></div>

<h3><a id="user-content-rotation-parameterizations" class="anchor" aria-hidden="true" href="#rotation-parameterizations"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Rotation Parameterizations</h3>
<ol>
<li>
<p><strong>Rotation Matrix</strong> <code>RotMatrix{N, T}</code></p>
<p>An N x N rotation matrix storing the rotation.  This is a simple wrapper for
a <a href="https://github.com/JuliaArrays/StaticArrays.jl">StaticArrays</a> <code>SMatrix{N,N,T}</code>.
A rotation matrix <code>R</code> should have the property <code>I = R * R'</code>, but this isn't
enforced by the constructor. On the other hand, all the types below are
guaranteed to be "proper" rotations for all input parameters (equivalently:
parity conserving, in <em>SO(3)</em>, <code>det(r) = 1</code>, or a rotation without
reflection).</p>
</li>
<li>
<p><strong>Arbitrary Axis Rotation</strong> <code>AngleAxis{T}</code></p>
<p>A 3D rotation with fields <code>theta</code>, <code>axis_x</code>, <code>axis_y</code>, and
<code>axis_z</code> to store the rotation angle and axis of the rotation.
Like all other types in this package, once it is constructed it acts and
behaves as a 3×3 <code>AbstractMatrix</code>. The axis will be automatically
renormalized by the constructor to be a unit vector, so that <code>theta</code> always
represents the rotation angle in radians.</p>
</li>
<li>
<p><strong>Quaternions</strong> <code>UnitQuaternion{T}</code></p>
<p>A 3D rotation parameterized by a unit quaternion. Note that the constructor
will renormalize the quaternion to be a unit quaternion, and that although
they follow the same multiplicative <em>algebra</em> as quaternions, it is better
to think of <code>Quat</code> as a 3×3 matrix rather than as a quaternion <em>number</em>.</p>
<p>Previously <code>Quat</code>.</p>
</li>
<li>
<p><strong>Rotation Vector</strong> <code>RotationVec{T}</code></p>
<p>A 3D rotation encoded by an angle-axis representation as <code>angle * axis</code>.
This type is used in packages such as <a href="http://docs.opencv.org/2.4/modules/calib3d/doc/camera_calibration_and_3d_reconstruction.html#void%20Rodrigues%28InputArray%20src,%20OutputArray%20dst,%20OutputArray%20jacobian%29" rel="nofollow">OpenCV</a>.</p>
<p>Note: If you're differentiating a Rodrigues Vector check the result is what
you expect at theta = 0.  The first derivative of the rotation <em>should</em>
behave, but higher-order derivatives of it (as well as parameterization
conversions) should be tested.  The Stereographic Quaternion Projection is
the recommended three parameter format for differentiation.</p>
<p>Previously <code>RodriguesVec</code>.</p>
</li>
<li>
<p><strong>Rodrigues Parameters</strong> <code>RodriguesParam{T}</code>
A 3-parameter representation of 3D rotations that has a singularity at 180 degrees. They can be interpreted as a projection of the unit quaternion onto the plane tangent to the quaternion identity. They are computationally efficient and do not have a sign ambiguity.</p>
</li>
<li>
<p><strong>Modified Rodrigues Parameter</strong> <code>MRP{T}</code>
A 3D rotation encoded by the stereographic projection of a unit quaternion.  This projection can be visualized as a pin hole camera, with the pin hole matching the quaternion <code>[-1,0,0,0]</code> and the image plane containing the origin and having normal direction <code>[1,0,0,0]</code>.  The "null rotation" <code>Quaternion(1.0,0,0,0)</code> then maps to the <code>MRP(0,0,0)</code></p>
<p>These are similar to the Rodrigues vector in that the axis direction is stored in an unnormalized form, and the rotation angle is encoded in the length of the axis.  This type has the nice property that the derivatives of the rotation matrix w.r.t. the <code>MRP</code> parameters are rational functions, making the <code>MRP</code> type a good choice for differentiation / optimization.</p>
<p>They are frequently used in aerospace applications since they are a 3-parameter representation whose singularity happens at 360 degrees. In practice, the singularity can be avoided with some switching logic between one of two equivalent MRPs (obtained by projecting the negated quaternion).</p>
<p>Previously <code>SPQuat</code>.</p>
</li>
<li>
<p><strong>Cardinal axis rotations</strong> <code>RotX{T}</code>, <code>RotY{T}</code>, <code>RotZ{T}</code></p>
<p>Sparse representations of 3D rotations about the X, Y, or Z axis, respectively.</p>
</li>
<li>
<p><strong>Two-axis rotations</strong> <code>RotXY{T}</code>, etc</p>
<p>Conceptually, these are compositions of two of the cardinal axis rotations above,
so that <code>RotXY(x, y) == RotX(x) * RotY(y)</code> (note that the order of application to
a vector is right-to-left, as-in matrix-matrix-vector multiplication: <code>RotXY(x, y) * v == RotX(x) * (RotY(y) * v)</code>).</p>
</li>
<li>
<p><strong>Euler Angles - Three-axis rotations</strong> <code>RotXYZ{T}</code>, <code>RotXYX{T}</code>, etc</p>
<p>A composition of 3 cardinal axis rotations is typically known as a Euler
angle parameterization of a 3D rotation. The rotations with 3 unique axes,
such as <code>RotXYZ</code>, are said to follow the <a href="https://en.wikipedia.org/wiki/Euler_angles#Tait.E2.80.93Bryan_angles" rel="nofollow"><strong>Tait Bryan</strong></a> angle ordering,
while those which repeat (e.g. <code>EulerXYX</code>) are said to use <a href="https://en.wikipedia.org/wiki/Euler_angles#Conventions" rel="nofollow"><strong>Proper Euler</strong></a> angle ordering.</p>
<p>Like the two-angle versions, the order of application to a vector is right-to-left, so that <code>RotXYZ(x, y, z) * v == RotX(x) * (RotY(y) * (RotZ(z) * v))</code>.  This may be interpreted as an "extrinsic" rotation about the Z, Y, and X axes or as an "intrinsic" rotation about the X, Y, and Z axes.  Similarly, <code>RotZYX(z, y, x)</code> may be interpreted as an "extrinsic" rotation about the X, Y, and Z axes or an "intrinsic" rotation about the Z, Y, and X axes.</p>
</li>
</ol>
<h3><a id="user-content-the-rotation-error-state-and-linearization" class="anchor" aria-hidden="true" href="#the-rotation-error-state-and-linearization"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>The Rotation Error state and Linearization</h3>
<p>It is often convenient to consider perturbations or errors about a particular 3D rotation, such as applications in state estimation or optimization-based control. Intuitively, we expect these errors to live in three-dimensional space. For globally non-singular parameterizations such as unit quaternions, we need a way to map between the four parameters of the quaternion to this three-dimensional plane tangent to the four-dimensional hypersphere on which quaternions live.</p>
<p>There are several of these maps provided by Rotations.jl:</p>
<ul>
<li>
<p><code>ExponentialMap</code>: A very common mapping that uses the quaternion
exponential and the quaternion logarithm. The quaternion logarithm
converts a 3D rotation vector (i.e. axis-angle vector) to a unit quaternion.
It tends to be the most computationally expensive mapping.</p>
</li>
<li>
<p><code>CayleyMap</code>: Represents the differential quaternion using Rodrigues
parameters. This parameterization goes singular at 180° but does not
inherit the sign ambiguities of the unit quaternion. It offers an
excellent combination of cheap computation and good behavior.</p>
</li>
<li>
<p><code>MRPMap</code>: Uses Modified Rodrigues Parameters (MRPs) to represent the
differential unit quaternion. This mapping goes singular at 360°.</p>
</li>
<li>
<p><code>QuatVecMap</code>: Uses the vector part of the unit quaternion as the
differential unit quaternion. This mapping also goes singular at 180° but is
the computationally cheapest map and often performs well.</p>
</li>
</ul>
<p>Rotations.jl provides the <code>RotationError</code> type for representing rotation errors, that act just like a <code>SVector{3}</code> but carry the nonlinear map used to compute the error, which can also be used to convert the error back to a <code>UnitQuaternion</code> (and, by extention, any other 3D rotation parameterization). The following methods are useful for computing <code>RotationError</code>s and adding them back to the nominal rotation:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="rotation_error(R1::Rotation, R2::Rotation, error_map::ErrorMap)  # compute the error between `R1` and `R2` using `error_map`

<p>For a practical application of these ideas, see the quatenrion-multiplicative Extended Kalman Filter (MEKF). <a href="https://ntrs.nasa.gov/archive/nasa/casi.ntrs.nasa.gov/20040037784.pdf" rel="nofollow">This article</a> provides a good description.</p>
<p>When taking derivatives with respect to quaternions we need to account both for these mappings and the fact that local perturbations to a rotation act through composition instead of addition, as they do in vector space (e.g. <code>q * dq</code> vs <code>x + dx</code>). The following methods are useful for computing these Jacobians for <code>UnitQuaternion</code>, <code>RodriguesParam</code> or <code>MRP</code></p>
<ul>
<li><code>∇rotate(q,r)</code>: Jacobian of the <code>q*r</code> with respect to the rotation</li>
<li><code>∇composition1(q2,q1)</code>: Jacobian of <code>q2*q1</code> with respect to q1</li>
<li><code>∇composition2(q2,q1,b)</code>: Jacobian of <code>q2*q1</code> with respect to q2</li>
<li><code>∇²composition1(q2,q1)</code>: Jacobian of <code>∇composition1(q2,q2)'b</code> where b is an arbitrary vector</li>
<li><code>∇differential(q)</code>: Jacobian of composing the rotation with an infinitesimal rotation, with
respect to the infinitesimal rotation. For unit quaternions, this is a 4x3 matrix.</li>
<li><code>∇²differential(q,b)</code>: Jacobian of <code>∇differential(q)'b</code> for some vector b.</li>
</ul>
