<p dir="auto"><a href="https://github.com/c42f/ResourceContexts.jl/actions"><img src="https://github.com/c42f/ResourceContexts.jl/workflows/CI/badge.svg" alt="Build Status" style="max-width: 100%;"></a></p>
<p dir="auto"><code>ResourceContexts</code> is an experimental Julia package for <strong>composable resource
management</strong> without <code>do</code> blocks.</p>
<p dir="auto">Resources are objects which need cleanup code to run to finalize their state.
For example,</p>
<ul dir="auto">
<li>Open file handles</li>
<li>Temporary files and directories</li>
<li>Background <code>Task</code>s</li>
<li>Many other things which are currently handled with <code>do</code>-blocks.</li>
</ul>
<p dir="auto">The <code>@!</code> macro calls or defines "context functions" â€” functions which take an
<code>AbstractContext</code> as the first argument and associate any resources with that
context. This package provides context-based overrides for <code>Base</code> functions
<code>open</code>, <code>mktemp</code>, <code>mktempdir</code>, <code>cd</code>, <code>run</code>, <code>lock</code> and
<code>redirect_{stdout,stderr,stdin}</code>.</p>
<p dir="auto">The <code>@context</code> macro associates a context with the current code block which
will be passed to any context functions invoked with <code>@!</code>. When a <code>@context</code>
block exits the cleanup code associated with the context runs.</p>
<p dir="auto">The <code>@defer</code> macro defers an arbitrary cleanup expression to the end of the
current <code>@context</code>.</p>

<h1 dir="auto"><a id="user-content-design" class="anchor" aria-hidden="true" href="#design"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Design</h1>
<p dir="auto">The standard solution for Julian resource management is still the <code>do</code> block,
but this has some severe ergonomic disadvantages:</p>
<ul dir="auto">
<li>It's extremely inconvenient at the REPL; you cannot work with the
intermediate open resources without entering the context of the <code>do</code> block.</li>
<li>It creates highly nested code when many resources are present. This is both
visually confusing and the excess nesting leads to very deep stack traces.</li>
<li>Custom cleanup code is separated from the resource creation in a <code>finally</code>
block.</li>
</ul>
<p dir="auto">The ergonomic factors mean that people often prefer the non-scoped form as
argued <a href="https://github.com/JuliaLang/julia/issues/7721#issuecomment-171345256" data-hovercard-type="issue" data-hovercard-url="/JuliaLang/julia/issues/7721/hovercard">here</a>.
However this also suffers some severe disadantages:</p>
<ul dir="auto">
<li>Resources leak (or must be finalized by the GC) when people forget to guard
resource cleanup with a <code>try ... finally</code>.</li>
<li>Finalizers run in a restricted environment where any errors occur outside the
original context where the resource was created. This makes for <em>unstructured
error handling</em> where it's impossible to propagate errors in a natural way.</li>
<li>Functions which return objects must keep the backing resources alive by
holding references to them somewhere. There's two ways to do this:
<ul dir="auto">
<li>Have the returned object hold a reference to each resource. This is bad
for the implementer because it reduces composability: one cannot combine
any desired return type with arbitrary backing resources.</li>
<li>Have multiple returns such as <code>(object,resource)</code>. This is unnatural
because it forces the user to unpack return values.</li>
</ul>
</li>
</ul>
<h2 dir="auto"><a id="user-content-the-solution" class="anchor" aria-hidden="true" href="#the-solution"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>The solution</h2>
<p dir="auto">This package uses the macro <code>@!</code> as a proxy for the <a href="https://github.com/JuliaLang/julia/issues/7721#issuecomment-170942938" data-hovercard-type="issue" data-hovercard-url="/JuliaLang/julia/issues/7721/hovercard">proposed postfix <code>!</code>
syntax</a>
and adds some new ideas:</p>
<p dir="auto"><strong>The user should not be able to "forget the <code>!</code>"</strong>. We prevent this by
introducing a new <em>context calling convention</em> for resource creation functions
where the current <code>AbstractContext</code> is passed as the first argument. The
<code>@context</code> macro creates a new context in lexical scope and the <code>@!</code> macro is
syntactic sugar for calling with the current context.</p>
<p dir="auto"><strong>Resource creation functions should be able to <em>compose</em> any desired object
return type with arbitrary resources</strong>. This preserves the <a href="https://github.com/JuliaLang/julia/issues/7721#issuecomment-719152859" data-hovercard-type="issue" data-hovercard-url="/JuliaLang/julia/issues/7721/hovercard">composability of
the <code>do</code> block form</a>
by rejecting the conflation of the returned object and its backing resources.
This is a break with some familiar APIs such as the standard file handles
returned by <code>open(filename)</code> which are both a stream interface and a resource
in need of cleanup.</p>
<h2 dir="auto"><a id="user-content-possible-language-integration" class="anchor" aria-hidden="true" href="#possible-language-integration"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Possible language integration</h2>
<p dir="auto">What would all this look like as a language feature?</p>
<ul dir="auto">
<li><code>@!</code> could be replaced with a postfix <code>!</code> as proposed way back in 2015 or so.</li>
<li><code>defer</code> might become a keyword so that it can have special behavior such as
ignoring its return value. In a similar way to the code which runs inside
<code>finally</code>, there's no sense in having a "value returned by" <code>defer</code>. In
particular, I've observed that it frequently leads to the introduction of
temporary variables simply to transfer the result of the expression occurring
prior to the <code>defer</code> line.</li>
<li><code>@context</code> would be implicit at function boundaries, global <code>let</code> blocks, and
potentially other scopes within functions. Getting this part correct is
still a tricky design problem. For example, looping constructs should
introduce an implicit context, but how then can the user disable this in
particular cases?</li>
</ul>
