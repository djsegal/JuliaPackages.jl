<p dir="auto">ContinuousTimePolicyGradients.jl has been developed considering control tasks as the main application. The CTPG method for cost gradient computation can be utilised to perform policy optimisation in the setup of either background planning (offline control law learning) or decision-time planning (online control profile optimisation). The following paper addresses the optimisation of structured neural controller using CTPG with the flight vehicle normal acceleration tracking controller as an illustrating example. Please consider citing the paper if you find this package useful.</p>
<ul dir="auto">
<li>Namhoon Cho, and Hyo-Sang Shin, "Optimisation of Structured Neural Controller Based on Continuous-Time Policy Gradient," <a href="https://arxiv.org/abs/2201.06262" rel="nofollow">arXiv:2201.06262</a>, January 2022.</li>
</ul>

<p dir="auto"><code>CTPG_train()</code> provides a high-level interface for optimisation of the neural networks inside an ODE-represented dynamics based on Continuous-Time Policy Gradient (CTPG) methods that belong to the adjoint sensitivity analysis techniques. The code implemented and the default values for keyword arguments are specified considering training of a neural controller as the main application. In the context herein, a neural controller refers to a dynamic controller that incorporates neural-network-represented components at some points in its mathematical description.</p>
<p dir="auto">The code utilises the functionalities provided by the <a href="https://github.com/SciML/DiffEqFlux.jl">DiffEqFlux.jl</a> and <a href="https://github.com/SciML/DiffEqSensitivity.jl">DiffEqSensitivity.jl</a> packages, and the Automatic Differentiation (AD) capabilities provided by the <a href="https://github.com/FluxML/Zygote.jl">Zygote.jl</a> package that is integrated in DiffEqFlux.jl. <code>CTPG_train()</code> presumes the consistency of the functions provided as its input arguments with the AD tool, hence, the dynamics and cost functions should maintain their transparence against AD tools.</p>
<p dir="auto">The optimisation (training) problem minimises the cost function defined over deterministic samples of the initial plant state <code>x₀</code> and the reference <code>r</code> by performing ensemble simulation based on parallelised computation.</p>
<p dir="auto">The signals are defined as described below:</p>
<ul dir="auto">
<li><code>t</code>: time</li>
<li><code>x</code>: plant state</li>
<li><code>y</code>: plant output (= sensor output)</li>
<li><code>x_c</code>: controller state</li>
<li><code>u</code>: plant input (= controller output)</li>
<li><code>r</code>: exogenous reference</li>
<li><code>x_aug</code>: augmented forward dynamics state (= <code>[x; x_c; ∫cost_running]</code>)</li>
<li><code>p_NN</code>: neural network parameter</li>
</ul>
<p dir="auto">The arguments should be provided as explained below:</p>
<ul dir="auto">
<li><code>dynamics_plant</code>: Describes the dynamics of the plant to be controlled. Input arguments <code>x</code> and <code>u</code> should be of Vector type.</li>
<li><code>dynamics_controller</code>: Describes the dynamics of the controller that includes neural networks components. Input arguments <code>x_c</code>, <code>y</code>, <code>r</code>, and <code>p_NN</code> should be of Vector type.</li>
<li><code>dynamics_sensor</code>: Describes the dynamics of the sensor that measures output variables fed to the controller. Input arguments <code>x</code> should be of Vector type:</li>
<li><code>cost_running</code>: Describes the running cost defined as the integrand of the Lagrange-form continuous functional. Input arguments <code>x</code>, <code>y</code>, <code>u</code>, and <code>r</code> should be of Vector type.</li>
<li><code>cost_terminal</code>: Describes the terminal cost defined as the Mayer-form problem cost function. Defines a Bolza-form problem along with <code>cost_running</code>. Input arguments <code>x_f</code> and <code>r</code> should be of Vector type.</li>
<li><code>cost_regularisor</code>: Describes the regularisation term appended to the cost (loss) function. Input argument <code>p_NN</code> should be of Vector type.</li>
<li><code>policy_NN</code>: The neural networks entering into the controller dynamics. DiffEqFlux-based FastChain is recommended for its construction.</li>
<li><code>scenario</code>: Contains the parameters related with the ensemble-based training scenarios.
<ul dir="auto">
<li><code>ensemble</code>: A vector of the initial plant state <code>x₀</code> and the reference <code>r</code> constituting the trajectory realisations.</li>
<li><code>t_span</code>: Time span for forward-pass integration</li>
<li><code>t_save</code>: Array of time points to be saved while solving ODE. Typically defined as <code>t_save = t_span[1]:Δt_save:t_span[2]</code></li>
<li><code>dim_x</code>: <code>length(x)</code></li>
<li><code>dim_x_c</code>: <code>length(x_c)</code></li>
</ul>
</li>
</ul>
<p dir="auto">The keyword arguments should be provided as explained below:</p>
<ul dir="auto">
<li><code>solve_alg</code>: The algorithm used for solving ODEs. Default value is <code>Tsit5()</code></li>
<li><code>sense_alg</code>: The algorithm used for adjoint sensitivity analysis. Default value is <code>InterpolatingAdjoint(autojacvec = ZygoteVJP())</code>, because the control problems usually render the <code>BacksolveAdjoint()</code> unstable. The vjp choice <code>autojacvec = ReverseDiffVJP(true)</code> is usually faster than <code>ZygoteVJP()</code>, when the ODE function does not have any branching inside. Please refer to the <a href="https://diffeqflux.sciml.ai/dev/ControllingAdjoints/" rel="nofollow">DiffEqFlux documentation</a> for further details.</li>
<li><code>ensemble_alg</code>: The algorithm used for handling ensemble of ODEs. Default value is <code>EnsembleThreads()</code> for multi-threaded computation in CPU.</li>
<li><code>opt_1</code>: The algorithm used for the first phase of optimisation which rapidly delivers the parameter to a favourable region around a local minimum. Default value is <code>ADAM(0.01)</code>.</li>
<li><code>opt_2</code>: The algorithm used for the second phase of opitmisaiton. Defalut value is <code>LBFGS()</code> which refines the result of the first phase to find a more precise minimum. Please refer to the <a href="https://diffeqflux.sciml.ai/dev/sciml_train/" rel="nofollow">DiffEqFlux documentation</a> for further details about two-phase composition of optimisers.</li>
<li><code>maxiters_1</code>: The maximum number of iterations allowed for the first phase of optimisation with <code>opt_1</code>. Defalut value is <code>100</code>.</li>
<li><code>maxiters_2</code>: The maximum number of iterations allowed for the second phase of optimisation with <code>opt_2</code>. Defalut value is <code>100</code>.</li>
<li><code>progress_plot</code>: The indicator to plot the state history for a nominal condition among the ensemble during the learning process. Default value is <code>true</code>.</li>
<li><code>i_nominal</code>: The index to select the case to plot using <code>progress_plot</code> during optimisation process from the <code>ensemble</code> defined in <code>scenario</code>. Defalut value is <code>nothing</code>.</li>
<li><code>p_NN_0</code>: Initial value of the NN parameters supplied by the user to bypass random initialisation of <code>p_NN</code> or to continue optimisation from the previous result. Defalut value is <code>nothing</code>.</li>
<li><code>solve_kwargs...</code>: Additional keyword arguments that are passed onto the ODE solver.</li>
</ul>
<p dir="auto"><code>CTPG_train()</code> returns the following outputs:</p>
<ul dir="auto">
<li><code>result</code>: The final result of parameter optimisation.</li>
<li><code>fwd_ensemble_sol</code>: The ensemble solution of forward simulation using the final neural network parameters.</li>
<li><code>loss_history</code>: The history of loss function evaluated at each iteration.</li>
</ul>
