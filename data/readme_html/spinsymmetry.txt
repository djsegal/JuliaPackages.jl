<p dir="auto">Transform your spin system to a symmetry sector of your choice!</p>
<p dir="auto">SpinSymmetry.jl is a light-weight, reasonably fast package with no dependencies. Some commonly used symmetries are implemented, there's support for user-defined symmetries and every symmetry can be combined!</p>

<h1 dir="auto"><a id="user-content-features" class="anchor" aria-hidden="true" href="#features"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Features</h1>
<ul dir="auto">
<li>Use <code>symmetrized_basis</code> to construct a collection of your symmetries. Provide as the first argument the system's size (and optionally magnetizaion block) and then follow with symmetry operations and their sector alternating.</li>
<li>Apply the symmetries to a state or an operator using <code>symmetrize_state</code> and <code>symmetrize_operator</code></li>
<li>Find the size of the symmetry sector with <code>basissize</code></li>
</ul>
<p dir="auto">The symmetry operations supported are:</p>
<ul dir="auto">
<li>z-magnetization block (via <code>zbasis(N, k)</code>)</li>
<li>Spin flip via <code>Flip(positions)</code> or <code>Flip(N)</code></li>
<li>Shift symmetry via <code>Shift(N, amount=1)</code></li>
<li>Swap/Exchange symmetry via <code>Swap(pos1, pos2)</code></li>
<li>Spatial reflection via <code>SpatialReflection(N)</code></li>
</ul>
<p dir="auto">where <code>N</code> denotes the number of spins in the system and their positions should be given as a Julian index, i.e. in the range <code>1:N</code>.</p>
<p dir="auto">To get the tranformation matrix to the symmetrized subspace just use <code>transformationmatrix(symmetrized_basis)</code>.</p>
<p dir="auto"><strong>Note:</strong> The projection on a specific magnetization block is applied first. Thus if you have spin flip symmetry and restrict to a magnetization block, your symmetrized basis states look like "|‚Üë..‚Üë‚ü© ¬± |‚Üì..‚Üì‚ü©". So in this case you effectively specified S_z^2 and parity.</p>

<p dir="auto">It's also quite easy to define your own symmetry operations.
Simply define a function <code>f</code> that maps one basis index to the next.
Note that these basis indices are a binary representation (range 0:2^N-1) of the spin basis where the first spin is represented by the <em>least</em> significant bit.
Then you can use <code>GenericSymmetry(f, L)</code> where <code>L</code> denotes the order of your symmtry.
The order is the smallest number <code>L</code> s. t. <code>f</code> applied <code>L</code> is the identity for all indices.</p>

<h1 dir="auto"><a id="user-content-implementation-details" class="anchor" aria-hidden="true" href="#implementation-details"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Implementation details</h1>
<p dir="auto">Imagine all basis vectors as the vertices of a graph and the symmetries generate
(directed) edges between them. These edges carry a phase factor that's <code>exp(i2œÄ*k/L)</code>,
where <code>k</code> is the symmetry sector and <code>L</code> the order of the symmetry.
To find the basis transformation from the old basis to the new symmetrized basis,
we need to find the connected components. Each connected component corresponds to a
new basis vector. All vectors in a component contribute equally (by magnitude) and to
find the phase factors we just follow the edges.
This is complicated by the fact, that the graph is not acyclic! Thus we need to make
sure that the different symmetries are actually compatible in the component. That
means that every starting and ending at specified nodes needs to have the same phase.
Otherwise the whole component just vanishes under the combined symmetry.</p>
<p dir="auto">Implementation detail:
We use Rational numbers <code>r</code> instead of Complex64 <code>œÜ</code> to track the phases. Their relation is
<code>œÜ = exp(i2œÄ*r)</code></p>
<p dir="auto">Algorithm:</p>
<ol dir="auto">
<li>Start at a basis vector</li>
<li>Find connected vectors by applying all symmetries and checking for compatibility
of their phases</li>
<li>Track the phases of all vectors seen so far (check for compatibility of phases)</li>
<li>Continue generating connected vectors for all vectors until there are no new discoveries</li>
<li>Repeat all steps for the next unseen basis vector</li>
</ol>
