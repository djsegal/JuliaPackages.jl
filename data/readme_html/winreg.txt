<p dir="auto">The nomenclature of the Windows Registry is a bit confusing: a <em>registry key</em> is a container object in a tree. Each registry key can itself have subkeys (so the keys form a hierarchy), but can also have <em>values</em> which are really key-value pairs (usually referred to as <em>value names</em> and <em>value data</em>).</p>
<p dir="auto">In WinReg.jl, a registry key is represented by a <code>RegKey</code> object. There are several pre-defined root <code>RegKey</code>s:</p>
<ul dir="auto">
<li><code>WinReg.HKEY_CLASSES_ROOT</code></li>
<li><code>WinReg.HKEY_CURRENT_USER</code></li>
<li><code>WinReg.HKEY_LOCAL_MACHINE</code></li>
<li><code>WinReg.HKEY_USERS</code></li>
<li><code>WinReg.HKEY_PERFORMANCE_DATA</code></li>
<li><code>WinReg.HKEY_CURRENT_CONFIG</code></li>
<li><code>WinReg.HKEY_DYN_DATA</code></li>
</ul>
<p dir="auto"><code>WinReg.openkey(basekey, path)</code> opens a subkey of <code>basekey</code> at <code>path</code>, returning a <code>RegKey</code> object.</p>
<p dir="auto"><code>subkeys(key)</code> gives an iterator over the names of the subkeys of <code>key</code>.</p>
<p dir="auto">A <code>RegKey</code> acts like a <code>Dict</code> for the registry values: values can be read from the registry by <code>getindex</code>, existence checked by <code>haskey</code>, etc. Writing values (<code>setindex!</code>) is <em>not</em> currently supported (see Requests below).</p>
<p dir="auto">For convenience, the function <code>querykey</code> can be used to query a single value:</p>
