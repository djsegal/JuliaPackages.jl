<p dir="auto">Julia package for creating new packages and updating existing ones, following common practices and workflow recommendations.</p>
<p dir="auto"><strong>This package may overwrite existing files.</strong> While care has been taken to ensure no data loss, it may nevertheless happen. Keep backups, commit to a non-local git repository, and <strong>use at your own risk</strong>.</p>

<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="import PkgSkeleton
PkgSkeleton.generate(&quot;target_directory&quot;) # uses default template"><pre><span class="pl-k">import</span> PkgSkeleton
PkgSkeleton<span class="pl-k">.</span><span class="pl-c1">generate</span>(<span class="pl-s"><span class="pl-pds">"</span>target_directory<span class="pl-pds">"</span></span>) <span class="pl-c"><span class="pl-c">#</span> uses default template</span></pre></div>
<p dir="auto">Then</p>
<ol dir="auto">
<li>
<p dir="auto">Various defaults <a href="#substitutions">(described below)</a> are collected from your environment, eg your name, e-mail address, and Github account name (from <code>git</code> global settings). You can override these using a keyword argument.</p>
</li>
<li>
<p dir="auto">If <code>target_directory</code> does not exist, it is created with an empty git repository. Conversely, if the directory exits but is not a git repository, generation is aborted.</p>
</li>
<li>
<p dir="auto">Files in template are copied recursively, with various <a href="#substitutions">substitutions (as described below)</a>. Unless you are explicitly allowing overwrites, uncommitted files in the repository are not modified.</p>
</li>
</ol>
<p dir="auto">After this, you probably want to <code>pkg&gt; dev destination/directory</code> in Julia, and add your Github repository as a remote.</p>
<p dir="auto">See <code>?PkgSkeleton.generate</code> for details.</p>
<h3 dir="auto"><a id="user-content-updating-existing-packages" class="anchor" aria-hidden="true" href="#updating-existing-packages"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Updating existing packages</h3>
<p dir="auto">Best practices and recommended setups change with time. The recommended workflow for updating <em>existing</em> packages using templates from this package is the following.</p>
<ol dir="auto">
<li>
<p dir="auto">Make sure that this package is of the latest version, eg with <code>pkg&gt; up</code>.</p>
</li>
<li>
<p dir="auto">Make sure that <em>everything</em> that is part of a template is committed in version control. This is very important: when files are overwritten, work may be lost, so <code>PkgSkeleton.generate</code> prefers not to overwrite existing files.</p>
</li>
<li>
<p dir="auto">Run</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="PkgSkeleton.generate(&quot;/path/to/pkg&quot;)"><pre>PkgSkeleton<span class="pl-k">.</span><span class="pl-c1">generate</span>(<span class="pl-s"><span class="pl-pds">"</span>/path/to/pkg<span class="pl-pds">"</span></span>)</pre></div>
<p dir="auto">and see the output for what was modified.</p>
</li>
<li>
<p dir="auto">Use your favorite git interface for reviewing the change. Pick and commit what you like, discard the rest of the changes.</p>
</li>
</ol>

<p dir="auto">Just create directories with text (code, Markdown, TOML) files, <a href="#substitutions">substitutions</a> between <code>{}</code>s will be replaced in <em>filenames</em> and their <em>contents</em>.</p>

<p dir="auto">Templates replace the following in files <em>and filenames</em>:</p>

</tr>

</tr>

</tr>

<h2 dir="auto"><a id="user-content-design-principles" class="anchor" aria-hidden="true" href="#design-principles"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Design principles</h2>
<ol dir="auto">
<li>
<p dir="auto"><a href="https://en.wikipedia.org/wiki/KISS_principle" rel="nofollow">Keep it simple</a>: do nothing more than substitute strings into templates, with a few safeguards. This keeps the code simple: currently <a href="src/PkgSkeleton.jl">less than 300 LOC</a> without docstrings. For me, this covers 99% of the use cases; the rest I edit manually.</p>
</li>
<li>
<p dir="auto">Tread ligthly: don't modify uncommitted files (unless asked to, but really don't do that), or files with the same content (to preserve timestamps).</p>
</li>
<li>
<p dir="auto">Assume that tooling for packages will keep changing, make it easy to update.</p>
</li>
</ol>
