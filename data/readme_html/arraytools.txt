<p>This <a href="https://julialang.org/" rel="nofollow">Julia</a> package provides a number of methods and types to deal
with the variety of array types (sub-types of <code>AbstractArray</code>) that exist in
Julia and to help building custom array-like types without sacrificing
performances.</p>
<p>These are useful to implement methods to process arrays in a generic way.</p>
<h2><a id="user-content-rubber-indices" class="anchor" aria-hidden="true" href="#rubber-indices"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Rubber indices</h2>
<p>The constants <code>..</code> and <code>…</code> (type <code>\dots</code> and hit the <code>tab</code> key) can be used in
array indexation to left or right justify the other indices.  For instance,
assuming <code>A</code> is a <code>3×4×5×6</code> array, then all the following equalities hold:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="A[..]          === A # the two are the same object
A[..,3]         == A[:,:,:,3]
A[2,..]         == A[2,:,:,:]
A[..,2:4,5]     == A[:,:,2:4,5]
A[:,2:3,..]     == A[:,2:3,:,:]
A[2:3,..,1,2:4] == A[2:3,:,1,2:4]
"><pre>A[<span class="pl-k">..</span>]          <span class="pl-k">===</span> A <span class="pl-c"><span class="pl-c">#</span> the two are the same object</span>
A[<span class="pl-k">..</span>,<span class="pl-c1">3</span>]         <span class="pl-k">==</span> A[:,:,:,<span class="pl-c1">3</span>]
A[<span class="pl-c1">2</span>,<span class="pl-k">..</span>]         <span class="pl-k">==</span> A[<span class="pl-c1">2</span>,:,:,:]
A[<span class="pl-k">..</span>,<span class="pl-c1">2</span><span class="pl-k">:</span><span class="pl-c1">4</span>,<span class="pl-c1">5</span>]     <span class="pl-k">==</span> A[:,:,<span class="pl-c1">2</span><span class="pl-k">:</span><span class="pl-c1">4</span>,<span class="pl-c1">5</span>]

A[<span class="pl-c1">2</span><span class="pl-k">:</span><span class="pl-c1">3</span>,<span class="pl-k">..</span>,<span class="pl-c1">1</span>,<span class="pl-c1">2</span><span class="pl-k">:</span><span class="pl-c1">4</span>] <span class="pl-k">==</span> A[<span class="pl-c1">2</span><span class="pl-k">:</span><span class="pl-c1">3</span>,:,<span class="pl-c1">1</span>,<span class="pl-c1">2</span><span class="pl-k">:</span><span class="pl-c1">4</span>]</pre></div>
<p>As can be seen, the advantage of the <em>rubber index</em> <code>..</code> is that it
automatically expands as the list of colons needed to have the correct number
of indices.  The expressions are also more readable.  The idea comes from the
<a href="http://yorick.github.com/"><code>Yorick</code></a> language by Dave Munro.</p>
<p>The rubber index may also be used for setting values.  For instance:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="A[..] .= 1         # to fill A with ones
A[..,3] = A[..,2]  # to copy A[:,:,:,2] in A[:,:,:,3]
A[..,3] .= A[..,2] # idem but faster
A[2,..] = A[3,..]  # to copy A[3,:,:,:] in A[2,:,:,:]
A[..,2:4,5] .= 7   # to set all elements in A[:,:,2:4,5] to 7
"><pre>A[<span class="pl-k">..</span>] <span class="pl-k">.=</span> <span class="pl-c1">1</span>         <span class="pl-c"><span class="pl-c">#</span> to fill A with ones</span>

A[<span class="pl-c1">2</span>,<span class="pl-k">..</span>] <span class="pl-k">=</span> A[<span class="pl-c1">3</span>,<span class="pl-k">..</span>]  <span class="pl-c"><span class="pl-c">#</span> to copy A[3,:,:,:] in A[2,:,:,:]</span>

<p>Leading/trailing indices may also be specified as Cartesian indices (of type
<code>CartesianIndex</code>).</p>
<p>Technically, the constant <code>..</code> is defined as <code>RubberIndex()</code> where <code>RubberIndex</code>
is the singleton type that represents any number of indices.</p>
<p>Call <code>colons(n)</code> if you need a <code>n</code>-tuple of colons <code>:</code>.  When <code>n</code> is known at
compile time, it is faster to call <code>colons(Val(n))</code>.</p>
<p><g-emoji class="g-emoji" alias="warning" fallback-src="https://github.githubassets.com/images/icons/emoji/unicode/26a0.png">⚠️</g-emoji> <strong>Warning</strong> A current limitation of the rubber index is that it will
confuse the interpretation of the <code>end</code> token appearing in the same index list
<em>after</em> the rubber index.  This is beacuse the parser wrongly assumes that the
rubber index counts for a single dimension.  The <code>end</code> token may however
appears <em>before</em> the rubber index.  Example:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="A = rand(5,10,4,3);

"><pre lang=".julia"><code>A = rand(5,10,4,3);

<h2><a id="user-content-array-like-objects" class="anchor" aria-hidden="true" href="#array-like-objects"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Array-like objects</h2>
<h3><a id="user-content-defining-custom-array-like-objects" class="anchor" aria-hidden="true" href="#defining-custom-array-like-objects"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Defining custom array-like objects</h3>
<p>Julia array interface is very powerful and flexible, it is therefore tempting
to define custom array-like types, that is Julia types that behave like arrays,
without sacrificing efficiency.  The <code>ArrayTools</code> package provides simple means
to define such array-like types if the values to be accessed as if in an array
are stored in an array (of any concrete type) embedded in the object instance.</p>
<p>This is as simple as:</p>
<ol>
<li>
<p>Make your type inherit from <code>LinearArray{T,N}</code> or <code>CartesianArray{T,N}</code>
depending whether the index style of the embedded array is <code>IndexLinear()</code>
or <code>IndexCartesian()</code>.</p>
</li>
<li>
<p>Extend the <code>Base.parent(A)</code> method for your custom type so that it returns
the embedded array of an instance <code>A</code>.</p>
</li>
</ol>
<p>For instance (of course replacing the ellipsis <code>...</code>):</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="using ArrayTools.PseudoArrays
struct CustomArray{T,N,...} &lt;: LinearArray{T,N}
    arr::Array{T,N} # can be any array type with linear index style
    ...             # another member
    ...             # yet another member
    ...             # etc.
end

@inline Base.parent(A::CustomArray) = A.arr
"><pre><span class="pl-k">using</span> ArrayTools<span class="pl-k">.</span>PseudoArrays
<span class="pl-k">struct</span> CustomArray{T,N,<span class="pl-k">...</span>} <span class="pl-k">&lt;:</span> <span class="pl-c1">LinearArray{T,N}</span>
    arr<span class="pl-k">::</span><span class="pl-c1">Array{T,N}</span> <span class="pl-c"><span class="pl-c">#</span> can be any array type with linear index style</span>
    <span class="pl-k">...</span>             <span class="pl-c"><span class="pl-c">#</span> another member</span>
    <span class="pl-k">...</span>             <span class="pl-c"><span class="pl-c">#</span> yet another member</span>
    <span class="pl-k">...</span>             <span class="pl-c"><span class="pl-c">#</span> etc.</span>
<span class="pl-k">end</span>

<span class="pl-c1">@inline</span> Base<span class="pl-k">.</span><span class="pl-en">parent</span>(A<span class="pl-k">::</span><span class="pl-c1">CustomArray</span>) <span class="pl-k">=</span> A<span class="pl-k">.</span>arr</pre></div>
<p>As a result, instances of your <code>CustomArray{T,N}</code> will be also seen as
instances of <code>AbstractArray{T,N}</code> and will behave as if they implement linear
indexing.  Apart from the needs to extend the <code>Base.parent</code> method, the
interface to <code>LinearArray{T,N}</code> should provide any necessary methods for
indexation, getting the dimensions, the element type, <em>etc.</em> for the derived
custom type.  You may however override these definitions by more optimized or
more suitable methods specialized for your custom array-like type.</p>
<p>If your custom array-like type is based on an array whose index style is
<code>IndexCartesian()</code> (instead of <code>IndexLinear()</code> in the above example), just make
your custom type derived from <code>CartesianArray{T,N}</code> (instead of
<code>LinearArray{T,N}</code>).  For such array-like object, index checking requires an
efficient implementation of the <code>Base.axes()</code> method which you may have to
specialize.  The default implementation is:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="@inline Base.axes(A::CartesianArray) = axes(parent(A))
"><pre><span class="pl-c1">@inline</span> Base<span class="pl-k">.</span><span class="pl-en">axes</span>(A<span class="pl-k">::</span><span class="pl-c1">CartesianArray</span>) <span class="pl-k">=</span> <span class="pl-c1">axes</span>(<span class="pl-c1">parent</span>(A))</pre></div>
<h3><a id="user-content-array-like-objects-with-properties" class="anchor" aria-hidden="true" href="#array-like-objects-with-properties"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Array-like objects with properties</h3>
<p>As a working example of custom array-like objects, the <code>ArrayTools</code> package
provides <code>AnnotatedArray{T,N,P}</code> objects which store values like arrays but
also have properties stored in a dictionary or a named tuple (of type <code>P</code>).
Here the parameters are the element type <code>T</code> of the values in the array part,
the number <code>N</code> of dimensions of the array part and the type <code>P</code> of the object
storing the properties.</p>
<p>Building annotated arrays is easy:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="using ArrayTools.AnnotatedArrays
dims = (100, 50)
T = Float32
A = AnnotatedArray(zeros(T, dims), units = &quot;photons&quot;, Δx = 0.20, Δy = 0.15)
B = AnnotatedArray{T}(undef, dims, units = &quot;µm&quot;, Δx = 0.10, Δy = 0.20)
"><pre><span class="pl-k">using</span> ArrayTools<span class="pl-k">.</span>AnnotatedArrays

<p>Here the initial properties of <code>A</code> and <code>B</code> are specified by the keywords in the
call to the constructor; their properties will have symbolic names with any
kind of value.  The array contents of <code>A</code> is an array of zeros, while the array
contents of <code>B</code> is created by the constructor with undefined values.  Indexing
<code>A</code> or <code>B</code> with integers of Cartesian indices is the same as accessing the
values of their array contents while indexing <code>A</code> or <code>B</code> by symbols is the same
as accessing their properties.  For example:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="A.Δx             # yields 0.2

<p>Creating an annotated array is summarized by:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="using ArrayTools.AnnotatedArrays
A = AnnotatedArray(arr, prop)
B = AnnotatedArray{T}(init, dims, prop)
"><pre><span class="pl-k">using</span> ArrayTools<span class="pl-k">.</span>AnnotatedArrays
A <span class="pl-k">=</span> <span class="pl-c1">AnnotatedArray</span>(arr, prop)
B <span class="pl-k">=</span> <span class="pl-c1">AnnotatedArray</span><span class="pl-c1">{T}</span>(init, dims, prop)</pre></div>
<p>where <code>arr</code> is an existing array or an expression whose result is an array,
<code>prop</code> specifies the initial properties (more on this below), <code>T</code> is the type
of array element, <code>init</code> is usually <code>undef</code> and <code>dims</code> is a tuple of array
dimensions.  If <code>arr</code> is an existing array, the object <code>A</code> created above will
reference this array and hence share its contents with the caller (call
<code>copy(arr)</code> to avoid that).  The same applies if the initial properties are
specified by a dictionary.</p>
<p>The properties <code>prop</code> can be specified by keywords, by key-value pairs, as a
dictionary or as a named tuple.  To avoid ambiguities, these different styles
cannot be mixed.  Below are a few examples:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="using ArrayTools.AnnotatedArrays
arr = zeros(3,4,5)
A = AnnotatedArray(arr,  units  =  &quot;µm&quot;,  Δx  =  0.1,  Δy  =  0.2)
B = AnnotatedArray(arr, :units  =&gt; &quot;µm&quot;, :Δx  =&gt; 0.1, :Δy  =&gt; 0.2)
C = AnnotatedArray(arr, &quot;units&quot; =&gt; &quot;µm&quot;, &quot;Δx&quot; =&gt; 0.1, &quot;Δy&quot; =&gt; 0.2)
D = AnnotatedArray(arr, (units  =  &quot;µm&quot;,  Δx  =  0.1,  Δy  =  0.2))
"><pre><span class="pl-k">using</span> ArrayTools<span class="pl-k">.</span>AnnotatedArrays
arr <span class="pl-k">=</span> <span class="pl-c1">zeros</span>(<span class="pl-c1">3</span>,<span class="pl-c1">4</span>,<span class="pl-c1">5</span>)
A <span class="pl-k">=</span> <span class="pl-c1">AnnotatedArray</span>(arr,  units  <span class="pl-k">=</span>  <span class="pl-s"><span class="pl-pds">"</span>µm<span class="pl-pds">"</span></span>,  Δx  <span class="pl-k">=</span>  <span class="pl-c1">0.1</span>,  Δy  <span class="pl-k">=</span>  <span class="pl-c1">0.2</span>)
B <span class="pl-k">=</span> <span class="pl-c1">AnnotatedArray</span>(arr, <span class="pl-c1">:units</span>  <span class="pl-k">=&gt;</span> <span class="pl-s"><span class="pl-pds">"</span>µm<span class="pl-pds">"</span></span>, <span class="pl-c1">:Δx</span>  <span class="pl-k">=&gt;</span> <span class="pl-c1">0.1</span>, <span class="pl-c1">:Δy</span>  <span class="pl-k">=&gt;</span> <span class="pl-c1">0.2</span>)
C <span class="pl-k">=</span> <span class="pl-c1">AnnotatedArray</span>(arr, <span class="pl-s"><span class="pl-pds">"</span>units<span class="pl-pds">"</span></span> <span class="pl-k">=&gt;</span> <span class="pl-s"><span class="pl-pds">"</span>µm<span class="pl-pds">"</span></span>, <span class="pl-s"><span class="pl-pds">"</span>Δx<span class="pl-pds">"</span></span> <span class="pl-k">=&gt;</span> <span class="pl-c1">0.1</span>, <span class="pl-s"><span class="pl-pds">"</span>Δy<span class="pl-pds">"</span></span> <span class="pl-k">=&gt;</span> <span class="pl-c1">0.2</span>)
D <span class="pl-k">=</span> <span class="pl-c1">AnnotatedArray</span>(arr, (units  <span class="pl-k">=</span>  <span class="pl-s"><span class="pl-pds">"</span>µm<span class="pl-pds">"</span></span>,  Δx  <span class="pl-k">=</span>  <span class="pl-c1">0.1</span>,  Δy  <span class="pl-k">=</span>  <span class="pl-c1">0.2</span>))</pre></div>
<p>The two first examples (<code>A</code> and <code>B</code>) both yield an annotated array whose
properties have symbolic keys and can have any type of value.  The third
example (<code>C</code>) yields an annotated array whose properties have string keys and
can have any type of value.  The properties of <code>A</code>, <code>B</code> and <code>C</code> are <em>dynamic</em>:
they can be modified, deleted and new properties can be inserted.  The fourth
example (<code>D</code>) yields an annotated array whose properties are stored by a <em>named
tuple</em>, they are <em>immutable</em> and have symbolic keys.</p>
<p>Accessing a property is possible via the syntax <code>obj[key]</code> or, for symbolic and
textual keys, via the syntax <code>obj.key</code>.  Accessing <em>immutable</em> properties is
the fastest while accessing textual properties as <code>obj.key</code> is the slowest
(because it involves converting a symbol into a string).</p>
<p>When initially specified by keywords or as key-value pairs, the properties are
stored in a dictionary whose key type is specialized if possible (for
efficiency) but with value type <code>Any</code> (for flexibility).  If one wants specific
properties key and value types, it is always possible to explicitly specify a
dictionary in the call to <code>AnnotatedArray</code>.  For instance:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="E = AnnotatedArray(arr, Dict{Symbol,Int32}(:a =&gt; 1, :b =&gt; 2))
"><pre>E <span class="pl-k">=</span> <span class="pl-c1">AnnotatedArray</span>(arr, <span class="pl-c1">Dict</span><span class="pl-c1">{Symbol,Int32}</span>(<span class="pl-c1">:a</span> <span class="pl-k">=&gt;</span> <span class="pl-c1">1</span>, <span class="pl-c1">:b</span> <span class="pl-k">=&gt;</span> <span class="pl-c1">2</span>))</pre></div>
<p>yields an annotated array whose properties have symbolic keys and integer
values of type <code>Int32</code>.</p>
<p>Property key types are not limited to <code>Symbol</code> or <code>String</code>, but, to avoid
ambiguities, key types must be more specialized than <code>Any</code> and must not inherit
from types like <code>Integer</code> or <code>CartesianIndex</code> which are reserved for indexing
the array contents of annotated arrays.</p>
<p>If the dictionary is unspecified, the properties are stored in a, initially
empty, dictionary with symbolic keys and value of any type, <em>i.e.</em>
<code>Dict{Symbol,Any}()</code>.</p>
<p>Iterating on an annotated array is iterating on its array values.  To iterate
on its properties, call the <code>properties</code> method which returns the object
storing the properties:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="dims = (100, 50)

    println(k, &quot; =&gt; &quot;, v)

<h3><a id="user-content-array-indexing" class="anchor" aria-hidden="true" href="#array-indexing"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Array indexing</h3>
<p>The <code>all_indices</code> method takes any number of array arguments and yields an
efficient iterator for visiting all indices each index of the arguments.  Its
behavior is similar to that of <code>eachindex</code> method except that <code>all_indices</code>
throws a <code>DimensionMismatch</code> exception if the arrays have different axes.  It
is always safe to specify <code>@inbounds</code> (and <code>@simd</code>) for a loop like:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="for i in all_indices(A, B, C, D)
   A[i] = B[i]*C[i] + D[i]
end
"><pre><span class="pl-k">for</span> i <span class="pl-k">in</span> <span class="pl-c1">all_indices</span>(A, B, C, D)
   A[i] <span class="pl-k">=</span> B[i]<span class="pl-k">*</span>C[i] <span class="pl-k">+</span> D[i]
<span class="pl-k">end</span></pre></div>
<p>The <code>eachindex</code> and <code>all_indices</code> methods are very useful when writing loops
over array elements so as to be agnostic to which specfic indexing rule is the
most suitable.  Some algorithms are however more efficient or easier to write
if all involved arrays are indexed by a single 1-based index.  In that case,
<code>using ArrayTools</code> provides:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="to_fast_array(A)
"><pre><span class="pl-c1">to_fast_array</span>(A)</pre></div>
<p>which checks whether array <code>A</code> is suitable for fast indexing (by a single
integer starting at 1); if it does, <code>A</code> is returned to the caller; otherwise,
the contents of <code>A</code> is converted to a suitable array type implementing fast
indexing and is returned to the caller.</p>
<p>To just check whether array <code>A</code> is suitable for fast indexing, call:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="is_fast_array(A) -&gt; bool
"><pre><span class="pl-c1">is_fast_array</span>(A) <span class="pl-k">-&gt;</span> bool</pre></div>
<p>Multiple arguments can be checked at the same time: <code>is_fast_array(A,B,...)</code> is
the same as <code>is_fast_array(A) &amp;&amp; is_fast_array(B) &amp;&amp; is_flat_array(...)</code>.</p>

<p>When calling (with <code>ccall</code>) a compiled function coded in another language (C,
FORTRAN, etc.), you have to make sure that array arguments have the same
storage as assumed by these languages so that it is safe to pass the pointer of
the array to the compiled function.</p>
<p>Typically, you want to ensure that the elements are stored in memory
contiguously and in column-major order.  This can be ckecked by calling:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="is_flat_array(A) -&gt; bool
"><pre><span class="pl-c1">is_flat_array</span>(A) <span class="pl-k">-&gt;</span> bool</pre></div>
<p>or, with several arguments:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="is_flat_array(A, B, C, ...)  -&gt; bool
"><pre><span class="pl-c1">is_flat_array</span>(A, B, C, <span class="pl-k">...</span>)  <span class="pl-k">-&gt;</span> bool</pre></div>
<p>In order to get an array with such <em>flat</em> storage and possibly with a given
element type <code>T</code>, call:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="to_flat_array([T = eltype(A),] A)
"><pre><span class="pl-c1">to_flat_array</span>([T <span class="pl-k">=</span> <span class="pl-c1">eltype</span>(A),] A)</pre></div>
<p>which just returns <code>A</code> if the requirements hold or converts <code>A</code> to a suitable
array form.</p>
<h2><a id="user-content-faq" class="anchor" aria-hidden="true" href="#faq"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>FAQ</h2>
<ul>
<li>
<p>What is the difference between <code>IndexStyle</code> (defined in base Julia) and
<code>IndexingTrait</code> (defined in <code>ArrayTools</code>)?</p>
<p>If <code>IndexStyle(A) === IndexLinear()</code>, then array <code>A</code> can be efficiently
indexed by one integer (even if <code>A</code> is multidimensional) and column-major
ordering is used to access the elements of <code>A</code>.  The only (known) other
possibility is <code>IndexStyle(A) === IndexCartesian()</code>.</p>
<p>If <code>IndexingTrait(A) === FastIndexing()</code>, then <code>IndexStyle(A) === IndexLinear()</code> also holds (see above) <strong>and</strong> array <code>A</code> has standard 1-based
indices.</p>
</li>
<li>
<p>What is the difference between <code>Base.has_offset_axes</code> (provided by Julia) and
<code>has_standard_indexing</code> (provided by  <code>ArrayTools</code>)?</p>
<p>For the caller, <code>has_standard_indexing(args...)</code> yields the opposite result
as <code>Base.has_offset_axes(args...)</code>.  Furthermore, <code>has_standard_indexing</code> is
a bit faster.</p>
</li>
</ul>
