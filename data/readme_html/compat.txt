<p dir="auto"><a href="https://github.com/JuliaLang/Compat.jl/actions/workflows/CI.yml"><img src="https://github.com/JuliaLang/Compat.jl/actions/workflows/CI.yml/badge.svg" alt="Build Status" style="max-width: 100%;"></a></p>
<p dir="auto">The <strong>Compat</strong> package is designed to ease interoperability between
older and newer versions of the <a href="http://julialang.org/" rel="nofollow">Julia
language</a>.  In particular, in cases where it is
impossible to write code that works with both the latest Julia
<code>master</code> branch and older Julia versions, or impossible to write code
that doesn't generate a deprecation warning in some Julia version, the
Compat package provides a macro that lets you use the <em>latest syntax</em>
in a backwards-compatible way.</p>
<p dir="auto">This is primarily intended for use by other <a href="https://julialang.github.io/Pkg.jl/v1/creating-packages/" rel="nofollow">Julia
packages</a>, where
it is important to maintain cross-version compatibility.</p>
<h2 dir="auto"><a id="user-content-usage" class="anchor" aria-hidden="true" href="#usage"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Usage</h2>
<p dir="auto">To use Compat in your Julia package, add it as a dependency of your package using the package manager</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="pkg&gt; add Compat"><pre>pkg<span class="pl-k">&gt;</span> add Compat</pre></div>
<p dir="auto">and add a <a href="https://julialang.github.io/Pkg.jl/v1/compatibility/#Version-specifier-format-1" rel="nofollow">version specifier line</a>
such as <code>Compat = "3.22, 4"</code> in the <code>[compat]</code>section of the <code>Project.toml</code> file
in your package directory. The version in the latter should be the minimum
version that supports all needed features (see list below). Note that Compat v4
requires Julia v1.6, but some features may have been backported to Compat v3
(see the
<a href="https://github.com/JuliaLang/Compat.jl/tree/release-3#supported-features">feature list of the release-3 branch</a>).
If you require any of those backported features, be sure to specify the correct
compatibility in your <code>Project.toml</code>. E.g. if the feature from Compat v4.x has
been backported to v3.y, use <code>Compat = 3.y, 4.x</code>. If you use a feature that had
originally been added in Compat v3 (e.g. in 3.x), don't forget to also declare
compatibility with v4 with <code>Compat = 3.x, 4</code> (unless you use one the very few
<a href="https://github.com/JuliaLang/Compat.jl/releases/tag/v4.0.0">things that got removed between Compat v3 and v4</a>,
which you most probably don't).</p>
<p dir="auto">To minimize dependency conflicts between packages it is recommended that packages
allow for both appropriate v4 and v3 versions of Compat.jl in their Project.toml
(except for rare cases of packages that support only v4 or v3 version of Compat.jl).</p>
<p dir="auto">Then, in your package, shortly after the <code>module</code> statement include a line like
this:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="using Compat"><pre><span class="pl-k">using</span> Compat</pre></div>
<p dir="auto">and then as needed add</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="@compat ...compat syntax..."><pre><span class="pl-c1">@compat</span> <span class="pl-k">...</span>compat syntax<span class="pl-k">...</span></pre></div>
<p dir="auto">wherever you want to use syntax that differs in the latest Julia
<code>master</code> (the development version of Julia). The <code>compat syntax</code> is usually
the syntax on Julia <code>master</code>. However, in a few cases where this is not possible,
a slightly different syntax might be used.
Please check the list below for the specific syntax you need.</p>
<h2 dir="auto"><a id="user-content-compatibility" class="anchor" aria-hidden="true" href="#compatibility"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Compatibility</h2>
<p dir="auto">Features in the development versions of <code>julia</code> may be added and released in
Compat.jl.  However, such features are considered experimental until the
relevant <code>julia</code> version is released.  These features can be changed or removed
without incrementing the major version of Compat.jl if necessary to match
changes in <code>julia</code>.</p>
<h2 dir="auto"><a id="user-content-supported-features" class="anchor" aria-hidden="true" href="#supported-features"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Supported features</h2>
<ul dir="auto">
<li>
<p dir="auto"><code>splat(f)</code> which is equivalent to <code>args -&gt; f(args...)</code>. (<a href="https://github.com/JuliaLang/julia/issues/42717" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/42717/hovercard">#42717</a>, <a href="https://github.com/JuliaLang/julia/issues/48038" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/48038/hovercard">#48038</a>) (since Compat 4.6.0) (Note: for historical reasons, <code>Compat</code> on Julia before v1.9 also exports <code>Splat</code>; its usage is discouraged, however.)</p>
</li>
<li>
<p dir="auto"><code>Compat.@assume_effects setting... ex</code> overrides the compiler's effect modeling for the method definition <code>ex</code> on Julia versions that support this feature. Julia version without support just pass back <code>ex</code>. (<a href="https://github.com/JuliaLang/julia/issues/43852" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/43852/hovercard">#43852</a>) (since Compat 4.4.0)</p>
</li>
<li>
<p dir="auto"><code>div</code>, <code>lcm</code>, <code>gcd</code>, <code>/</code>, <code>rem</code>, and <code>mod</code> will <code>promote</code> heterogenous <code>Dates.Period</code>s (<a href="https://github.com/JuliaLang/julia/commit/bdf9ead91e5a8dfd91643a17c1626032faada329"><code>@bdf9ead9</code></a>). (since Compat 4.3.0)</p>
</li>
<li>
<p dir="auto"><code>stack</code> combines a collection of slices into one array (<a href="https://github.com/JuliaLang/julia/issues/43334" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/43334/hovercard">#43334</a>). (since Compat 4.2.0)</p>
</li>
<li>
<p dir="auto"><code>keepat!</code> removes the items at all the indices which are not given and returns
the modified source (<a href="https://github.com/JuliaLang/julia/issues/36229" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/36229/hovercard">#36229</a>, <a href="https://github.com/JuliaLang/julia/issues/42351" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/42351/hovercard">#42351</a>). (since Compat 4.1.0)</p>
</li>
<li>
<p dir="auto"><code>@compat (; a, b) = (; c=1, b=2, a=3)</code> supports property descturing assignment syntax (<a href="https://github.com/JuliaLang/julia/issues/39285" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/39285/hovercard">#39285</a>).</p>
</li>
<li>
<p dir="auto"><code>allequal</code>, the opposite of <code>allunique</code> (<a href="https://github.com/JuliaLang/julia/issues/43354" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/43354/hovercard">#43354</a>). (since Compat 3.42.0)</p>
</li>
<li>
<p dir="auto"><code>eachsplit</code> for iteratively performing split(str). (<a href="https://github.com/JuliaLang/julia/issues/39245" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/39245/hovercard">#39245</a>). (since Compat 3.41.0)</p>
</li>
<li>
<p dir="auto"><code>ismutabletype(t::Type)</code> check whether a type is mutable (the field <code>mutable</code> of <code>DataType</code> was removed. <a href="https://github.com/JuliaLang/julia/issues/39037" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/39037/hovercard">#39037</a>) (since Compat 3.40)</p>
</li>
<li>
<p dir="auto"><code>convert(::Type{&lt;:Period}, ::CompoundPeriod)</code> can convert <code>CompoundPeriod</code>s into the specified <code>Period</code> type (<a href="https://github.com/JuliaLang/julia/issues/40803" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/40803/hovercard">#40803</a>) (since Compat 3.38.0)</p>
</li>
<li>
<p dir="auto"><code>Compat.@inline</code> and <code>Compat.@noinline</code> can be used at function callsites to encourage the compiler to (not) inline the function calls on Julia versions that support these features, and otherwise do not have any effects (<a href="https://github.com/JuliaLang/julia/issues/41312" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/41312/hovercard">#41312</a>) (since Compat 3.37)</p>
</li>
<li>
<p dir="auto"><code>Compat.@inline</code> and <code>Compat.@noinline</code> can be used within function body to hint to the compiler the inlineability of the defined function (<a href="https://github.com/JuliaLang/julia/issues/41312" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/41312/hovercard">#41312</a>) (since Compat 3.37)</p>
</li>
<li>
<p dir="auto"><code>Compat.@constprop :aggressive ex</code> and <code>Compat.@constprop :none ex</code> allow control over constant-propagation during inference on Julia versions that support this feature, and otherwise just pass back <code>ex</code>. (<a href="https://github.com/JuliaLang/julia/issues/42125" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/42125/hovercard">#42125</a>) (since Compat 3.36)</p>
</li>
<li>
<p dir="auto"><code>Returns(value)</code> returns <code>value</code> for any arguments (<a href="https://github.com/JuliaLang/julia/issues/39794" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/39794/hovercard">#39794</a>) (since Compat 3.35)</p>
</li>
<li>
<p dir="auto">The function <code>current_exceptions()</code> has been added to get the current
exception stack. Julia-1.0 lacks runtime support for full execption stacks,
so we return only the most recent exception in that case. (<a href="https://github.com/JuliaLang/julia/issues/29901" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/29901/hovercard">#29901</a>) (since
Compat 3.34)</p>
</li>
<li>
<p dir="auto">Two argument methods <code>findmax(f, domain)</code>, <code>argmax(f, domain)</code> and the corresponding <code>min</code> versions (<a href="https://github.com/JuliaLang/julia/issues/35316" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/35316/hovercard">#35316</a>, <a href="https://github.com/JuliaLang/julia/issues/41076" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/41076/hovercard">#41076</a>) (since Compat 3.31.1)</p>
</li>
<li>
<p dir="auto"><code>isunordered(x)</code> returns true if <code>x</code> is value that is normally unordered, such as <code>NaN</code> or <code>missing</code> (<a href="https://github.com/JuliaLang/julia/issues/35316" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/35316/hovercard">#35316</a>) (since Compat 3.31.1)</p>
</li>
<li>
<p dir="auto"><code>get</code> accepts tuples and numbers (<a href="https://github.com/JuliaLang/julia/issues/41007" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/41007/hovercard">#41007</a>, <a href="https://github.com/JuliaLang/julia/issues/41032" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/41032/hovercard">#41032</a>) (since Compat 3.31)</p>
</li>
<li>
<p dir="auto"><code>@something</code> and <code>@coalesce</code> as short-circuiting versions of <code>something</code> and <code>coalesce</code> (<a href="https://github.com/JuliaLang/julia/issues/40729" data-hovercard-type="pull_request" data-hovercard-url="/JuliaLang/julia/pull/40729/hovercard">#40729</a>) (since Compat 3.29)</p>
</li>
</ul>

<p dir="auto">One of the most important rules for <code>Compat.jl</code> is to avoid breaking user code
whenever possible, especially on a released version.</p>
<p dir="auto">Although the syntax used in the most recent Julia version
is the preferred compat syntax, there are cases where this shouldn't be used.
Examples include when the new syntax already has a different meaning
on previous versions of Julia, or when functions are removed from <code>Base</code>
Julia and the alternative cannot be easily implemented on previous versions.
In such cases, possible solutions are forcing the new feature to be used with
qualified name in <code>Compat.jl</code> (e.g. use <code>Compat.&lt;name&gt;</code>) or
reimplementing the old features on a later Julia version.</p>
<p dir="auto">If you're adding additional compatibility code to this package, the <a href="https://github.com/JuliaLang/julia/blob/master/contrib/commit-name.sh"><code>contrib/commit-name.sh</code></a> script in the base Julia repository is useful for extracting the version number from a git commit SHA. For example, from the git repository of <code>julia</code>, run something like this:</p>
<div class="highlight highlight-source-shell notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="bash $ contrib/commit-name.sh a378b60fe483130d0d30206deb8ba662e93944da
0.5.0-dev+2023"><pre>bash $ contrib/commit-name.sh a378b60fe483130d0d30206deb8ba662e93944da
0.5.0-dev+2023</pre></div>
<p dir="auto">This prints a version number corresponding to the specified commit of the form
<code>X.Y.Z-aaa+NNNN</code>, and you can then test whether Julia
is at least this version by <code>VERSION &gt;= v"X.Y.Z-aaa+NNNN"</code>.</p>

<p dir="auto">Note that you should specify the correct minimum version for <code>Compat</code> in the
<code>[compat]</code> section of your <code>Project.toml</code>, as given in above list.</p>
