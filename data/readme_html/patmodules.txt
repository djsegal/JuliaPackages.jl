<h1><a id="user-content-patmodulesjl" class="anchor" aria-hidden="true" href="#patmodulesjl"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>PatModules.jl</h1>
<p>Writing modular, reusable code in Julia is harder than in other languages. To get access to any particular file you need to <code>include(...)</code> it -- but each file can only be <code>include</code>-d once. (Otherwise the definitions are evaluated multiple times, possibly in different contexts, bad things start happening.) And if a file can only be <code>include</code>-d once, then it's hard to re-use its code.</p>
<p>Previous "best practice" has been to do all <code>include</code>s in some "master" global file, and have everything else implicitly assume that whatever it needs <code>include</code>-ing will in fact have been included for it. You may need to get the <code>include</code>s in the global file in the right order, and you have very little way of tracking what actually depends on what. That's hard to maintain, and not a good way to scale to projects larger than just a few files.</p>
<p>&gt;&gt;&gt; <strong>PatModules.jl is the solution.</strong> &lt;&lt;&lt;</p>

<p>Have a look at <a href="./examples/example">the example</a> for a quick example. In particular, notice that:</p>
<ul>
<li><code>subpackage/one.jl</code> and <code>subpackage/two.jl</code> <em>both explicitly</em> depend on <code>subpackage/utils.jl</code>, without errors due to redefining a module.</li>
<li>both <code>utils.jl</code> and <code>subpackage/utils.jl</code> create modules with the same name without them clashing.</li>
<li><code>example.jl</code> can include the whole "subpackage" <em>folder</em> just by specifying "subpackage", and can descend into subsubpackages at will.</li>
<li>Every file can be run stand-alone: no implicit assumptions about the context in which the file will be <code>include</code>d.</li>
</ul>
<h3><a id="user-content-remind-me-how-includeimportusing-works-in-julia" class="anchor" aria-hidden="true" href="#remind-me-how-includeimportusing-works-in-julia"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Remind me how <code>include</code>/<code>import</code>/<code>using</code> works in Julia?</h3>
<p>Briefly: <code>include</code> just copy-pastes the contents of one <em>file</em> into another <em>file</em>. (Note that this is generally considered a bad way to move code around, see <a href="https://stackoverflow.com/questions/13570947/what-is-the-difference-between-import-and-include-choices-in-language-design" rel="nofollow">this</a> StackOverflow post.) Meanwhile, <code>import</code>/<code>using</code> get the objects defined in one <em>module</em>, and make them available in another <em>module</em>.</p>
<p>What this means for writing re-usable code: you're forced to use <code>include</code> just to make the contents of one file visible to another, and then <em>also</em> use either <code>import</code>/<code>using</code> to access it in Julia afterwards. Besides a two-step process being a bit ugly, this has all the issues mentioned in the introduction.</p>
<h3><a id="user-content-how-does-patmodulesjl-work--how-do-i-use-it" class="anchor" aria-hidden="true" href="#how-does-patmodulesjl-work--how-do-i-use-it"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>How does PatModules.jl work / how do I use it?</h3>
<p>PatModules.jl introduces two new macros, <code>@mainmodule</code> and <code>@auxmodule</code>. <strong>Between them, you should never have to write a single <code>include</code> again.</strong></p>
<p>Every folder must have a file of the same name, e.g. <code>subpackage/subpackage.jl</code>, which is denoted a "main module". (Equivalent to <code>__init__.py</code> in Python.) Every other file is an "auxiliary module".</p>
<p>The syntax for both is</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="&lt;@mainmodule|@auxmodule&gt; &lt;name&gt; (&quot;&lt;import1&gt;&quot;, &quot;&lt;import2&gt;&quot;, ...) begin
    &lt;module contents&gt;
end
"><pre><span class="pl-k">&lt;</span><span class="pl-c1">@mainmodule</span><span class="pl-k">|</span><span class="pl-c1">@auxmodule</span><span class="pl-k">&gt;</span> <span class="pl-k">&lt;</span>name<span class="pl-k">&gt;</span> (<span class="pl-s"><span class="pl-pds">"</span>&lt;import1&gt;<span class="pl-pds">"</span></span>, <span class="pl-s"><span class="pl-pds">"</span>&lt;import2&gt;<span class="pl-pds">"</span></span>, <span class="pl-k">...</span>) <span class="pl-k">begin</span>
    <span class="pl-k">&lt;</span><span class="pl-k">module</span> contents<span class="pl-k">&gt;</span>
<span class="pl-k">end</span></pre></div>
<p>for example</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="@auxmodule my_amazing_module (&quot;another_module&quot;,) begin
    function my_amazing_function()
        another_module.some_functionality()
        ...
    end
    ...
end
"><pre><span class="pl-c1">@auxmodule</span> my_amazing_module (<span class="pl-s"><span class="pl-pds">"</span>another_module<span class="pl-pds">"</span></span>,) <span class="pl-k">begin</span>
    <span class="pl-k">function</span> <span class="pl-en">my_amazing_function</span>()
        another_module<span class="pl-k">.</span><span class="pl-c1">some_functionality</span>()
        <span class="pl-k">...</span>
    <span class="pl-k">end</span>
    <span class="pl-k">...</span>
<span class="pl-k">end</span></pre></div>
<p><em>Each file should have at its top level a single <code>@mainmodule</code>/<code>@auxmodule</code> with the same name as the file.</em> For example <code>subpackage/utils.jl</code> should have (and only have) <code>@auxmodule utils ...</code> at its top level.</p>
<p>The next argument is a tuple of strings, specifying what you'd like to import. These can be from among the other auxiliary modules in the same folder, or the names of any subfolders (with associated main modules one level down) in the same folder. The contents of these files will be <code>include</code>d in the correct manner, and will also all be <code>import</code>ed automatically. For example see how <a href="./examples/example/subpackage/subpackage.jl">subpackage.jl</a> gets access to <code>one</code> and <code>two</code>. (If you would prefer it is <code>using</code>-d automatically then use <code>@mainmodule!</code> and <code>@auxmodule!</code> instead, see <a href="./examples/example_use"><code>example_use</code></a>.)</p>
<p>If you want to access something in a subsubfolder, subsubsubfolder, etc., then do so by importing the subfolder and then doing dotted look up. (For example see how <a href="./examples/example/example.jl">example.jl</a> gets access to <code>subpackage.two</code>.) Do <em>not</em> try to <code>include</code> the subsub-* directly: this kind of behaviour is dangerous, and will likely result in two mutually unintelligible copies of the same code being compiled. (Which recall was one of the things that PatModules.jl sought to fix in the first place.)</p>
<p>If you want to access some globally installed module (i.e. from outside your project) then do <code>import</code>/<code>using</code> in the usual way, <em>within</em> the block of the <code>@auxmodule</code> or <code>@mainmodule</code>. (If it's outside that then the contents of your module can't see it, which is normal in Julia.)</p>
<p>If you want to access some nonlocal module, then also do that with <code>import</code>/<code>using</code>, like doing <code>import ...mypackage</code>. Note that in this case the <code>include</code> part is not guaranteed to happen automatically, but provided the relevant module is imported with PatModules.jl at the higher level, then this should still work.</p>

<h3><a id="user-content-i-want-a-lower-level-interface" class="anchor" aria-hidden="true" href="#i-want-a-lower-level-interface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>I want a lower-level interface.</h3>
<p>PatModules.jl also makes available another macro, <code>@once</code>, which can be called on a module definition so that it is only created once, <em>within the same context</em>, i.e. the outer module in which it is called. (It won't save you from <code>include</code>-ing the module in multiple different places.) This is what <code>@mainmodule</code> and <code>@auxmodule</code> rely on to not have to worry about importing things multiple times. Have a look at <a href="./examples/example_once"><code>example_once</code></a> for how this is used; this example mirrors the main <a href="./examples/example"><code>example</code></a>, so you can see what <code>@mainmodule</code> and <code>@auxmodule</code> get converted into.</p>
<h3><a id="user-content-why-is-it-implemented-this-way" class="anchor" aria-hidden="true" href="#why-is-it-implemented-this-way"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Why is it implemented this way?</h3>
<p>There's many sensible alternative ways of creating these sorts of import systems. One way would be to emulate Python: store all the modules somewhere global (like <code>sys.modules</code> in Python) and then route all import calls there. Another way would be to build a tree of modules and then linearise it with a topological sort relative to some specified root / entry point.</p>
<p>The main advantage of the approach we use here is that after macro expansion the end result still looks like normal Julia, and doing so doesn't involve too much magic. (Which the metaprogramming involved in the other approaches probably would be.) This means that PatModules.jl should be able to be used (or introduced incrementally into a project) without too many surprises.</p>
