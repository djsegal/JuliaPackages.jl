<p dir="auto">The main aim of this package is to implement concepts of graph theory on hypergraphs. At the most basic level, it allows to represent high-order relationships between objects, with complete freedom to choose the type of these objects. A secondary aim follows from the realisation that most flavours of (hyper)graphs are a specific case of oriented, weighted hypergraphs; from this, it should be possible to define all methods at the hypergraph level and to programmatically specialise them for other (hyper)graph types.</p>
<p dir="auto">Currently implemented are fundamental constructors for unoriented, unweighted hypergraphs (via the <code>HyperGraph</code> and <code>HyperEdge</code> types) and for a specific case of oriented, weighted hypergraphs: chemical hypergraphs (via the <code>ChemicalHyperGraph</code> and <code>ChemicalHyperEdge</code> types), as well as a set of functions that allow to modify hypergraphs (adding and deleting hyperedges and vertices). Flavours of hypergraphs and hyperedges are implemented with traits via <a href="https://github.com/mauro3/SimpleTraits.jl">SimpleTraits.jl</a>; traits implemented so far are <code>IsOriented</code> and <code>IsWeighted</code>.</p>
<p dir="auto">For some example applications of HyperGraphs.jl, check out its sibling package <a href="https://github.com/lpmdiaz/Simulacrum.jl">Simulacrum.jl</a>.</p>
<h2 dir="auto"><a id="user-content-notation-and-naming-rules" class="anchor" aria-hidden="true" href="#notation-and-naming-rules"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Notation and naming rules</h2>
<p dir="auto">In the code, <code>x</code> and <code>xs</code> refer to one and several hypergraphs, respectively; the same applies for <code>e</code> and <code>es</code> with hyperedges, and for <code>v</code> and <code>vs</code> with vertices.</p>
<p dir="auto">The current idea to allow for natural extension of the core functions is to respect a set of standard field names when defining a new custom concrete type. If custom field names are needed, these should be explicitly connected to core methods (<code>vertices</code> and <code>hyperedges</code> mainly). Currently, field names should be:</p>
<ul dir="auto">
<li><code>V</code>, the set of vertices in a hypergraph and a hyperedge</li>
<li><code>E</code>, the set of hyperedges in a hypergraph</li>
<li><code>src</code>, the set of source vertices in an oriented hyperedge</li>
<li><code>tgt</code>, the set of target vertices in an oriented hyperedge</li>
<li><code>objs</code>, the objects of an incidence</li>
<li><code>mults</code>, the multiplicities of the objects of an incidence</li>
<li><code>w</code>, the weight of a weighted hyperedge</li>
</ul>
<p dir="auto">Then, custom names are built on top e.g. <code>rate(e::ChemicalHyperEdge) = weight(e)</code> (which is already implemented).</p>
<p dir="auto">These symbols were chosen to differentiate hypergraphs from graphs (all graphs are hypergraphs but not all hypergraphs are graphs), and so a hypergraph is defined as X = (V, E). An alternative option was to name hypergraphs with the letter H; however, following that logic would have meant changing the symbol for hyperedges too, which would have been impractical (also H). This way, using X emphasises the difference between hypergraphs and graphs and using E for edges retains some continuity with ordinary graphs notation G = (V, E).</p>
<p dir="auto">Functions are defined on hypergraphs <code>x</code> and edges <code>e</code> for consistency. It may be useful to add some clarity and differentiate between variables though, and so a chemical hypergraph may be defined as <code>chx</code> and a chemical hyperedge as <code>che</code>.</p>

<p dir="auto">A word of caution: this is a work in progress, and so functions have not been perfectly proofread for correctness (especially those in operations, neighbor functions, parallel and multi-hyperedges, loops...). E.g. currently, it is possible to add hyperedges to a hypergraph that does not have all the vertices in said hyperedges in its vertex set; this will be prevented from happening eventually.</p>
<p dir="auto">Each graph flavour should be implemented with performance and ease of interfacing (i.e. of accessing information) in mind: there is no need to carry redundant information just because the mathematical syntax does. Hopefully this is true in the current code, but some improvements are definitely possible.</p>

<p dir="auto">The source and the target of an oriented hyperedge are also referred to as head and tail, but the former notation is more explicit.</p>

<p dir="auto">Default field values are implemented by overloading <code>Base.getproperty</code>. This means <code>object.some_field</code> can return a value despite <code>object</code> having no corresponding <code>:some_field</code> field.</p>
<p dir="auto">So far, objects of type <code>AbstractHyperEdge</code> have a default <code>:weight</code> value of <code>1</code>.</p>
<h4 dir="auto"><a id="user-content-chemical-hypergraphs-and-chemical-hyperedges" class="anchor" aria-hidden="true" href="#chemical-hypergraphs-and-chemical-hyperedges"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Chemical hypergraphs and chemical hyperedges</h4>
<p dir="auto">Chemical hypergraphs represent reaction networks but are rooted in graph theory. The main reference is probably <a href="#Jost2019">[Jost2019]</a>.</p>
<p dir="auto">In this implementation, I took the liberty to name <em>chemical hyperedges</em> the reactions, following the same logic as naming <em>chemical hypergraph</em> the hypergraph that represents a system of reactions (this is because in each case it is a specific case of hypergraph and hyperedge, namely one where the hyperedge incidence multiplicity is restricted to the positive integers, thus representing stoichiometries).</p>
<p dir="auto">Graph theoretical concepts have natural interpretations in the context of chemical reactions represented on chemical hypergraphs. For instance, a vertex-hyperedge incidence multiplicity represents how many times a vertex occurs in a hyperedge and as such encodes stoichiometry information, while a hyperedge-hypergraph incidence multiplicity encodes for parallel hyperedges (this is not currently implemented).</p>
<p dir="auto">Constructors set 1 as the default value for reactions rate and for stoichiometries; this is to simplify the syntax. For instance, all these calls are equivalent: <code>SpeciesSet(["X"])</code>, <code>SpeciesSet("X")</code>, <code>SpeciesSet("X", 1)</code>, <code>SpeciesSet(["X"], [1])</code>. Compared to other implementations of reaction networks, it is also easier to specify a different stoichiometry for only part of the reaction (e.g. if one needs only some of the reactants to have stoichiometries different from 1, only the reactant stoichiometries have to be specified and not those of all the species involved).</p>

<h4 dir="auto"><a id="user-content-vertex-degree" class="anchor" aria-hidden="true" href="#vertex-degree"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Vertex degree</h4>
<p dir="auto">The usual definition of the <em>degree</em> of a vertex v is <em>the number of edges incident on v</em>; this definition however breaks down in the case of a loop: following it strictly implies a loop has degree 1 (a loop is one edge incident on one vertex) when it is generally agreed it has degree 2 <a href="#Bretto2013">[Bretto2013]</a>, <a href="#Bollobas1998">[Bollobas1998]</a>, <a href="#Kaminski2019">[Kaminski2019]</a>, <a href="#Zaslavsky1982">[Zaslavsky1982]</a>.</p>
<p dir="auto">A more general definition is then probably <em>the number of incidences at v</em>, as given by Zaslavsky <a href="#Zaslavsky1982">[Zaslavsky1982]</a>. Following this more general definition gives us degree 2 for a loop (since a loop on v is twice incident on v <a href="#Zaslavsky1982">[Zaslavsky1982]</a>) while working in the same way as the former definition in other cases.</p>
<p dir="auto">Here we implement an even more general definition of degree: the degree of vertex v is defined as <em>the sum of the weights of edges incident on v</em>, where the weight of edge e appears according to the multiplicity of v in e. In other words, the degree of v is given by the sum over edges of edge weight times the number of incidences of v in each edge. Other definitions discussed above are special cases of this more general definition.</p>
<p dir="auto">This however means that <code>degree</code> may now return values that are not of type <code>Number</code>, depending on what type edge weights are. This may cause unexpected behaviours since some concepts rely on e.g. <code>maximum</code> or <code>sum</code> being defined on degree values. Those effectively assume that degrees are defined on the positive integers including zero.</p>
<p dir="auto">The vertex degree is also referred to as <em>valency</em>. Additionally, the <em>degree</em> of a graph is its maximum vertex degree <a href="#Zhu2019">[Zhu2019]</a>, which is not implemented here to avoid confusion.</p>
<h4 dir="auto"><a id="user-content-edge-cardinality" class="anchor" aria-hidden="true" href="#edge-cardinality"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Edge cardinality</h4>
<p dir="auto">The <em>cardinality</em> of an edge is its <em>number of endpoints</em>; this can be interpreted in different ways, which becomes obvious when working with loops. If the vertices of an edge are treated as a multiset or as an <em>n</em>-tuple (e.g. when working with oriented edges), a loop should have cardinality 2. This comes from the fact that a loop has two (coinciding) endpoints (<a href="#Zaslavsky1982">[Zaslavsky1982]</a>, <a href="#Zaslavsky1991">[Zaslavsky1991]</a>), with the cardinality of a multiset being the sum of the multiplicities of its elements. If it is instead treated as a set, a loop has cardinality 1, as is often defined (e.g. in <a href="#Dorfler1980">[Dorfler1980]</a>, <a href="#Berge1989">[Berge1989]</a>, <a href="#Bretto2013">[Bretto2013]</a>). Concisely, a loop has cardinality <strong>2</strong> when counting its number of endpoints; <strong>2</strong> when counting the number of elements of a multiset / <em>n</em>-tuple; <strong>1</strong> when counting the number of elements of a set. This seems to be what is suggested on page 4 of  <a href="#Spivak2009">[Spivak2009]</a>: the map that sends the set of non-empty tuples on a set S (i.e. edges) to the set non-empty subsets of S (i.e. vertices) "may decrease cardinality." The approach taken here is to adopt the most general definition (i.e. considering the vertices of edges as a multiset / an <em>n</em>-tuple) which then leaves freedom to build more specialised functions on top.</p>
<p dir="auto">Hyperedge cardinality is also referred to as <em>order</em> <a href="#Zhu2019">[Zhu2019]</a>, and as <em>size</em>. Neither of these are implemented; the former may be confused with the order of a hypergraph, and the latter may conflict with <code>Base.size</code>.</p>

<p dir="auto">The <em>size</em> of a hypergraph is defined in <a href="#Gallo1993">[Gallo1993]</a> and in <a href="#Cambini1997">[Cambini1997]</a>; this is implemented as <code>hypergraph_size</code> to avoid confusion with <code>Base.size</code> again.</p>
<p dir="auto">The <em>order</em> of a hypergraph is its number of vertices <a href="#Wang2018">[Wang2018]</a>, <a href="#Zaslavsky2010">[Zaslavsky2010]</a>. (Also note that order is used to refer to the maximum cardinality of a hypergraph in <a href="#Zhu2019">[Zhu2019]</a>, but this use seems unusual.)</p>
<p dir="auto">Note that the <em>rank</em> of a reaction network is the maximum number of linearly independent reactions <a href="#Shinar2010">[Shinar2010]</a>, which may be confused with the rank of a hypergraph.</p>
<p dir="auto">A reference for the <em>volume</em> of a hypergraph is <a href="#Kaminski2019">[Kaminski2019]</a>.</p>

<p dir="auto">Functions only do what their name implies, and so if a loop is undesirable in a specific application, one must check that e.g. the hyperedge that is about to be added to a hypergraph is not a loop.</p>
<p dir="auto"><em>Weak</em> deletion only deletes incidences, whereas <em>strong</em> deletion deletes any object incident on the object(s) being deleted; this means that e.g. weak vertex deletion removes any occurrence of the given vertex in the hyperedges incident on that vertex but does not delete those hyperedges <a href="#Chen2018">[Chen2018]</a>, <a href="#Rusnak2013">[Rusnak2013]</a>. Note: vertex deletion is (incidence) dual to edge deletion <a href="#Rusnak2013">[Rusnak2013]</a>.</p>
<p dir="auto">Note: currently, the internal function <code>_del_vertex!(e)</code> must have two methods: one for unoriented and one for oriented hyperedges; this is because the former method only needs to remove a vertex from <code>vertices(e)</code> when the latter needs to remove it from both the source and target sets (and both objects and multiplicities), which does not naturally work with <code>vertices</code> of an oriented hyperedge. This is somewhat messy; ideally, as mentioned above, methods would be built automatically from the hyperedge flavour (here oriented vs. unoriented), via traits.</p>
<p dir="auto">Edge <em>switching</em> is implemented at a high level by swapping the source and target sets of an oriented hyperedge. Switching is more fundamentally defined as the negation of incidences, e.g. in <a href="#Reff2012">[Reff2012]</a>, <a href="#Rusnak2013">[Rusnak2013]</a>.</p>

<p dir="auto">This is mainly a personal repository to play around with hypergraphs, but I do plan to add more functionalities over time.</p>
<p dir="auto">Note a potential future breaking change about the behaviour of <code>src</code> and <code>tgt</code>: these currently return the objects of the set of incidences but may return the set of incidences themselves in the future, depending on what makes sense. This also means that currently, the extension of <code>Base.==</code> does not check for equal multiplicity (which may be slightly incorrect).</p>
