<p><a href="https://ci.appveyor.com/project/oxinabox/multiresolutioniterators-jl/branch/master" rel="nofollow"><img src="https://camo.githubusercontent.com/b53442640f35dd87ba702dc7cdd93ffc808d373aa2de41327ffc64c6bc75db79/68747470733a2f2f63692e6170707665796f722e636f6d2f6170692f70726f6a656374732f7374617475732f747275746138786b743965326b356d692f6272616e63682f6d61737465723f7376673d74727565" alt="Build status" data-canonical-src="https://ci.appveyor.com/api/projects/status/truta8xkt9e2k5mi/branch/master?svg=true" style="max-width:100%;"></a></p>
<p>There are many different ways to look at text corpora.
The true structure of a corpus might be:</p>
<ul>
<li><strong>Corpus</strong></li>
<li>made up of: <strong>Documents</strong></li>
<li>made up of: <strong>Paragraphs</strong></li>
<li>made up of: <strong>Sentences</strong></li>
<li>made up of: <strong>Words</strong></li>
<li>made up of: <strong>Characters</strong></li>
</ul>
<p>Very few people want to consider it at that level.</p>
<ul>
<li>Someone working in <strong>Information Retrieval</strong> might want to consider the corpus as <strong>Corpus made up of Documents made up of Words</strong>.</li>
<li>Someone working on <strong>Language Modeling</strong> might want to consider <strong>Corpus made up of Words</strong></li>
<li>Someone working on <strong>Parsing</strong> might want to consider <strong>Corpus made up Sentences made up of Words</strong>.</li>
<li>Someone training a <strong>Char-RNN</strong> might want to consider <strong>Corpus made up of Characters</strong>.</li>
</ul>
<p>This package lets you better work with iterators of iterators to allow them to be flattened and viewed at different levels.</p>
<h2><a id="user-content-operations" class="anchor" aria-hidden="true" href="#operations"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Operations</h2>
<h3><a id="user-content-consolidate" class="anchor" aria-hidden="true" href="#consolidate"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><code>consolidate</code></h3>
<p>This is a variation on <code>collect</code>.
It only promises the returned result will be indexable,
not that it will be a <code>Vector</code>.
<code>consolidate</code>ing iterators with an <code>eltype</code> of <code>Char</code>, will give you Strings, which for most of our uses is better than <code>Vector{Char}</code>.
Beyond its use for <code>consolidate_levels</code> it is also useful as a nonleveled operation as it may be faster than convert, and/or make use of costomized behavour from  overloading <code>apply</code>, (see <a href="#customizing-behavour">Customizing Behavour</a>).</p>
<h3><a id="user-content-consolidate_levels--full_consolidate" class="anchor" aria-hidden="true" href="#consolidate_levels--full_consolidate"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><code>consolidate_levels</code> &amp; <code>full_consolidate</code></h3>
<p>This converts the given levels from iterators to <code>Vector</code>s.
The most useful is likely <code>consolidate(iter, ALL_LEVELS)</code> which we export under the alias <code>full_consolidate</code>.</p>
<h3><a id="user-content-flatten_levels" class="anchor" aria-hidden="true" href="#flatten_levels"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><code>flatten_levels</code></h3>
<p>This is the levelled version of flatten.
<code>flatten_levels(iter, 1)</code> is the same as <code>Base.Iterators.flatten(iter)</code>.
<code>flatten_levels(iter, 2)</code> is the same as <code>Base.Iterators.flatten.(iter)</code> (assuming iter is Vector or similar)
<code>flatten_levels(iter, 1:2)</code> is the same as <code>Base.Iterators.flatten(Base.Iterators.flatten.(iter))</code>
<code>flatten_levels(iter, ALL_LEVELS)</code> results in a fully flat output.</p>
<h3><a id="user-content-join_levels" class="anchor" aria-hidden="true" href="#join_levels"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a><code>join_levels</code></h3>
<p>This is a generalization of <code>join(strings, delim)</code>
Pass in a dictionary from levels to the character/string to be used to join that level.
e.g. <code>join_levels(animal_info, Dict(2=&gt;"\n", 3=&gt;" "))</code></p>
<h2><a id="user-content-customizing-behavior" class="anchor" aria-hidden="true" href="#customizing-behavior"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Customizing Behavior</h2>
<h3><a id="user-content-controlling-return-type-of-applying-functions-to-levelselements-by-overloading-apply" class="anchor" aria-hidden="true" href="#controlling-return-type-of-applying-functions-to-levelselements-by-overloading-apply"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Controlling return type of applying functions to levels/elements by overloading <code>apply</code></h3>
<p>MultiResolutionIterators will by default destroy all types at all levels it touches,
because it needs to replace their iterators with new modified versions (from some operation happening far below).
To avoid that you can overload <code>MultiResolutionIterators.apply(f, xs::MyType)</code>,
which is the internal function that is used to apply <code>f</code> to <code>xs</code> -- it defaults to <code>f(xs)</code>.
You could overload it to <code>MultiResolutionIterators.apply(f, xs::MyType) = MyType(f(xs)</code> for example.</p>
<h3><a id="user-content-make-named-levels-functionality-work-by-defining-a-levelname_map-for-an-indexer" class="anchor" aria-hidden="true" href="#make-named-levels-functionality-work-by-defining-a-levelname_map-for-an-indexer"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Make named levels functionality work by defining a <code>levelname_map</code> for an Indexer.</h3>
<p>To make <code>lvls</code> work to provide <strong>named levels</strong> functionality,
you need to define some type to be your indexer,
and overload <code>MultiResolutionIterators.levelname_map</code>
to return a vector of Pairs mapping your index (normally some symbol or a string, but could be anything, baring an Integer) to the integer that is the level number.</p>
<h2><a id="user-content-usage" class="anchor" aria-hidden="true" href="#usage"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Usage</h2>
<p>A simple example we have a corpus, made of documents (on about turtles and one about cats).
The documents are made up of sentences, which are made up of words, which are made up of characters.</p>
<p>The basic usage is to specify levels to act on by directly specifing the number.
The more advances usage is to declare an <strong>indexer</strong>, then refer to the levels by <strong>name</strong>.</p>

<h3><a id="user-content-working-with-named-levels" class="anchor" aria-hidden="true" href="#working-with-named-levels"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Working with Named Levels</h3>
<p>If we declare an indexer,
we can use <code>lvls(indexer, names...)</code> to select which level to include by name,
or <code>(!lvls)(indexer, names...)</code> to select which levels to exlude by name.
Using level numbers also works with <code>lvls</code> and <code>(!lvls)</code>.</p>
