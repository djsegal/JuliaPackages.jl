<h2><a id="user-content-usage" class="anchor" aria-hidden="true" href="#usage"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Usage</h2>
<p>**Warning: This part of the documentation is not up to date, the documentation website will be updated soon, the notebook tutorial is up to date with AutoViz v0.8 **</p>
<p>AutoViz works by adding rendering instructions to a <code>RenderModel</code>,
and finally applying those instructions to a canvas using the
<code>render</code> function.</p>
<h3><a id="user-content-basic-usage" class="anchor" aria-hidden="true" href="#basic-usage"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Basic Usage</h3>
<p>In the simplest case, a <code>roadway</code> and a <code>scene</code> of types <code>Roadway</code> and <code>Scene</code>
respectively can be rendered via</p>
<div class="highlight highlight-source-julia"><pre><span class="pl-c1">render</span>([roadway, scene])</pre></div>
<p>However, the rendering interface is much more flexible than that,
supporting custom rendering and cameras.</p>
<h3><a id="user-content-renderable-objects" class="anchor" aria-hidden="true" href="#renderable-objects"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Renderable Objects</h3>
<p>In order for an object of type <code>ObjectType</code> to be "renderable",
we need to provide a function with the signature</p>
<div class="highlight highlight-source-julia"><pre>    <span class="pl-c1">add_renderable!</span>(rendermodel<span class="pl-k">::</span><span class="pl-c1">RenderModel</span>, obj<span class="pl-k">::</span><span class="pl-c1">ObjectType</span>)</pre></div>
<p>The basic example above works, because AutoViz implements the
<code>add_renderable!</code> function for commonly used types such as
<code>Roadway</code>, <code>Entity</code> or <code>Scene</code>.
In general, the <code>render(renderables)</code> function can take any collection
of renderable objects.
AutoViz provides a series of convenient wrapper objects such as
<code>FancyCar</code>, <code>FancyPedestrian</code>, <code>EntityRectangle</code> to make entities
renderable.</p>
<h3><a id="user-content-cameras" class="anchor" aria-hidden="true" href="#cameras"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Cameras</h3>
<p>A <code>Camera</code> object specifies what portion of the scene should be rendered.</p>
<p>By default, <code>render(renderables)</code> uses a static camera.
In order to use an adaptive camera, the <code>camera</code> object
needs to be updated before rendering and passed in to the
<code>render</code> function as a keyword argument.</p>
<div class="highlight highlight-source-julia"><pre><span class="pl-c1">update_camera!</span>(scene, camera)
<span class="pl-c1">render</span>([roadway, scene], camera<span class="pl-k">=</span>camera)</pre></div>

<h3><a id="user-content-how-to-make-types-renderable" class="anchor" aria-hidden="true" href="#how-to-make-types-renderable"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>How to make types renderable</h3>
<p>There are two ways to make renderable types.</p>
<ol>
<li>You can make your existing types renderable by conversion by defining <code>convert(::Type{Renderable}, ::MyType)</code> which should return a directly renderable object, e.g. an <code>ArrowCar</code>.</li>
<li>You can make types directly renderable by defining <code>render!(::RenderModel, ::MyType)</code>. To make things easier for the compiler, you can also define <code>isrenderable(::Type{MyType}) = true</code>. If you want to allow others to convert to this type to make their types renderable by conversion, make your type a subtype of <code>Renderable</code>.</li>
</ol>

</ul>
<h2><a id="user-content-saving-images" class="anchor" aria-hidden="true" href="#saving-images"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Saving images</h2>
<p>Png images can be saved with <code>write_to_png(render(scene), "filename.png")</code> or <code>write_to_svg(render(scene), "filename.svg")</code>.
Gif animations may be created with e.g. <a href="https://github.com/shashi/Reel.jl">Reel.jl</a>.</p>
<h2><a id="user-content-rendermodels" class="anchor" aria-hidden="true" href="#rendermodels"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a><code>RenderModel</code>s</h2>
<p>The mid-level interface for this package (which is what you will use when you write <code>render!()</code> for your types or when you write an overlay) revolves around adding instructions to a <code>RenderModel</code>. Each instruction consists of a function and a tuple of arguments for the function. This is not documented in this readme, but it is fairly easy to figure out by reading <code>rendermodels.jl</code>, <code>overlays.jl</code>, and <code>arrowcar.jl</code>.</p>
<h2><a id="user-content-customization" class="anchor" aria-hidden="true" href="#customization"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Customization</h2>
<p>AutoViz.jl has two display mode: a "fancy" mode (default) that uses the svg representations in <code>icons/</code> to display cars and pedestrian, and a more basic mode where cars are rendered as rounded rectangles. To turn-off the fancy mode you can run:</p>
<div class="highlight highlight-source-julia"><pre>AutoViz<span class="pl-k">.</span><span class="pl-c1">set_render_mode</span>(<span class="pl-c1">:basic</span>) <span class="pl-c"><span class="pl-c">#</span> set to :fancy for fancy mode</span></pre></div>
<p>In addition you can also change the color theme. Three color themes are provided: <code>MONOKAY</code> (default), <code>OFFICETHEME</code>, <code>LIGHTTHEME</code>. You can change the color theme by running:</p>
<div class="highlight highlight-source-julia"><pre><span class="pl-k">using</span> AutoViz
<span class="pl-c1">set_color_theme</span>(LIGHTTHEME)</pre></div>
<p>You can also define your own color theme using a dictionary. Look at the example in <code>src/colorscheme.jl</code> to have the correct key names.</p>
<h2><a id="user-content-change-log" class="anchor" aria-hidden="true" href="#change-log"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Change Log</h2>
<h3><a id="user-content-v08x" class="anchor" aria-hidden="true" href="#v08x"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>v0.8.x</h3>
<h4><a id="user-content-rendering" class="anchor" aria-hidden="true" href="#rendering"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Rendering</h4>
<ul>
<li>Clean-up of the rendering interface: there is now only one single rendering function with the signature</li>
</ul>
<pre><code>render!(rendermodel::RenderModel, renderables::AbstractVector; canvas_width::Int, canvas_height::Int, surface::CairoSurface))
</code></pre>
<p>All keyword arguments are optional. Objects of type <code>Renderable</code> now no longer have to implement the <code>render!</code> function (which is a misleading name). Instead one must implement the <code>add_renderable!</code> function which adds the rendering instructions to the <code>RenderModel</code>.</p>
<ul>
<li>Implicit conversions of non-renderable objects (such as <code>obj::Scene{Entity{S,D,I}}</code>) via implementations of <code>Base.convert(Renderable, obj)</code> are now discouraged. Instead, one can overwrite the <code>add_renderable!</code> method for such types. This is done for some very common types.</li>
<li>The new <code>render!</code> function now only takes objects which are renderable, i.e. which implement the <code>add_renderable(rm::RenderModel, obj)</code> function. There is no longer a distinction between drawing roadways, scenes or overlays. They all need to satisfy the same interface, and they are drawn in the order in which they are passed to the <code>render!</code> function. This change decreases the number of available render functions from almost ten to one and should make the control flow more clear.</li>
<li>Additional arguments to <code>render!</code> such as <code>camera</code> and <code>car_colors</code> are no longer supported. Camera effects should be applied before calling <code>render!</code> (see section below) and rendering attributes such as colors should be passed in as part of a renderable object.</li>
</ul>
<h4><a id="user-content-overlays-1" class="anchor" aria-hidden="true" href="#overlays-1"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Overlays</h4>
<ul>
<li>Changed the interface for rendering overlays to only take an instance of <code>RenderModel</code> and the overlay itself. All additional data must be stored as part of the overlay if it is needed during rendering.</li>
<li>Added a <code>RenderableOverlay</code> wrapper which makes the legacy overlays work with the new rendering interface (in which overlays do not get any input arguments for rendering)</li>
</ul>
<h4><a id="user-content-cameras-1" class="anchor" aria-hidden="true" href="#cameras-1"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Cameras</h4>
<ul>
<li>Changed the camera interface. The full state of the camera, such as <code>camera_pos</code>, <code>camera_zoom</code>, <code>camera_rotation</code> is stored in <code>RenderModel</code> (this has already been the case in previous AutoViz versions). A <code>Camera</code> acts upon a <code>RenderModel</code> by changing these internal variables. The function <code>camera_set!</code> now becomes <code>update_camera!</code>.</li>
<li>Many setter functions for the camera have been replaced by the <code>set_camera!()</code> function which takes keyword arguments for <code>x</code>, <code>y</code> and <code>zoom</code>.</li>
<li>The implementations of <code>TargetFollowCamera</code> (former <code>CarFollowCamera</code>) and <code>SceneFollowCamera</code> have been reviewed and simplified. Additionally, a <code>ZoomingCamera</code> type which gradually changes the zoom level has been introduced and for easy extensibility there is also a <code>ComposedCamera</code> type which takes a list of cameras and applies their effects sequentially to the <code>RenderModel</code>.</li>
<li>The new <code>render!</code> function no longer takes a camera as an input argument, but assumes that the camera settings have already been applied to the <code>RenderModel</code> via <code>update_camera!</code> prior to calling <code>render!</code>. User code should be adapted accordingly.</li>
</ul>
<h4><a id="user-content-visualization-of-entities" class="anchor" aria-hidden="true" href="#visualization-of-entities"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Visualization of Entities</h4>
<ul>
<li>Controlling the appearance of vehicles by setting <code>set_render_mode(:basic|:fancy)</code> is no longer encouraged. Instead, we provide new renderable types such as <code>EntityRectangle</code>, <code>FancyCar</code>, <code>FancyPedestrian</code>, <code>VelocityArrow</code> in addition to the already implemented <code>ArrowCar</code> type which can all be used to conveniently display entities.</li>
<li>A convenience function for rendering scenes directly (i.e. without explicit conversion to a <code>Renderable</code> type) is still supported.</li>
<li>TODO: make FancyCar work on my platform</li>
</ul>
<h4><a id="user-content-1d-vehicles" class="anchor" aria-hidden="true" href="#1d-vehicles"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>1D Vehicles</h4>
<ul>
<li>Support for 1D vehicles has mostly been discontinued and some of the related functions were removed. However, the new functions should work seamlessly in many cases as long as the 1D vehicles implement basic functions such as <code>posg</code>, <code>width</code>, <code>length</code> from <code>AutomotiveDrivingModels.jl</code></li>
</ul>
