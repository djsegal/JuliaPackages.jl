<p dir="auto">This Julia package provides support for decoding and encoding texts between multiple character encodings. It is currently based on the iconv interface, and supports all major platforms using <a href="https://www.gnu.org/software/libiconv/" rel="nofollow">GNU libiconv</a>. In the future, native Julia support for major encodings will be added.</p>

<h2 dir="auto"><a id="user-content-the-encoding-type" class="anchor" aria-hidden="true" href="#the-encoding-type"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>The <code>Encoding</code> type</h2>
<p dir="auto">In the examples above, the encoding was specified as a standard string. Though, in order to avoid ambiguities in multiple dispatch and to increase performance via type specialization, the package offers a special <code>Encoding</code> parametric type. Each parameterization of this type represents a character encoding. The <a href="http://docs.julialang.org/en/stable/manual/strings/#man-non-standard-string-literals" rel="nofollow">non-standard string literal</a> <code>enc</code> can be used to create an instance of this type, like so: <code>enc"UTF-16"</code>.</p>
<p dir="auto">Since there is no ambiguity, the <code>encode</code> and <code>decode</code> functions accept either a string or an <code>Encoding</code> object. On the other hand, other functions presented below only support the latter to avoid creating conflicts with other packages extending Julia Base methods.</p>
<p dir="auto">In future versions, the <code>Encoding</code> type will allow getting information about character encodings, and will be used to improve the performance of conversions.</p>

<p dir="auto">The convenience functions presented above are based on the <code>StringEncoder</code> and <code>StringDecoder</code> types, which wrap I/O streams and offer on-the-fly character encoding conversion facilities. They can be used directly if you need to work with encoded text on an already existing I/O stream. This can be illustrated using an <code>IOBuffer</code>:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="julia&gt; b = IOBuffer();

julia&gt; s = StringEncoder(b, &quot;UTF-16&quot;);

julia&gt; write(s, &quot;café&quot;); # Encoding happens automatically here

julia&gt; close(s); # Essential to complete encoding

julia&gt; seek(b, 0); # Move to start of buffer

<p dir="auto">Do not forget to call <code>close</code> on <code>StringEncoder</code> and <code>StringDecoder</code> objects to finish the encoding process. For <code>StringEncoder</code>, this function calls <code>flush</code>, which writes any characters still in the buffer, and possibly some control sequences (for stateful encodings). For both <code>StringEncoder</code> and <code>StringDecoder</code>, <code>close</code> checks that there are no incomplete sequences left in the input stream, and raise an <code>IncompleteSequenceError</code> if that's the case. It will also free iconv resources immediately, instead of waiting for garbage collection.</p>
<p dir="auto">Conversion currently raises an error if an invalid byte sequence is encountered in the input, or if some characters cannot be represented in the target enconding. It is not yet possible to ignore such characters or to replace them with a placeholder.</p>
