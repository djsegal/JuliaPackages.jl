<h2><a id="user-content-purpose" class="anchor" aria-hidden="true" href="#purpose"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Purpose</h2>
<p>One of the difficulties in transitioning to a new programming language is not just learning how to do things in the new language, but the difference in the package ecosystem. <code>RCall</code> helps with both aspects when moving from R to Julia, at least when it comes to basic data manipulation. <code>JellyMe4</code> takes advantage of <code>RCall</code>'s extensibility to provide a way to transfer mixed-effects models fit between R's <code>lme4</code> and Julia's <code>MixedEffectsModels</code>. This means that it is now possible to fit a model in Julia, but then take advantage of existing R packages for examing the model such as <code>car</code> and <code>effects</code>.</p>
<h2><a id="user-content-installation" class="anchor" aria-hidden="true" href="#installation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Installation</h2>
<p><code>JellyMe4</code> is now registered in the Julia package registry and can be installed by</p>
<div class="highlight highlight-source-julia"><pre>julia<span class="pl-k">&gt;</span> <span class="pl-k">using</span> Pkg
julia<span class="pl-k">&gt;</span> Pkg<span class="pl-k">.</span><span class="pl-c1">add</span>(<span class="pl-s"><span class="pl-pds">"</span>JellyMe4<span class="pl-pds">"</span></span>)
</pre></div>
<p>or, in the pkg REPL,</p>
<div class="highlight highlight-source-julia"><pre>(<span class="pl-c1">@v1</span><span class="pl-c1">.4</span>) pkg<span class="pl-k">&gt;</span> add JellyMe4</pre></div>
<p>To get the latest pre-release features, you can install the development version:</p>
<div class="highlight highlight-source-julia"><pre>(<span class="pl-c1">@v1</span><span class="pl-c1">.4</span>) pkg<span class="pl-k">&gt;</span> add JellyMe4<span class="pl-c"><span class="pl-c">#</span>master</span></pre></div>
<p>Generally speaking, the development version should work, but especially until version 1.0, there is no guarantee that there won't be breaking changes compared to the latest release.</p>

<h2><a id="user-content-limitations-and-warnings" class="anchor" aria-hidden="true" href="#limitations-and-warnings"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Limitations and warnings</h2>
<p>This is alpha software. It has some functionality that should work well for common use cases and even a testsuite, but this testsuite depends on two different software environments (R and Julia) and can afoul of all sorts of nasty version interactions. The testuite only tests against a single version of R and the current version of lme4. In other words, even for the parts that do work for me, they may not work for you.</p>
<p>Parts of the API aren't as nice as I would like them to be (especially in the Julia to R direction) and may change if I figure out something nicer.</p>
<p>Only a subset of all the options available in <code>MixedModels</code> and <code>lme4</code> are supported. Unsupported things should break in an obvious way, but here's a list of things that are more commonly used but may break non obviously:</p>
<ul>
<li><del><strong>custom contrast coding</strong>. If you really need this and you know what you're doing, then you can set up your own numeric variables representing appropriate contrasts, as numeric variables survive the transition without difficulty.</del> This should work if you're not doing anything weird with your interaction terms.</li>
<li><strong>advanced models in either language</strong> (e.g., <code>zerocorr!</code> in Julia or <code>||</code> in R, which are not completely synonymous anyway). There's not really a trivial way to deal with this at the moment, sorry.</li>
<li><strong>fancy transformations within model formulae</strong>, especially those that have different names (e.g. <code>scale()</code> in R). If in doubt, pre-transform your data in the dataframe before fitting.</li>
<li><strong>missing data</strong> is handled differently in R and Julia, both in its representation and when it's eliminated. If in doubt, remove missing data before fitting models for consistency.</li>
<li><strong>All the different ways to specify binomial data in R</strong>. Just stick to Bernoulli models with a numeric 0-1 response if you can.</li>
<li><strong>Almost everything related to GLMMs</strong>. This should error when unsupported model types are encountered, and currently only the Julia-to-R direction works.</li>
<li><strong>A number of scratch variables in R prefixed by <code>jellyme4_</code> are created.</strong> We need scratch variables when moving things to R, so we use  identifiers beginning with <code>jellyme4_</code>.</li>
</ul>
<p>Finally, to work its magic, this package hooks into <code>MixedModels</code> and <code>lme4</code> internals. <code>lme4</code>'s internals are quite stable now and this is taken advantage of by several other R packages (<code>ordinal</code>, <code>gamm4</code>, <code>robustlmm</code>). <code>MixedModels</code> are also fairly stable for the core things, but not everything. I am a contributor to <code>MixedModels</code>, so I generally know when something is going to change, but things may still break with changing <code>MixedModels</code> versions, especially for major version changes.</p>
