<h2><a id="user-content-purpose" class="anchor" aria-hidden="true" href="#purpose"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Purpose</h2>
<p>One of the difficulties in transitioning to a new programming language is not just learning how to do things in the new language, but the difference in the package ecosystem. <code>RCall</code> helps with both aspects when moving from R to Julia, at least when it comes to basic data manipulation. <code>JellyMe4</code> takes advantage of <code>RCall</code>'s extensibility to provide a way to transfer mixed-effects models fit between R's <code>lme4</code> and Julia's <code>MixedModels</code>. This means that it is now possible to fit a model in Julia, but then take advantage of existing R packages for examining the model such as <code>car</code> and <code>effects</code>.</p>

<h2><a id="user-content-limitations-and-warnings" class="anchor" aria-hidden="true" href="#limitations-and-warnings"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Limitations and warnings</h2>
<p>This is alpha software. It has some functionality that should work well for common use cases and even a testsuite, but this testsuite depends on two different software environments (R and Julia) and can afoul of all sorts of nasty version interactions. The testuite only tests against a single version of R and the current version of lme4. In other words, even for the parts that do work for me, they may not work for you.</p>
<p>Parts of the API aren't as nice as I would like them to be (especially in the Julia to R direction) and may change if I figure out something nicer.
This package generally follows the Julia community standard <a href="https://semver.org/" rel="nofollow">SemVer</a>.
As this package has not yet hit 1.0, we follow these general conventions:</p>
<ul>
<li>Non-exported functions may change or disappear even with a patch release. Exported functions should otherwise only increase in functionality between patch versions; supporting additional syntax is not considered a breaking change. In other words, errors and exceptions disappearing as more conversions are supported is not breaking. Small improvements will be accompanied by a patch version bump.</li>
<li>Large improvements and potentially breaking changes will be accompanied by a minor version bump.</li>
</ul>
<p>Only a subset of all the options available in <code>MixedModels</code> and <code>lme4</code> are supported. Unsupported things should break in an obvious way, but here's a list of things that are more commonly used but may break non obviously:</p>
<ul>
<li><del><strong>custom contrast coding</strong>. If you really need this and you know what you're doing, then you can set up your own numeric variables representing appropriate contrasts, as numeric variables survive the transition without difficulty.</del> This should work if you're not doing anything weird with your interaction terms.</li>
<li><del><strong>advanced models in either language</strong> (e.g., <code>zerocorr!</code>, <code>fulldummy</code> in Julia or <code>||</code> in R, which are not completely synonymous anyway).</del> This should largely work, but there are a few edge cases, especially when combining these, that might not work. <code>dummy</code> in R will not be translated -- create your indicator variables ahead of time and not as part of the formula.  <code>zerocorr</code> will work with continuous variables and categorical variables with two levels using <code>lme4</code> on the R side. Categorical variables with more than two levels require the R package <a href="https://cran.r-project.org/web/packages/afex" rel="nofollow"><code>afex</code></a> to be installed for <code>zerocorr</code> to be translated correctly. See <a href="#alternative--lmer-">Alternative lmer</a> below. Using <code>||</code> in R and translating to Julia will currently fail with categorical variables.</li>
<li><strong>fancy transformations within model formulae</strong>, especially those that have different names (e.g. <code>scale()</code> in R). If in doubt, pre-transform your data in the dataframe before fitting. A few transformations are supported (e.g. <code>log</code>, <code>exp</code>) and you should receive an error for unsupported transformations, but it's always a good idea to compare the estimates of the original and copied models.</li>
<li><strong>missing data</strong> is handled differently in R and Julia, both in its representation and when it's eliminated. If in doubt, remove missing data before fitting models for consistency.</li>
<li><strong>interactions specified in R with <code>^</code></strong>. This part of the parsing is handled by RCall and winds up getting translated into simple exponentiation instead of "interactions below a certain order".  Consider using the <a href="https://www.rdocumentation.org/packages/MuMIn/versions/1.9.5/topics/Formula%20manipulation" rel="nofollow"><code>simplify.formula</code></a> function from the <a href="https://cran.r-project.org/web/packages/MuMIn/index.html" rel="nofollow"><code>MuMIn</code> package</a> in R to simplify your formula to primitive operations before calling lme4.</li>
<li><strong>other R-specific extensions to Wilkinson-Roger notation</strong>. This includes <code>%in%</code> for nesting, (<code>/</code> for nesting, with reversed order should work because lme4 and MixedModels both handle that specially), <code>I()</code> for literal interpretation of arithmetic, <code>offset()</code>.</li>
<li><strong>rank deficiency in the fixed effects</strong> is handled differently in R and Julia. If in doubt, <a href="https://juliastats.org/MixedModels.jl/dev/rankdeficiency/#Rank-deficiency-in-mixed-effects-models" rel="nofollow">remove extraneous columns/predictors</a> before fitting models for consistency.</li>
<li><strong>GLMMs with dispersion parameters</strong>. This should error when unsupported model types are encountered. This is intentionally not supported at the moment because there are <a href="JuliaStats/MixedModels#291">known problems with these models in MixedModels.jl</a>.</li>
<li><strong>getting binomial GLMMs from R with any of the alternative specifications</strong>, e.g. <code>cbind(successes, failures)</code> or <code>successes/total</code>. You will receive a gentle reminder in the form of an error. Note that even once this is supported, you're subject to the constraints above.</li>
<li><strong>getting GLMMs with <code>quasi</code> families from R</strong>. The corresponding distributions aren't in Distributions.jl and are only "quasi" in the GLM/GLMM framework.</li>
<li><strong>a number of scratch variables in R prefixed by <code>jellyme4_</code> are created.</strong> We need scratch variables when moving things to R, so we use  identifiers beginning with <code>jellyme4_</code>.</li>
</ul>
<p>Finally, to work its magic, this package hooks into <code>MixedModels</code> and <code>lme4</code> internals. <code>lme4</code>'s internals are quite stable now and this is taken advantage of by several other R packages (<code>ordinal</code>, <code>gamm4</code>, <code>robustlmm</code>). <code>MixedModels</code> are also fairly stable for the core things, but not everything. I am a contributor to <code>MixedModels</code>, so I generally know when something is going to change, but things may still break with changing <code>MixedModels</code> versions, especially for major version changes.</p>

<p>By default, JellyMe4 uses <code>[g]lmer</code> from <code>lme4</code>, as this is the closest equivalent to <code>MixedModels</code> (and was in no small part written by the same individual). It is, however, possible to set a different <code>lmer</code> implementation. Notably, the <code>afex</code> and <code>lmerTest</code> packages extend <code>lmer</code> in ways that are often appealing. If you set the Julia environment variable <code>ENV["LMER"]</code> before loading JellyMe4, then JellyMe4 will use the alternative specification. For example:</p>

<div class="highlight highlight-source-r position-relative" data-snippet-clipboard-copy-content="R&gt; mod = as(mod, &quot;merModLmerTest&quot;)

If <code>afex</code> is installed and you attempt to convert a <code>zerocorr</code> model with a categorical variable with many levels, then JellyMe4 will swap itself to use <code>afex::lmer_alt</code> for the correct handling of these. If <code>afex</code> is not installed, then JellyMe4 can still convert two-level factors and continuous variables using the machinery built into <code>lme4</code>, but will error for multi-level factors. Once JellyMe4 has swapped to using <code>afex::lmer_alt</code>, it will continue doing so for the remainder of the session, but it will only swap when it first encounters a multi-level factor.</p>
<h3><a id="user-content-r-to-julia" class="anchor" aria-hidden="true" href="#r-to-julia"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>R to Julia</h3>
<p>Models fit with the R package <a href="https://cran.r-project.org/web/packages/lmerTest/" rel="nofollow"><code>lmerTest</code></a> should work without difficulty as <code>lmerTest</code> does not modify the fitting process nor the relevant internal structure of the model of <code>lme4</code>.
Note that the functionality related to denominator degrees of freedom (e.g. Satterthwaite or Kenward-Roger approximations) are not currently implemented in <code>MixedModels</code> and thus this functionality will be lost.</p>
<p>Models fit with the R package <a href="https://cran.r-project.org/web/packages/afex/" rel="nofollow"><code>afex</code></a> should also largely be compatible, as <code>afex</code> works by providing a more convenient interface to <code>lme4</code>.
Note however that if any of <code>afex</code>'s behind the scenes rewrite-rules invoke features not yet supported, then the resulting models will not work with JellyMe4.
In particular, <code>afex</code> directly modifies the model matrix to fix the limitation in the <code>||</code> syntax and this will almost definitely not work in JellyMe4.
This could probably be made to work, but it seems needlessly complicated given that the assumption is that most models are easier to fit with <code>MixedModels</code> than <code>lme4</code> and that the package is primarily for moving models from Julia to R.
In any case, please check your model summary on both sides to make sure they line up!</p>
