<p dir="auto">This package is a lot like Pkg from the Julia standard library, except that it is for
managing Conda packages.</p>
<ul dir="auto">
<li>Conda dependencies are defined in <code>CondaPkg.toml</code>, which is analogous to <code>Project.toml</code>.</li>
<li>CondaPkg will install these dependencies into a Conda environment specific to the current
Julia project. Hence dependencies are isolated from other projects or environments.</li>
<li>Functions like <code>add</code>, <code>rm</code>, <code>status</code> exist to edit the dependencies programmatically.</li>
<li>Or you can do <code>pkg&gt; conda add some_package</code> to edit the dependencies from the Pkg REPL.</li>
</ul>

<p dir="auto">The simplest way to specify Conda dependencies is through the Pkg REPL, just like for Julia
dependencies. For example:</p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="julia&gt; using CondaPkg

<p dir="auto">For more information do <code>?</code> or <code>?conda</code> from the Pkg REPL.</p>
<p dir="auto"><strong>Note:</strong> We recommend against adding Pip packages unless necessary - if there is a
corresponding Conda package then use that. Pip does not handle version conflicts
gracefully, so it is possible to get incompatible versions.</p>

<p dir="auto">These functions are intended to be used interactively when the Pkg REPL is not available
(e.g. if you are in a notebook):</p>
<ul dir="auto">
<li><code>status()</code> shows the Conda dependencies of the current project.</li>
<li><code>add(pkg; version="", channel="")</code> adds/replaces a dependency.</li>
<li><code>rm(pkg)</code> removes a dependency.</li>
<li><code>add_channel(channel)</code> adds a channel.</li>
<li><code>rm_channel(channel)</code> removes a channel.</li>
<li><code>add_pip(pkg; version="")</code> adds/replaces a pip dependency.</li>
<li><code>rm_pip(pkg)</code> removes a pip dependency.</li>
</ul>

<ul dir="auto">
<li><code>envdir()</code> returns the root directory of the Conda environment.</li>
<li><code>withenv(f)</code> returns <code>f()</code> evaluated in the Conda environment.</li>
<li><code>which(progname)</code> find the program in the Conda environment.</li>
<li><code>resolve(; force=false)</code> resolves dependencies. You don't normally need to call this
because the other API functions will automatically resolve first. Pass <code>force=true</code> if
you change a <code>CondaPkg.toml</code> file mid-session.</li>
<li><code>update()</code> update the conda and pip installed packages.</li>
<li><code>gc()</code> removes unused caches to save disk space.</li>
</ul>

<p dir="auto">You can specify the channel to install a particular package from, such as with
<code>pkg&gt; conda add some-channel::some-package</code>.</p>
<p dir="auto">You can also specify a top-level list of channels, from which all other packages are
installed, such as with <code>pkg&gt; conda channel_add some-channel</code>.</p>
<p dir="auto">By default, packages are installed from the <code>conda-forge</code> channel.</p>

<p dir="auto">Direct references such as <code>foo@http://example.com/foo.zip</code> are allowed. As a special case
if the URL starts with <code>.</code> then it is interpreted as a path relative to the directory
containing the <code>CondaPkg.toml</code> file.</p>
<p dir="auto">Additionally the binary mode specifies whether to only use binary distributions ("only") or
to never use them ("no").</p>
<h3 dir="auto"><a id="user-content-backends" class="anchor" aria-hidden="true" href="#backends"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Backends</h3>
<p dir="auto">This package has a number of different "backends" which control exactly which implementation
of Conda is used to manage the Conda environments. You can explicitly select a backend
by setting the environment variable <code>JULIA_CONDAPKG_BACKEND</code> to one of the following values:</p>
<ul dir="auto">
<li><code>MicroMamba</code>: Uses MicroMamba from the package
<a href="https://github.com/cjdoris/MicroMamba.jl">MicroMamba.jl</a>.</li>
<li><code>System</code>: Use a pre-installed Conda. If <code>JULIA_CONDAPKG_EXE</code> is set, that is used.
Otherwise we look for <code>conda</code>, <code>mamba</code> or <code>micromamba</code> in your <code>PATH</code>.</li>
<li><code>Current</code>: Use the currently activated Conda environment instead of creating a new one.
This backend will only ever install packages, never uninstall. The Conda executable used
is the same as for the System backend. Similar to the default behaviour of
<a href="https://github.com/JuliaPy/Conda.jl">Conda.jl</a>.</li>
<li><code>Null</code>: Don't use CondaPkg to manage dependencies. Use this if you are in a pre-existing
Conda environment that already satisfies the dependencies of your project. It is up to you
to ensure any required packages are installed.</li>
</ul>
<p dir="auto">The default backend is an implementation detail, but is currently <code>MicroMamba</code>.</p>
<p dir="auto">If you set <code>JULIA_CONDAPKG_EXE</code> but not <code>JULIA_CONDAPKG_BACKEND</code> then the <code>System</code> backend
is used.</p>

<p dir="auto">You may activate "offline mode" by setting the environment variable
<code>JULIA_CONDAPKG_OFFLINE=yes</code>. This will prevent CondaPkg from attempting to download or
install new packages. In this case, it is up to you to ensure that any required packages are
already available (such as by having previously called <code>CondaPkg.resolve()</code>).</p>
<h3 dir="auto"><a id="user-content-conda-environment-path" class="anchor" aria-hidden="true" href="#conda-environment-path"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Conda environment path</h3>
<p dir="auto">By default, CondaPkg installs Conda packages into the current project, so that different
projects can have different dependencies. If you wish to centralize the Conda environment,
you can set one of these environment variables:</p>
<ul dir="auto">
<li><code>JULIA_CONDAPKG_ENV=@&lt;name&gt;</code> for a named shared environment, stored in <code>~/.julia/conda_environments/&lt;name&gt;</code>.</li>
<li><code>JULIA_CONDAPKG_ENV=&lt;some absolute path&gt;</code> for a shared environment at the given path.</li>
<li><code>JULIA_BACKEND=Current</code> to use the currently activated Conda environment.</li>
</ul>
<p dir="auto"><strong>Warning:</strong> If you do this, the versions specified in a per-julia-version <code>CondaPkg.toml</code>
can become un-synchronized with the packages installed in the shared Conda environment.
In this case, you will have to re-resolve the dependencies using <code>resolve(force=true)</code>.
This restriction might be alleviated in future CondaPkg versions.</p>

<p dir="auto">You can control the verbosity of any <code>conda</code> or <code>pip</code> commands executed by setting the
