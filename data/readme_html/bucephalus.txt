<p dir="auto">Bucephalus.jl is a package that aims to automate the solution of heterogeneous-agent economic models using model-based reinforcement learning.</p>
<p dir="auto">This is very much a work in progress! Do not expect your model to converge correctly at this stage.</p>

<h2 dir="auto"><a id="user-content-use-cases" class="anchor" aria-hidden="true" href="#use-cases"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Use Cases</h2>
<p dir="auto">This package is designed to solve heterogenous-agent macroeconomic models, with possible market or spatial segmentation.</p>
<p dir="auto">In these models, many agents interact in an environment in which each agent has an individual state (e.g. wealth), and the environment also may have some state variables common to all agents (e.g. total factor productivity). Agents make choices each period to maximize the net present value of present and future period utilities (called the "return" in RL), given their individual state. Stochasticity takes the form of random variables drawn each period which are taken as inputs to the transition function.</p>
<p dir="auto">One key detail is that, in this framework, agents are assumed to be either "representative" or "small", in that they do not internalize their individual effects on aggregate quantities (see roadmap).</p>
<h2 dir="auto"><a id="user-content-framework" class="anchor" aria-hidden="true" href="#framework"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Framework</h2>
<p dir="auto">Bucephalus has two components</p>
<ol dir="auto">
<li>A <code>Model</code> object, which represents a description of a heterogenous-agent economic model.</li>
<li>A learning algorithm which, given a <code>Model</code> object, attempts to solve for the policy function, value function, and steady-state distribution of that model.</li>
</ol>
<p dir="auto">In future, we aim to include several solvers in this package, as well as an easy and accessible interface for defining your own solvers and learning algorithms.</p>
<h2 dir="auto"><a id="user-content-the-model-object" class="anchor" aria-hidden="true" href="#the-model-object"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>The <code>Model</code> Object</h2>
<p dir="auto">The <code>Model</code> object is a description of the model being solved, and is described using Julia macros. (There is also an under-the-hood syntax for defining models which does not use macros.) Examples of model description can be found in the <code>examples</code> folder. In each example, the <code>Model</code> object is defined, and then the algorithm is called using the <code>train</code> function.</p>
<h3 dir="auto"><a id="user-content-syntax" class="anchor" aria-hidden="true" href="#syntax"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Syntax</h3>
<p dir="auto">The model is defined by defining agent types and possibly segmentations (such as locations), and then defining variables.</p>
<p dir="auto">There are nine "stages" of variables, in terms of when they are computed in the course of a single iteration of the simulation:
0. Parameters. These are set to constants at the beginning of the model definition and never changed.</p>
<ol dir="auto">
<li>State variables. These are variables that define the state of the economy at the beginning of a period -- the output of the transition function.</li>
<li>State-dependent variables. These are variables that depend only on state variables (and parameters), but are not given as the output of the transition function.</li>
<li>Equilibrium variables. Conventionally thought of as "prices," these are variables that are defined in terms of an equilibrium condition (e.g. "market clearing conditions").</li>
<li>Equilibrium-dependent variables. Variables that depend on only equilibrium (and earlier-stage) variables.</li>
<li>Choice variables. Variables chosen by agents -- the output of the policy function.</li>
<li>Choice-dependent variables. These include utility, or reward.</li>
<li>Shock varaibles. Variables with a specified random distribution. These can be used as inputs to the transition function to model stochasticity.</li>
<li>Next-state variables. Strictly speaking, these are the same as the state variables, but must be assigned a rule for computing them: the transition functions.</li>
</ol>
<p dir="auto">In general, each variable has the following components:</p>
<ul dir="auto">
<li>A stage, determining when it is computed in the simulation of the model.</li>
<li>A host: either an agent type, a segmentation, or the model itself. This is the level of the model that the variable is specific to. For example, a variable with the model as a host is common to all agents, e.g. total factor productivity. A variable with an agent type as a host will be specific to each agent, e.g. wealth or labor endowment. A variable with a segmentation as a host will have a different value for each segmentation. For example, a variable "location-specific productivity" might have a different value for each location.</li>
<li>A function or distribution defining it. This determines how the variable is computed or drawn.</li>
</ul>
<p dir="auto">The general syntax looks like this:</p>
<p dir="auto"><code>@vartype host.var(inputvar1, inputvar2) = f_var(inputvar1,inputvar2)</code></p>
<p dir="auto">When an input variable <code>inputvar</code> has <code>host</code> as its host, the function will implicitly take only the individual value of <code>inputvar</code> as its input. To take the whole vector of values as input, use <code>Ref(inputvar)</code>:</p>
<p dir="auto"><code>@statedep hh.meanpeerwealth(Ref(wealth)) = mean(wealth)</code></p>
<p dir="auto">This will differ somewhat by stage. For example, choice variables will not have an associated function or distribution, because they are implicitly taken from the policy function.</p>
<p dir="auto">A macro defining a variable will create a Julia variable in the macro's calling context with the same name as the model variable. For example,</p>
<p dir="auto"><code>@statevar hh.l = LogNormal()</code></p>
<p dir="auto">is unrolled into</p>
<p dir="auto"><code>l = Bucephalus.addStateVariable!(hh, Float64, :l; init=LogNormal())</code></p>
<p dir="auto">When segmentations are present, an additional variable type, <code>Match</code> is provided, along with two additional operators on variables: <code>By</code> and <code>ByMembers</code>.</p>
<p dir="auto">A <code>Match</code> defines a relationship between an agent and a segmentation. Take "location" as an example. An agent does not simply "belong" to a single location, but may have relations such as <code>residence</code> and <code>workplace</code> that take a location as a value. These are declared using the <code>@matchvar</code> macro:</p>
<p dir="auto"><code>@matchvar hh.residence::Match{:loc} = rand(1:2, n_hh)</code></p>
<p dir="auto">This defines a variable <code>residence</code> as a state variable of agent type <code>hh</code>, which takes a <code>loc</code> as a value. The resulting variable <code>residence</code> will have a property <code>host</code> (in this case, <code>hh</code>) and <code>partnerhost</code> (in this case, <code>loc</code>).  I use the term "match" in the hopes of one day supporting relations between agents, such as "spouse" or "landlord."</p>
<p dir="auto">The construct <code>By(var, matchvar)</code> is a construct that acts like a variable. The host of the <code>By</code> construct must be the host of <code>matchvar</code>, and <code>var</code> must be a variable on the partnerhost of <code>matchvar</code>. For example, <code>By(rent, residence)</code> refers to the rent in an agent's residence location. Rent is a location-specific value, and so <code>By(rent, residence)</code> is computed by finding the residence location of an agent, and returning the rent at that location.</p>
<p dir="auto">The construct <code>ByMembers(var, matchvar)</code> is similarly a construct that acts like a variable. The host of the <code>ByMembers</code> construct must be the partnerhost of <code>matchvar</code>, and <code>var</code> must be a variable whose host is the host of the <code>matchvar</code>. For example, <code>ByMembers(wealth, residence)</code> refers to the vector of the <code>wealth</code> values of all the residents of a certain location.</p>
<p dir="auto">These descriptions are quite abstract. Please check out the examples to see them in action!</p>
<h2 dir="auto"><a id="user-content-the-learning-algorithm" class="anchor" aria-hidden="true" href="#the-learning-algorithm"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>The Learning Algorithm</h2>
<p dir="auto">The learning algorithm currently implemented is a model-based actor-critic algorithm, some details of which are tailored to the our use case.</p>
<p dir="auto">One non-standard feature of this setting is that we can differentiate through the transition function. This allows us to use automatic differentiation to directly compute the derivative of the future state with respect to the (continuous) action. Thus, we update the policy function parameters by directly computing the derivative of the state-value function with respect to the policy function parameters.</p>
<p dir="auto">A representation in pseudocode of the algorithm is given in algorithm_pseudocode.pdf</p>
<p dir="auto">Note: A performance improvement can be obtained by specifying in the model that equilibrium variables are not used as inputs to actions. E.g. agents choose expenditure instead of quantities, so that their choice will be valid regardless of prices.</p>
<h4 dir="auto"><a id="user-content-limitations" class="anchor" aria-hidden="true" href="#limitations"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Limitations</h4>
<p dir="auto">This algorithm has many limitations:</p>
<ol dir="auto">
<li>The algorithm may not perform well in the presence of strong non-convexities in the value function. Because the policy function network parameters are updated using gradient descent, it is quite possible for it to become stuck at local minima.</li>
<li>When the steady state of the model is non-ergodic, the algorithm may fail to cover the state space well. That is, because the steady state is traversed simply by simulating the model forward, the algorithm will fail to cover the steady state space when the steady state exhibits path-dependence. In future, I will try to remedy this by providing the option to begin the simulation at various starting points and simulating in parallel, or by periodically restarting the simulation from a new random initial state.</li>
<li>Currently, the inputs to the policy function must be specified manually, and performance is poor when the entire state space is used. This will be unnecessary after implementing the "generalized moments" of <a href="https://arxiv.org/pdf/2112.14377.pdf" rel="nofollow">DeepHAM</a>.</li>
</ol>

<ol dir="auto">
<li>Replace large parts of the hand-coded backend with existing packages. This should be a great improvement to the package, in addition to significantly reducing the size of its source code.</li>
<li>Implement the generalized moments of <a href="https://arxiv.org/pdf/2112.14377.pdf" rel="nofollow">DeepHAM</a>. In the case of segmentations, implement nested generalized moments, where one set of generalized moments is computed for each segmentation, and another set is computed using the vector of generalized moments for each segmentation as its input.</li>
<li>Implement other solvers and learning algorithms, including the brute-force solver.</li>
<li>Find robust strategies to draw starting states for learning episodes. Allow the user to configure the method used.</li>
<li>Implement methods to test the optimality of policy networks, such as Euler equation errors.</li>
<li>Allow agents to internalize their effect on aggregate quantities. This is necessary, e.g., for models of monopolistic competition.</li>
</ol>
