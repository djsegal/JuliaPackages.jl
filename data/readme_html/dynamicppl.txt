<p dir="auto"><em>A domain-specific language and backend for probabilistic programming, used by <a href="https://github.com/TuringLang/Turing.jl">Turing.jl</a>.</em></p>
<p dir="auto">DynamicPPL is the part of Turing.jl that deals with defining, running, and manipulating models. DynamicPPL provides:</p>
<ul dir="auto">
<li>General-purpose probabilistic programming with an intuitive syntax.</li>
<li>The <code>@model</code> syntax and macro for easily specifying probabilistic generative models.</li>
<li>A tracing data-structure for tracking random variables in dynamic probabilistic models.</li>
<li>A rich contextual dispatch system allowing for tailored behaviour during model execution.</li>
<li>A user-friendly syntax for probabilistic queries.</li>
</ul>
<p dir="auto">Information on how to use the DynamicPPL frontend to build Bayesian models can be found on the <a href="https://turing.ml/" rel="nofollow">Turing website</a>. Tutorials explaining how to use the backend can be found <a href="https://turinglang.github.io/DynamicPPL.jl/stable/" rel="nofollow">alongside the documentation</a>. More information can be found in our paper <a href="https://arxiv.org/pdf/2002.02702.pdf" rel="nofollow">DynamicPPL: Stan-like Speed for Dynamic Probabilistic Models</a>.</p>

<p dir="auto">This project follows the <a href="https://colprac.sciml.ai/" rel="nofollow"><img src="https://camo.githubusercontent.com/2496bdc13cbc9c458dfa19a108b1f333353f62917355a4cdee582bbdf8be43cf/68747470733a2f2f696d672e736869656c64732e696f2f62616467652f436f6c507261632d436f6e7472696275746f722532377325323047756964652d626c756576696f6c6574" alt="ColPrac: Contributor's Guide on Collaborative Practices for Community Packages" data-canonical-src="https://img.shields.io/badge/ColPrac-Contributor%27s%20Guide-blueviolet" style="max-width: 100%;"></a>, apart from the following slight variation:</p>
<ul dir="auto">
<li>The master branch contains the most recent release at any point in time. All non-breaking changes (bug fixes etc.) are merged directly into master and a new patch version is released immediately.</li>
<li>A separate dev branch contains all breaking changes, and is merged into master when a minor version release happens.</li>
</ul>
<p dir="auto">For instance, suppose we are currently on version 0.13.5.</p>
<ul dir="auto">
<li>If someone produces a bug fix, it is merged directly into master and bumps the version to 0.13.6. This change is also merged into dev so that it remains up-to-date with master.</li>
<li>If someone is working on a new feature that is not breaking (performance-related, fancy new syntax that is backwards-compatible etc.), the same happens.</li>
<li>New breaking changes are merged into dev until a release is ready to go, at which point dev is merged into master and version 0.14 is released.</li>
</ul>

<p dir="auto">This project uses a <a href="https://docs.github.com/en/repositories/configuring-branches-and-merges-in-your-repository/configuring-pull-request-merges/managing-a-merge-queue">merge queue</a> for merging PRs.
In this way, merge skew / semantic merge conflicts are prevented by testing the exact integration of pull requests before merging them.</p>
<p dir="auto">When a PR is good enough for merging and has been approved by at least one reviewer, instead of merging immediately, it is added to the merge queue.
If the CI tests pass, including downstream tests of Turing, the PR is merged into the main branch.</p>
