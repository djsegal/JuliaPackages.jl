<p>CompositeTypes.jl defines an interface for types that consist of multiple components.</p>
<h2><a id="user-content-interface" class="anchor" aria-hidden="true" href="#interface"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Interface</h2>
<p>The package defines:</p>
<ul>
<li><code>iscomposite(x)</code>: true if <code>x</code> is a composite object</li>
<li><code>components(x)</code>: returns the components of <code>x</code></li>
<li><code>component(x, I...)</code>: returns <code>components(x)[I...]</code></li>
<li><code>ncomponents(x)</code>: the number of components of <code>x</code></li>
<li><code>setcomponent!(x, v, I...)</code>: (if applicable) set a component to a given value</li>
</ul>
<p>A type can declare to be composite simply by implementing <code>components(x)</code>, and
returning something with non-zero <code>length</code>.</p>

<p>Composite types can opt-in to a structured multi-line representation by
defining a display stencil and specializing <code>show</code>. An example, again using
the <code>DomainSets.jl</code> package:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="julia&gt; using DomainSets

<p>The display routines recursively evaluate display stencils of all objects
appearing in the stencil of an object, up to a maximum recursion depth. An
attempt is made to define symbols, such that the output remains somewhat
readable.</p>
<p>Both the <code>ProductDomain</code> and <code>UnionDomain</code> types are composite types. They
combine their components using a <code>combinationsymbol</code>, in this case <code>∪</code> and <code>×</code>.
The output above is achieved with the definitions, for <code>ProductDomain</code>:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="using CompositeTypes.Display
Display.combinationsymbol(d::ProductDomain) = Display.Symbol('×')
Display.displaystencil(d::ProductDomain) = composite_displaystencil(d)

show(io::IO, mime::MIME&quot;text/plain&quot;, d::ProductDomain) = composite_show(io, mime, d)
show(io::IO, d::ProductDomain) = composite_show_compact(io, d)
"><pre><span class="pl-k">using</span> CompositeTypes<span class="pl-k">.</span>Display

<p>The invocation of <code>show</code> with the <code>mime</code> argument indicates that a multi-line
representation can be shown. The shorter <code>show(io, x)</code> function typically expects
a one-line representation. Types can choose to implement either function. Typically
the longer version is the most useful, while for the compact version it may be
safer to rely on Julia's default.</p>
<p>A type can define a custom display stencil, which is a vector in which each string
or character ends up being displayed, and each object is replaced by its own
display stencil or by a compact string representation. For example:</p>
<div class="highlight highlight-source-julia position-relative" data-snippet-clipboard-copy-content="Display.displaystencil(object::LinearMap) = [object.A, &quot; * x + &quot;, object.b]
"><pre>Display<span class="pl-k">.</span><span class="pl-en">displaystencil</span>(object<span class="pl-k">::</span><span class="pl-c1">LinearMap</span>) <span class="pl-k">=</span> [object<span class="pl-k">.</span>A, <span class="pl-s"><span class="pl-pds">"</span> * x + <span class="pl-pds">"</span></span>, object<span class="pl-k">.</span>b]</pre></div>
