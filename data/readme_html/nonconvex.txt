<p dir="auto"><code>Nonconvex.jl</code> is an umbrella package over implementations and wrappers of a number of nonconvex constrained optimization algorithms and packages making use of automatic differentiation. Zero, first and second order methods are available. Nonlinear equality and inequality constraints as well as integer and nonlinear semidefinite constraints are supported. A detailed description of all the algorithms and features available in <code>Nonconvex</code> can be found in the <a href="https://JuliaNonconvex.github.io/Nonconvex.jl/stable" rel="nofollow">documentation</a>.</p>

<p dir="auto">The <code>JuliaNonconvex</code> organization hosts a number of packages which are available for use in <code>Nonconvex.jl</code>. The correct package is loaded using the <code>Nonconvex.@load</code> macro with the algorithm or package name. See the <a href="https://JuliaNonconvex.github.io/Nonconvex.jl/stable" rel="nofollow">documentation</a> for more details. The following is a summary of all the packages in the <code>JuliaNonconvex</code> organization.</p>

<h2 dir="auto"><a id="user-content-design-philosophy" class="anchor" aria-hidden="true" href="#design-philosophy"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Design philosophy</h2>
<p dir="auto"><code>Nonconvex.jl</code> is a Julia package that implements and wraps a number of constrained nonlinear and mixed integer nonlinear programming solvers. There are 3 focus points of <code>Nonconvex.jl</code> compared to similar packages such as <code>JuMP.jl</code> and <code>NLPModels.jl</code>:</p>
<ol dir="auto">
<li>Emphasis on a function-based API. Objectives and constraints are normal Julia functions.</li>
<li>The ability to nest algorithms to create more complicated algorithms.</li>
<li>The ability to automatically handle structs and different container types in the decision variables by automatically vectorizing and un-vectorizing them in an AD compatible way.</li>
</ol>

<p dir="auto">The following is an explanation of all the columns in the table:</p>
<ul dir="auto">
<li>Algorithm name. This is the name of the algorithm and/or its acronym. Some algorithms have multiple variants implemented in their respective packages. When that's the case, the whole family of algorithms is mentioned only once.</li>
<li>Is meta-algorithm? Some algorithms are meta-algorithms that call a sub-algorithm to do the optimization after transforming the problem. In this case, a lot of the properties of the meta-algorithm are inherited from the sub-algorithm. So if the sub-algorithm requires gradients or Hessians of functions in the model, the meta-algorithm will also require gradients and Hessians of functions in the model. Fields where the property of the meta-algorithm is inherited from the sub-solver are indicated using the "Depends on sub-solver" entry. Some algorithms in <code>NLopt</code> have a "Limited" meta-algorithm status because they can only be used to wrap algorithms from <code>NLopt</code>.</li>
<li>Algorithm package. This is the Julia package that either implements the algorithm or calls it from another programming language. <code>Nonconvex</code> wraps all these packages using a consistent API while allowing each algorithm to be customized where possible and have its own set of options.</li>
<li>Order. This is the order of the algorithm. Zero-order algorithms only require the evaluation of the objective and constraint functions, they don't require any gradients or Hessians of objective and constraint functions. First-order algorithms require both the value and gradients of objective and/or constraint functions. Second-order algorithms require the value, gradients and Hessians of objective and/or constraint functions.</li>
<li>Finite bounds. This is true if the algorithm supports finite lower and upper bound constraints on the decision variables. One special case is the <code>TOBS</code> algorithm which only supports binary decision variables so an entry of "Binary" is used instead of true/false.</li>
<li>Infinite bounds. This is true if the algorithm supports unbounded decision variables either from below, above or both.</li>
<li>Inequality constraints. This is true if the algorithm supports nonlinear inequality constraints.</li>
<li>Equality constraints. This is true if the algorithm supports nonlinear equality constraints. Algorithms that only support linear equality constraints are given an entry of "Limited".</li>
<li>Semidefinite constraints. This is true if the algorithm supports nonlinear semidefinite constraints.</li>
<li>Integer variables. This is true if the algorithm supports integer/discrete/binary decision variables, not just continuous. One special case is the <code>TOBS</code> algorithm which only supports binary decision variables so an entry of "Binary" is used instead of true/false.</li>
</ul>

<p dir="auto"><strong>A beginner?</strong> The easiest way to contribute is to read the documentation, test the package and report issues.</p>
<p dir="auto"><strong>An impulsive tester?</strong> Improving the test coverage of any package is another great way to contribute to the <code>JuliaNonconvex</code> org. Check the coverage report of any of the packages above by clicking the coverage badge. Find the red lines in the report and figure out tests that would cover these lines of code.</p>
<p dir="auto"><strong>An algorithm head?</strong> There are plenty of optimization algorithms that can be implemented and interfaced in <code>Nonconvex.jl</code>. You could be developing the next big nonconvex semidefinite programming algorithm right now! Or the next constraint handling method for evolutionary algorithms!</p>
<p dir="auto"><strong>A hacker?</strong> Let's figure out how to wrap some optimization package in Julia in the unique, simple and nimble <code>Nonconvex.jl</code> style.</p>
<p dir="auto"><strong>A software designer?</strong> Let's talk about design decisions and how to improve the modularity of the ecosystem.</p>
