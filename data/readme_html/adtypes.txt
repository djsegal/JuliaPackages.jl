<p dir="auto"><a href="https://github.com/Vaibhavdixit02/ADTypes.jl/actions/workflows/CI.yml?query=branch%3Amain"><img src="https://github.com/Vaibhavdixit02/ADTypes.jl/actions/workflows/CI.yml/badge.svg?branch=main" alt="Build Status" style="max-width: 100%;"></a></p>
<p dir="auto">ADTypes.jl is a common system for implementing multi-valued logic for choosing which
automatic differentiation library to use.</p>

<p dir="auto">The current standard is to have a keyword argument with <code>autodiff = true</code> or <code>autodiff = false</code>.
However, boolean logic is not sufficient to be able to do all of the choices that are
required. For example for forward-mode AD there is ForwardDiff vs Enzyme, and thus <code>true</code>
is ambgiuous. As libraries begin to support more and more of these autodiff mechanisms
for their trade-offs, every library will have their own version of <code>ChooseForwardDiff()</code>
etc. which would be a mess. Hence ADTypes.jl giving a single set of shared types for this.
<code>ADTypes.AutoForwardDiff()</code> is the way to do it.</p>

Or EnumX?</h2>
<p dir="auto">If they were enums then they would not be dispatch type-level information. This is needed
for the internals of many of the packages to be type-stable because the choice of <code>config</code>
type is different per package, i.e. what needs to be cached in order to use everything in
a non-allocating manner.</p>
