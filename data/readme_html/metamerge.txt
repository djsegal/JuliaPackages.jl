<h3><a id="user-content-motivation--example-usage" class="anchor" aria-hidden="true" href="#motivation--example-usage"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Motivation &amp; example usage</h3>
<p>Suppose we create a function <code>f</code> in <code>Main</code>:</p>
<pre><code>julia&gt; f() = nothing
f (generic function with 1 method)
</code></pre>
<p>Suppose also that we also intend to use the following modules <code>A</code> and <code>B</code>:</p>
<div class="highlight highlight-source-julia"><pre><span class="pl-k">module</span> A

<span class="pl-k">export</span> f
immutable Foo <span class="pl-k">end</span>
<span class="pl-en">f</span>(<span class="pl-k">::</span><span class="pl-c1">Foo</span>) <span class="pl-k">=</span> <span class="pl-c1">print</span>(<span class="pl-s"><span class="pl-pds">"</span>This is Foo.<span class="pl-pds">"</span></span>)
<span class="pl-en">f</span>(x<span class="pl-k">::</span><span class="pl-c1">Int64</span>) <span class="pl-k">=</span> x

<span class="pl-k">end</span>

<span class="pl-k">module</span> B

<span class="pl-k">export</span> f
immutable Bar <span class="pl-k">end</span>
<span class="pl-en">f</span>(<span class="pl-k">::</span><span class="pl-c1">Bar</span>) <span class="pl-k">=</span> <span class="pl-c1">print</span>(<span class="pl-s"><span class="pl-pds">"</span>This is Bar.<span class="pl-pds">"</span></span>)
<span class="pl-en">f</span>(x<span class="pl-k">::</span><span class="pl-c1">Int64</span>) <span class="pl-k">=</span> <span class="pl-c1">2</span>x

<span class="pl-k">end</span></pre></div>
<p>As of Julia 0.3.7, unqualified use of a name common to both modules -- say, the name '<code>f</code>' -- will elicit behavior that depends on the order in which we declare to be <code>using</code> the modules:</p>
<pre><code>julia&gt; using A, B
Warning: using A.f in module Main conflicts with an existing identifier.
Warning: using B.f in module Main conflicts with an existing identifier.

julia&gt; methods(f)
# 1 method for generic function "f":
f() at none:1

julia&gt; f(A.Foo())
ERROR: `f` has no method matching f(::Foo)

julia&gt; A.f(A.Foo())
This is Foo.
</code></pre>
<p>But suppose we want unqualified use of '<code>f</code>' to refer to the correct object --- either <code>f</code>, <code>A.f</code> or <code>B.f</code> --- depending on the signature of the argument on which <code>f</code> is called. The present "package" offers this functionality through the <code>fmerge!()</code> function, which "merges" the methods of <code>A.f</code> and <code>B.f</code> into our original function <code>f</code> as defined in <code>Main</code>. (At its core, this is just extending the <code>f</code> defined in <code>Main</code>.) This allows unqualified use of the name '<code>f</code>' to dispatch on signatures for which methods are defined in other modules:</p>
<pre><code>julia&gt; fmerge!(f, (A,f), (B,f))

julia&gt; methods(f)
# 3 methods for generic function "f":
f() at none:1
f(x1_A::Foo)
f(x1_B::Bar)

julia&gt; f(A.Foo())
This is Foo.
julia&gt; f(B.Bar())
This is Bar.
</code></pre>
<p>For merged methods with at least one argument, the name of the module from which the method originates is appended to the first argument in the method definition, as can be seen above. This can help one keep track of which methods come from which modules. However, this machinery only keeps track of the most recent module from which the method originates. If a method has been merged multiple times through multiple modules, its ultimate origin will be obscured.</p>
<p>Note that no method for the signature <code>(x::Int64,)</code> was merged since both <code>A.f</code> and <code>B.f</code> have methods for this signature. To choose one to merge, use the optional <code>priority</code> keyword argument, which takes an array of <code>(::Module, ::Function)</code> tuples in the order of priority rank:</p>
<pre><code>julia&gt; fmerge!(f, (A,f), (B,f), priority=[(A,f)])

julia&gt; methods(f)
# 4 methods for generic function "f":
f() at none:1
f(x1_A::Foo)
f(x1_B::Bar)
f(x1_A::Int64)

julia&gt; f(3)
3
</code></pre>
<p>If, for a given signature, a method exists in both <code>Module1.f</code> and <code>Module2.f</code>, then the method from whichever of <code>(Module1, f)</code>, <code>(Module2, f)</code> with the greater rank (so <em>lower</em> numerical rank, e.g. 1 is greatest) will be merged. <code>(::Module, ::Function)</code> arguments passed to <code>fmerge!()</code> but omitted from <code>priority</code> are by default given the lowest possible rank. If <code>(Module1, f)</code>, <code>(Module2, f)</code> have the same rank (which will only occur if they are not specified in <code>priority</code>) then neither method will be merged. This means that if one omits the <code>priority</code> argument, then only those methods whose signatures unambiguously specify precisely one of the <code>(::Module, ::Function)</code> arguments passed to <code>fmerge!()</code> will be merged.</p>
<p>WARNING: As of yet I haven't figured out how to use reflection to distinguish between otherwise identical signatures with user-defined <em>types</em> of the same name. Thus if module <code>B</code> above also defined a <code>Foo</code> type and defined a method for <code>f(::Foo)</code>, these two methods would be seen to conflict by <code>fmerge!()</code>.</p>
<p>One can call <code>fmerge!()</code> in modules other than <code>Main</code>.</p>
<div class="highlight highlight-source-julia"><pre><span class="pl-k">module</span> C

<span class="pl-k">export</span> f
<span class="pl-k">using</span> MetaMerge, A, B
<span class="pl-en">f</span>(<span class="pl-k">::</span><span class="pl-c1">Union()</span>) <span class="pl-k">=</span> <span class="pl-c1">nothing</span>
<span class="pl-c1">fmerge!</span>(f, (A,f), (B,f), conflicts_favor<span class="pl-k">=</span>A)
<span class="pl-en">h</span>(x<span class="pl-k">::</span><span class="pl-c1">Int64</span>) <span class="pl-k">=</span> <span class="pl-c1">f</span>(x)

<span class="pl-k">end</span></pre></div>
<p>The result is that unqualified use of <code>f</code> in the module <code>C</code> will dispatch across methods defined for <code>A.f</code> and <code>B.f</code>. We can check this in the REPL:</p>
<pre><code>julia&gt; methods(C.f)
# 4 methods for generic function "f":
f(::None) at none:5
f(x1_A::Foo)
f(x1_A::Int64)
f(x1_B::Bar)

julia&gt; C.h(2)
2
</code></pre>
<p>I hope that this versatility makes <code>fmerge!()</code> suitable for more general use outside the REPL.</p>
<p>One is also free to <code>fmerge!()</code> functions of different names, as well as functions from the same module.</p>

<ol>
<li><del>Currently, <code>merge!()</code> only handles two <code>(Module, Function)</code> tuples in its argument. In the future, one should be able to call <code>merge!()</code> on any number of such arguments, e.g. <code>merge!(f, (A,f))</code> or <code>merge!(f, (A,f), (B,f), (C,f))</code>.</del> featured in v0.3.</li>
<li>Currently, if one wants to merge multiple functions from two+ modules, one has to <code>merge!()</code> each set of names individually. In the future, there should be a <code>mergeall()</code> function that automatically merges all commonly named functions between two modules, e.g. <code>mergeall(A, B, conflicts_favor=A)</code> generates a list of function names common to <code>A</code> and <code>B</code> and <code>merge!</code>s them.</li>
<li>Find a way to handle name clashes of user defined types from different modules (WARNING above).</li>
</ol>
