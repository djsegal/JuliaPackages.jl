<p dir="auto">Luna.jl is a flexible platform for the simulation of nonlinear optical dynamics—both in waveguides (such as optical fibres) and free-space geometries—using the unidirectional pulse propagation equation (UPPE) and its approximate forms, such as the commonly used generalised nonlinear Schrödinger equation (GNLSE). Some of the key features of Luna:</p>
<ul dir="auto">
<li>A variety of propagation geometries treated in a unified way:
<ul dir="auto">
<li>Single-mode (mode-averaged) propagation in waveguides</li>
<li>Multi-mode propagation in waveguides with arbitrary (including non-symmetric) mode-shapes, full polarisation resolution, and intermodal coupling for arbitrary nonlinear polarisation terms</li>
<li>Waveguides with arbitrarily varying material properties and cross-sections (e.g. tapered fibres)</li>
<li>Free-space propagation with radial symmetry</li>
<li>Full (3+1)-dimensional free-space propagation</li>
</ul>
</li>
<li>Both field-resolved and envelope propagation equations</li>
<li>A range of linear and nonlinear optical effects:
<ul dir="auto">
<li>Modal dispersion and loss in waveguides</li>
<li>Optical Kerr effect (third-order nonlinearity)</li>
<li>Raman scattering in molecular gases or glasses</li>
<li>Strong-field photoionisation and plasma dynamics</li>
</ul>
</li>
<li>A built-in interface for the running and processing of multi-dimensional <a href="#running-parameter-scans">parameter scans</a> in serial or parallel</li>
<li>A standard library of <a href="#plotting-results">plotting</a> and <a href="#output-processing">processing</a> functions, including calculation of spectrograms and beam properties</li>
</ul>
<p dir="auto">Luna is designed to be extensible: adding e.g. a new type of waveguide or a new nonlinear effect is straightforward, even without editing the main source code.</p>
<p dir="auto">Luna was originally developed for modelling ultrafast pulse propagation in gas-filled hollow capillary fibres and hollow-core photonic crystal fibres. Therefore, such simulations have particularly good support. It is also excellent for modelling propagation in solid-core fibres.</p>
<p dir="auto">Luna is written in the <a href="https://julialang.org/" rel="nofollow">Julia programming language</a>, chosen for its unique combination of readability, ease of use, and speed. If you want to use Luna but are new to Julia, see <a href="#new-to-julia">the relevant section of this README</a>.</p>
<p dir="auto">There are two ways of using Luna:</p>
<ol dir="auto">
<li>A very simple high-level interface for the most heavily optimised applications of Luna: propagation in gas-filled hollow capillary fibres and hollow-core photonic crystal fibres (consisting of the function <a href="#quickstart"><code>prop_capillary</code></a> and some helper functions to create input pulses); or propagation of simple GNLSE simulations (consisting of the function <a href="#gnlse"><code>prop_gnlse</code></a>).</li>
<li>A low-level interface which allows for full control and customisation of the simulation parameters, the use of custom waveguide modes and gas fills (including gas mixtures), and free-space propagation simulations.</li>
</ol>
<p dir="auto">For a short introduction on how to use the simple interface, see the <a href="#quickstart">Quickstart</a> or <a href="#gnlse">GNLSE</a> sections below. More information, including on the internals of Luna, can be found in the <a href="http://lupo-lab.com/Luna.jl" rel="nofollow">Documentation</a>.</p>

<p dir="auto">To run a simple simulation of ultrafast pulse propagation in a gas-filled hollow capillary fibre, you can use <code>prop_capillary</code>. We consider a frequency grid which spans from 120 nm to 4 μm and a time window of 1 ps.</p>

<p dir="auto">The shape of this array is <code>(Nω x Nz)</code> where <code>Nω</code> is the number of frequency samples and <code>Nz</code> is the number of steps that were saved during the propagation. By default, <code>prop_capillary</code> will solve the full-field (carrier-resolved) UPPE. <p dir="auto"><code>prop_capillary</code> accepts many keyword arguments (for a full list see the <a href="http://lupo-lab.com/Luna.jl/dev/interface.html" rel="nofollow">documentation</a>) to customise the simulation parameters and input pulse. One of the most important is <code>modes</code>, which defines whether mode-averaged or multi-mode propagation is used, and which modes are included. By default, <code>prop_capillary</code> considers mode-averaged propagation in the fundamental (HE₁₁) mode of the capillary, which is fast and simple but less accurate, especially at high intensity when self-focusing and photoionisation play important roles in the propagation dynamics.</p>
<p dir="auto">Mode-averaged propagation is activated using <code>modes=:HE11</code> (the default) or replacing the <code>:HE11</code> with a different mode designation (for mode-averaged propagation in a different mode). To run the same simulation as above with the first four modes (HE₁₁ to HE₁₄) of the capillary, set <code>modes</code> to <code>4</code> (this example also uses smaller time and frequency windows to make the simulation run a little faster):</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="julia&gt; prop_capillary(125e-6, 3, :He, 1; λ0=800e-9, modes=4, energy=120e-6, τfwhm=10e-15, trange=400e-15, λlims=(150e-9, 4e-6))"><pre>julia<span class="pl-k">&gt;</span> <span class="pl-c1">prop_capillary</span>(<span class="pl-c1">125e-6</span>, <span class="pl-c1">3</span>, <span class="pl-c1">:He</span>, <span class="pl-c1">1</span>; λ0<span class="pl-k">=</span><span class="pl-c1">800e-9</span>, modes<span class="pl-k">=</span><span class="pl-c1">4</span>, energy<span class="pl-k">=</span><span class="pl-c1">120e-6</span>, τfwhm<span class="pl-k">=</span><span class="pl-c1">10e-15</span>, trange<span class="pl-k">=</span><span class="pl-c1">400e-15</span>, λlims<span class="pl-k">=</span>(<span class="pl-c1">150e-9</span>, <span class="pl-c1">4e-6</span>))</pre></div>

<p dir="auto"><strong>NOTE:</strong> Setting <code>modes=:HE11</code> and <code>modes=1</code> are <strong>not</strong> equivalent, except if only the Kerr effect is included in the simulation. The former uses mode-averaged propagation (treating all spatial dependence of the nonlinear polarisation the same as the Kerr effect) whereas the latter projects the spatially dependent nonlinear polarisation onto a single mode. This difference is especially important when photoionisation plays a major role.</p>

<p dir="auto">The <code>Processing</code> module contains many useful functions for more detailed processing and manual plotting, including:</p>
<ul dir="auto">
<li>Spectral energy density on frequency or wavelength axis with optional spectral resolution setting (<code>Processing.getEω</code> and <code>Processing.getIω</code>)</li>
<li>Time-domain fields and pulse envelopes with flexible frequency bandpass and linear (dispersive) propagation operators (<code>Processing.getEt</code>)</li>
<li>Energy (<code>Processing.energy</code>) and peak power (<code>Processing.peakpower</code>) including after frequency bandpass</li>
<li>FWHM widths in frequency (<code>Processing.fwhm_f</code>) and time (<code>Processing.fwhm_t</code>) as well as time-bandwidth product (<code>Processing.time_bandwidth</code>)</li>
<li>g₁₂ coherence between multiple fields (<code>Processing.coherence</code>)</li>
</ul>

<p dir="auto">At its core, Luna is extremely flexible, and the simple interface using <code>prop_capillary</code> only exposes part of what Luna can do. There are lots of examples in the <a href="examples/low_level_interface">low-level interface examples folder</a>. These are not actively maintained and are not guaranteed to run. As a side effect of its flexibility, it is quite easy to make mistakes when using the low-level interface. For example, changing from single-mode to multi-mode propagation in a fibre requires several concurrent changes to your code. If you have trouble with this interface, <a href="https://github.com/LupoLab/Luna/issues/new">open an issue</a> with as much detail as possible and we will try to help you run it.</p>

<p dir="auto">Julia fully supports <a href="https://docs.julialang.org/en/v1/manual/variables/" rel="nofollow">Unicode symbols in code</a>, including Greek letters. Luna makes heavy use of this to name variables <code>ω</code> instead of <code>omega</code>, <code>π</code> instead of <code>pi</code>, etc. In any Julia console you can enter many Unicode characters using <a href="https://docs.julialang.org/en/v1/manual/unicode-input/" rel="nofollow">a backslash and the tab key</a>, for example <code>\omega&lt;tab&gt;</code> will result in <code>ω</code>, and <code>\ne&lt;tab&gt;</code> will result in <code>≠</code> (and the latter is equivalent to <code>!=</code>). For even faster entry of Greek letters specifically, you can use <a href="https://github.com/q2apro/ahk_greekletters">this AutoHotkey script</a> or a number of other solutions.</p>
