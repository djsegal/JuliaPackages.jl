<p dir="auto">Cluster Validity Indices (CVIs) are designed to be metrics of performance for unsupervised clustering algorithms.
In the absense of supervisory labels (i.e., ground truth), clustering algorithms - or any truly unsupervised learning algorithms - have no way to definitively know the stability of their learning and accuracy of their performance.
As a result, CVIs exist to provide metrics of partitioning stability/validity through the use of only the original data samples and the cluster labels prescribed by the clustering algorithm.</p>
<p dir="auto">This Julia package contains an outline of the conceptual usage of CVIs along with many <a href="https://ap6yc.github.io/ClusterValidityIndices.jl/dev/examples/" rel="nofollow">example scripts in the documentation</a>.
This outline begins with <a href="#implemented-cviicvis">a list of CVIs</a> that are implemented in the lastest version of the project.
<a href="#quickstart">Quickstart</a> provides an overview of how to use this project, while <a href="#structure">Structure</a> outlines the project file structure, giving context to the locations of every component of the project.
<a href="#usage">Usage</a> outlines the general syntax and workflow of the CVIs/ICVIs.</p>

<p dir="auto">This section provides a quick overview of how to use the project.
For more detailed code usage, please see the [Detailed Usage](@ref usage).</p>
<p dir="auto">First, import the package with:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="# Import the package
using ClusterValidityIndices"><pre><span class="pl-c"><span class="pl-c">#</span> Import the package</span>
<span class="pl-k">using</span> ClusterValidityIndices</pre></div>
<p dir="auto">CVI objects are instantiated with empty constructors:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="# Create a Davies-Bouldin (DB) CVI object
my_cvi = DB()"><pre><span class="pl-c"><span class="pl-c">#</span> Create a Davies-Bouldin (DB) CVI object</span>
my_cvi <span class="pl-k">=</span> <span class="pl-c1">DB</span>()</pre></div>
<p dir="auto">All CVIs are implemented with acronyms of their literature names.
A list of all of these are found in the <a href="#implemented-cviicvis">Implemented CVIs/ICVIs</a> section.</p>
<p dir="auto">Next, get data from a clustering process.
This is a set of samples of features that are clustered and prescribed cluster labels.</p>
<blockquote>
<p dir="auto"><span class="color-fg-accent"><svg class="octicon octicon-info mr-2" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="M0 8a8 8 0 1 1 16 0A8 8 0 0 1 0 8Zm8-6.5a6.5 6.5 0 1 0 0 13 6.5 6.5 0 0 0 0-13ZM6.5 7.75A.75.75 0 0 1 7.25 7h1a.75.75 0 0 1 .75.75v2.75h.25a.75.75 0 0 1 0 1.5h-2a.75.75 0 0 1 0-1.5h.25v-2h-.25a.75.75 0 0 1-.75-.75ZM8 6a1 1 0 1 1 0-2 1 1 0 0 1 0 2Z"></path></svg>Note</span></p>
<p dir="auto">The <code>ClusterValidityIndices.jl</code> package assumes data to be in the form of Float matrices where columns are samples and rows are features.
An individual sample is a single vector of features.
Labels are vectors of integers where each number corresponds to its own cluster.</p>
</blockquote>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="# Random data as an example; 10 samples with feature dimenison 3
dim = 3
n_samples = 10
data = rand(dim, n_samples)
labels = repeat(1:2, inner=n_samples)"><pre><span class="pl-c"><span class="pl-c">#</span> Random data as an example; 10 samples with feature dimenison 3</span>
dim <span class="pl-k">=</span> <span class="pl-c1">3</span>
n_samples <span class="pl-k">=</span> <span class="pl-c1">10</span>
data <span class="pl-k">=</span> <span class="pl-c1">rand</span>(dim, n_samples)
labels <span class="pl-k">=</span> <span class="pl-c1">repeat</span>(<span class="pl-c1">1</span><span class="pl-k">:</span><span class="pl-c1">2</span>, inner<span class="pl-k">=</span>n_samples)</pre></div>
<p dir="auto">The output of CVIs are called <em>criterion values</em>, and they can be computed both incrementally and in batch with <code>get_cvi!</code>.
Compute in batch by providing a matrix of samples and a vector of labels:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="criterion_value = get_cvi!(my_cvi, data, labels)"><pre>criterion_value <span class="pl-k">=</span> <span class="pl-c1">get_cvi!</span>(my_cvi, data, labels)</pre></div>
<p dir="auto">or incrementally with the same function by passing one sample and label at a time:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="# Create a fresh CVI object for incremental evaluation
my_icvi = DB()

# Create a container for the values and iterate
criterion_values = zeros(n_samples)
for i = 1:n_samples
    criterion_values[i] = get_cvi!(my_icvi, data[:, i], labels[i])
end"><pre><span class="pl-c"><span class="pl-c">#</span> Create a fresh CVI object for incremental evaluation</span>
my_icvi <span class="pl-k">=</span> <span class="pl-c1">DB</span>()

<span class="pl-c"><span class="pl-c">#</span> Create a container for the values and iterate</span>
criterion_values <span class="pl-k">=</span> <span class="pl-c1">zeros</span>(n_samples)
<span class="pl-k">for</span> i <span class="pl-k">=</span> <span class="pl-c1">1</span><span class="pl-k">:</span>n_samples
    criterion_values[i] <span class="pl-k">=</span> <span class="pl-c1">get_cvi!</span>(my_icvi, data[:, i], labels[i])
<span class="pl-k">end</span></pre></div>
<blockquote>
<p dir="auto"><span class="color-fg-accent"><svg class="octicon octicon-info mr-2" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="M0 8a8 8 0 1 1 16 0A8 8 0 0 1 0 8Zm8-6.5a6.5 6.5 0 1 0 0 13 6.5 6.5 0 0 0 0-13ZM6.5 7.75A.75.75 0 0 1 7.25 7h1a.75.75 0 0 1 .75.75v2.75h.25a.75.75 0 0 1 0 1.5h-2a.75.75 0 0 1 0-1.5h.25v-2h-.25a.75.75 0 0 1-.75-.75ZM8 6a1 1 0 1 1 0-2 1 1 0 0 1 0 2Z"></path></svg>Note</span></p>
<p dir="auto">Each module has a batch and incremental implementation, but <code>ClusterValidityIndices.jl</code> does not yet support switching between batch and incremental modes with the same CVI object.</p>
</blockquote>

<p dir="auto">This project has implementations of the following CVIs in both batch and incremental variants:</p>
<ul dir="auto">
<li><strong><a href="https://AP6YC.github.io/ClusterValidityIndices.jl/dev/man/full-index/#ClusterValidityIndices.CH" rel="nofollow"><code>CH</code></a></strong>: Calinski-Harabasz.</li>
<li><strong><a href="https://AP6YC.github.io/ClusterValidityIndices.jl/dev/man/full-index/#ClusterValidityIndices.cSIL" rel="nofollow"><code>cSIL</code></a></strong>: Centroid-based Silhouette.</li>
<li><strong><a href="https://AP6YC.github.io/ClusterValidityIndices.jl/dev/man/full-index/#ClusterValidityIndices.DB" rel="nofollow"><code>DB</code></a></strong>: Davies-Bouldin.</li>
<li><strong><a href="https://AP6YC.github.io/ClusterValidityIndices.jl/dev/man/full-index/#ClusterValidityIndices.GD43" rel="nofollow"><code>GD43</code></a></strong>: Generalized Dunn's Index 43.</li>

<p dir="auto">If you have a question or concern, please raise an <a href="https://github.com/AP6YC/ClusterValidityIndices.jl/issues">issue</a>.
For more details on how to work with the project, propose changes, or even contribute code, please see the <a href="https://ap6yc.github.io/ClusterValidityIndices.jl/dev/man/contributing/" rel="nofollow">Developer Notes</a> in the project's documentation.</p>
<p dir="auto">In summary:</p>
<ol dir="auto">
<li>Questions and requested changes should all be made in the <a href="https://github.com/AP6YC/ClusterValidityIndices.jl/issues">issues</a> page.
These are preferred because they are publicly viewable and could assist or educate others with similar issues or questions.</li>
<li>For changes, this project accepts pull requests (PRs) from <code>feature/&lt;my-feature&gt;</code> branches onto the <code>develop</code> branch using the <a href="https://nvie.com/posts/a-successful-git-branching-model/" rel="nofollow">GitFlow</a> methodology.
If unit tests pass and the changes are beneficial, these PRs are merged into <code>develop</code> and eventually folded into versioned releases.</li>
<li>The project follows the <a href="https://semver.org/" rel="nofollow">Semantic Versioning</a> convention of <code>major.minor.patch</code> incremental versioning numbers.
Patch versions are for bug fixes, minor versions are for backward-compatible changes, and major versions are for new and incompatible usage changes.</li>
</ol>
