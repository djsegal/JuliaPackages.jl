<p dir="auto">This package lets you define metadata about fields in a struct, like tags
in Go. It uses a similar syntax to Parameters.jl, with a <code>|</code> bar instead of <code>=</code>.
You can use it as a minimalist replacement for Parameters.jl with the aid of
<a href="https://github.com/rafaqz/FieldDefaults.jl">FieldDefaults.jl</a>.</p>
<p dir="auto">Note that development effort has shifted to <a href="https://github.com/rafaqz/ModelParameters.jl">ModelParameters.jl</a>,
which achieves similar goals  in an arguably cleaner way.</p>
<p dir="auto">FieldMetadata on nested structs can be flattened into a vector or tuple very efficiently with <a href="https://github.com/rafaqz/Flatten.jl">Flatten.jl</a>, where they are also used to exclude fields from flattening.</p>
<p dir="auto">This example that adds string description metadata to fields in a struct:</p>

<p dir="auto">You can chain as many metadata macros together as you want. As of
FieldMetadata.jl v0.2, macros are written in the same order as the metadata
columns, as opposed to the opposite order which was the syntax in v0.1</p>
<p dir="auto">However, @with_kw from Parameters.jl must be the last macro and the first field,
if it is used. Additionally, any field with a default value must also have a metadata
annotation. If you assign a default value but no metadata to any
field, it will raise a <code>LoadError</code> with a message <code>type XXX has no field head</code>.
You can use the default value by adding <code>_</code> as an annotation, e.g.</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="@bounds @with_kw struct DefaultWithKeyword{T}
    a::T = 0 | _  # omitting the `| _` will cause an errow
    b::T = 0 | (0, 1)
end"><pre><span class="pl-c1">@bounds</span> <span class="pl-c1">@with_kw</span> <span class="pl-k">struct</span> DefaultWithKeyword{T}
    a<span class="pl-k">::</span><span class="pl-c1">T</span> <span class="pl-k">=</span> <span class="pl-c1">0</span> <span class="pl-k">|</span> _  <span class="pl-c"><span class="pl-c">#</span> omitting the `| _` will cause an errow</span>
    b<span class="pl-k">::</span><span class="pl-c1">T</span> <span class="pl-k">=</span> <span class="pl-c1">0</span> <span class="pl-k">|</span> (<span class="pl-c1">0</span>, <span class="pl-c1">1</span>)
<span class="pl-k">end</span></pre></div>
<p dir="auto">You can also update or add fields on a type that is already declared using a
<code>begin</code> block syntax. You don't need to include all fields or their types.</p>
<p dir="auto">This is another change from the syntax in v0.1, where <code>@re</code> was prepended
to update using the same struct syntax.</p>

<p dir="auto">To use them, call:</p>
<div class="highlight highlight-source-julia notranslate position-relative overflow-auto" dir="auto" data-snippet-clipboard-copy-content="import FieldMetadata: @prior, prior"><pre><span class="pl-k">import</span> FieldMetadata<span class="pl-k">:</span> <span class="pl-c1">@prior</span>, prior</pre></div>
<p dir="auto">You <em>must</em> <code>import</code> at least the function to use these placeholders, <code>using</code> is
not enough as you are effectively adding methods for you own types.</p>
<p dir="auto">Calling <code>@prior</code> or similar on someone else's struct may be type piracy and
shouldn't be done in a published package unless the macro is also defined there.
However, it can be useful in scripts.</p>
