<p dir="auto">Kibisis provides a doubly linked list implementation with <code>DoublyLinkedList{T}</code>. This
data type supports fast insertion and deletion at both ends as well as fast iteration. The type parameter <code>T</code> is the type of elements in the list.</p>
<p dir="auto">Check out <code>Deque</code> in <a href="https://juliacollections.github.io/DataStructures.jl/stable/deque/" rel="nofollow">DataStructures.jl</a> as it is likely better for your use case. This interface is primarily defined only for the implementation of <code>LRUSet</code> seen below.</p>

<p dir="auto">Kibisis provides an implementation of a set using LRU semantics in <code>LRUSet{T}</code>. When the size of the set grows larger than the set's capcity items are vacated. Insertion, membership queries, and iteration in order of most recently used are all fast. The type parameter <code>T</code> is the type of the elements in the set.</p>

<p dir="auto">By default the size of an item in the set is 1. However you can customize that behavior by adding a method to <code>Kibisis.item_size(x)</code> for your type of element. The method should take in the item and return a <code>Float64</code> object corresponding to its size. This value should not change over the lifetime of the object in the <code>LRUSet</code>.</p>

<p dir="auto">Note that multiple items can be evicted at once because of the custom <code>item_size</code> method. Also note that they are evicted in order of least recently used. Finally note that the inserted item may be evicted in the same call to <code>pushpop!</code> if its size is larger than the <code>LRUSet</code>'s capacity.</p>
<p dir="auto">One can also add methods to <code>on_new_push(x)</code>, <code>on_old_push(x)</code>, and <code>on_pop(x)</code> that do some kind of mutating operation (perhaps the creation and deletion of files on disk). These methods are executed on the items going into or leaving the <code>LRUSet</code>. <code>on_new_push</code> is executed when an item is used that is not currently in the <code>LRUSet</code>, and <code>on_old_push</code> is executed when the item already exists. By default all three of these do nothing.</p>
<p dir="auto">It is worth noting that <code>on_new_push</code> is executed before computation of <code>item_size</code> and <code>on_pop</code> is executed after. This makes it easy to use <code>LRUSet</code> to implement a file system cache in which the set only contains the names of files. One would define <code>item_size</code> to return the size of those files, <code>on_new_push</code> to create the file, and <code>on_pop</code> to delete the file. (consider what would happen if you ran <code>item_size</code> before creating the file or after deleting the file)</p>
<p dir="auto">Finally each of <code>on_new_push</code>, <code>on_old_push</code>, and <code>on_pop</code> can take in a variable amount of additional arguments to aid in their mutating operations. These arguments are then passed in as additonal arguments to <code>pushpop!</code> and will be propogated to these other methods. Note that if you use additional arguments for even one of the above, you will need to specify them in the signature of any others that you define even if the arguments are not used. In the example of creating a file system cache one would pass the file name as well as the file contents to <code>pushpop!</code>. Those file contents would be used in <code>on_new_push</code> (to write the file), but would not be used in <code>on_pop</code>. <code>on_old_push</code> would not need to be defined for this use case.</p>
<p dir="auto">It is worth noting that the additional arguments passed to <code>pushpop!</code> are not saved and so when an item is popped, <code>on_pop</code> is called with the current additional arguments. <strong>Not</strong> the arguments that were passed when that item was first placed into the <code>LRUSet</code>.</p>
