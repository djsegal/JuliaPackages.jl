<h2><a id="user-content-introduction" class="anchor" aria-hidden="true" href="#introduction"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Introduction</h2>
<p><em>beeswax</em> is a stack-based/cell-based 2-dimensional self-modifiable esoteric programming language drawing inspiration from bees moving around on a honeycomb, and partly drawing inspiration from <a href="http://esolangs.org/wiki/Cardinal" rel="nofollow">Cardinal</a>, and the Mensa Select winner of 2006, the bug-themed abstract board game <a href="https://en.wikipedia.org/wiki/Hive_%28game%29" rel="nofollow">Hive</a>.
The instruction pointers (bees) move around on a 2D hexagonal grid (the honeycomb). beeswax programs can manipulate their own source code, change the program size, and can read and write files.</p>
<h3><a id="user-content-honeycomb-structure-and-source-code-layout" class="anchor" aria-hidden="true" href="#honeycomb-structure-and-source-code-layout"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Honeycomb structure and source code layout</h3>
<p>The programming language draws inspiration from bees and honeycombs, so the instruction pointers (<code>bees</code>) travel through the program (<code>honeycomb</code>) along a hexagonal grid. Every honeycomb location has 6 neighbors.</p>
<h4><a id="user-content-honeycomb-structure" class="anchor" aria-hidden="true" href="#honeycomb-structure"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Honeycomb structure</h4>
<pre><code>a — b — c — d

  e — f — g — h
   \ / \ / \ / \

</code></pre>

<h2><a id="user-content-stack-layout-and-types-of-stacks" class="anchor" aria-hidden="true" href="#stack-layout-and-types-of-stacks"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Stack layout and types of stacks</h2>
<h3><a id="user-content-local-stacks" class="anchor" aria-hidden="true" href="#local-stacks"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Local stacks</h3>
<p>Every bee has a “personal” fixed size stack, carrying three <code>UInt64</code> values, called local stack or <code>lstack</code>. Every local stack is initialized to <code>[0,0,0]•</code> at the start of the program. <code>•</code> marks the top of the stack.</p>
<h3><a id="user-content-global-stacks" class="anchor" aria-hidden="true" href="#global-stacks"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Global stacks</h3>
<p>To store bigger amounts of data there is also a global stack (<code>gstack</code>) available where bees can drop values onto and pick up values from.</p>
<p>The global stack is <em>not limited in size</em> and only allows basic stack operations, but no other forms of data manipulation.
All arithmetic, bit manipulation and other operations have to be executed by bees on their local “personal” stacks.
So, before any data on the global stack can be used for calculations etc. a bee has to pick up values from the global stack. After executing the instructions the resulting values can be put back onto the global stack.</p>
<h2><a id="user-content-source-code-manipulations-self-modification-coordinate-system" class="anchor" aria-hidden="true" href="#source-code-manipulations-self-modification-coordinate-system"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Source code manipulations, self modification, coordinate system</h2>
<p>Bees can also drop values anywhere in the source code, and can pick up values from anywhere (analogous to flying to a location and manipulating certain cells in the honeycomb). So, self-modifying source code can be realized.</p>
<h2><a id="user-content-source-code-coordinate-system" class="anchor" aria-hidden="true" href="#source-code-coordinate-system"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Source code coordinate system</h2>
<p>The origin of the coordinate system [row,column] = [1,1] is in the upper left corner. beeswax uses 1-based indexing.
Bees that step outside the given honeycomb area are deleted, unless they are dropping values to cells that lie outside the current area of the honeycomb, which would let the honeycomb grow to the proper dimensions.</p>
<h3><a id="user-content-examples-of-self-modification" class="anchor" aria-hidden="true" href="#examples-of-self-modification"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Examples of self modification</h3>
<h4><a id="user-content-positive-growth-of-the-honeycomb" class="anchor" aria-hidden="true" href="#positive-growth-of-the-honeycomb"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Positive growth of the honeycomb</h4>
<pre><code>*PF((F(D
</code></pre>
<p>The command <code>P</code> increments the top value of lstack.</p>
<p><code>[0,0,1]•</code></p>
<p>Then <code>F</code>sets all lstack values to the top value.</p>
<p><code>[1,1,1]•</code></p>
<p><code>((</code>executes two arithmetic shifts left.</p>
<p><code>[1,1,4]•</code></p>
<p><code>F</code> sets all lstack values to the top value again.</p>
<p><code>[4,4,4]•</code></p>
<p>The next <code>(</code> executes 4 arithmetic shifts, because the 2nd lstack value is 4 now.</p>
<p><code>[4,4,64]•</code></p>
<p>And the final <code>D</code> drops the lstack top value (its according character) to [row,column] = lstack[2nd,3rd].</p>
<pre><code>*PF((F(D


   @
</code></pre>
<p>The resulting source code gets extended to a rectangle encompassing all code.</p>
<h4><a id="user-content-negative-growth-of-honeycomb-coordinate-system-reset" class="anchor" aria-hidden="true" href="#negative-growth-of-honeycomb-coordinate-system-reset"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>“Negative” growth of honeycomb, coordinate system reset</h4>
<p>Because the coordinate indices of the honeycomb are 1-based, growth in negative direction (‘up’ and ‘left’ in the source code) is only possible if the 2nd and/or 3rd local stack values are set to zero. Growth in negative direction can only be realized by steps of 1.
The coordinate origin of the resulting source code is reset to (row,column) = (1,1) again:</p>
<pre><code>*4F(@0~0@D
</code></pre>
<p>results in</p>
<pre><code>@
 *4F(@0~0@D
</code></pre>
<p>The new coordinate origin (1,1) of the honeycomb is set to the new upper left corner, where the <code>@</code> was put.</p>

<h3><a id="user-content-initial-pointer-creation" class="anchor" aria-hidden="true" href="#initial-pointer-creation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Initial pointer creation</h3>

<pre><code>*      Create bees in all directions 0,1,2,3,4,5

\      Create bees moving along the main direction 2/5.
/      Create bees moving along the main direction 1/4.

<h3><a id="user-content-program-termination" class="anchor" aria-hidden="true" href="#program-termination"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Program termination</h3>

<h3><a id="user-content-movement-and-redirection-related-instructions" class="anchor" aria-hidden="true" href="#movement-and-redirection-related-instructions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Movement and redirection related instructions:</h3>
<pre><code>&gt;,d,b,&lt;,p,q     Redirect bee to direction according to the diagram below:

        b   d
         \ /
      &lt; — β — &gt;
         / \
        p   q

a      Turn direction 1 step clockwise.

<h4><a id="user-content-important-notes-about-bee-creation" class="anchor" aria-hidden="true" href="#important-notes-about-bee-creation"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Important notes about bee creation</h4>
<p>At the beginning of a beeswax program, during initialization, bees are created in a fixed order, as follows:</p>
<p>First, the interpreter reads in the program code and creates the according honeycomb.
Then, the honeycomb is scanned for the initial bee creation instructions <code>*</code>, <code>\</code>, <code>/</code> and <code>_</code>. The honeycomb is scanned column-wise, starting with the leftmost column. The column is scanned from top to bottom, then the process is continued in the next column, until the whole honeycomb is scanned.</p>
<p>Each of the instructions creates a set of bees, which are pushed on a pointer/bee stack in order of creation.
For each creation instruction, the standard creation order of bees always follows the pattern below, in a counterclockwise fashion, starting at the lowest number. This always leaves the bee that was created last on top of the pointer stack:</p>
<pre><code>  2   1
        
3   β   0
        
  4   5
</code></pre>
<p>In the following examples <code>•</code> marks the top of the stack.</p>
<p><code>*</code> → [... 0 1 2 3 4 5]•</p>
<p><code>\</code> → [... 2 5]•</p>
<p><code>/</code> → [... 1 4]•</p>
<p><code>_</code> → [... 0 3]•</p>
<p>During program execution, the instructions are executed according to the order of the bees on the pointer stack, from top to bottom. The last created bee is always on top of the stack and, every tick, its instruction is always executed first.</p>
<p>The cloning instructions <code>X</code>, <code>E</code>, <code>H</code> and <code>W</code> create bees in an analogous fashion, always leaving out the opposite direction the bee is coming from:</p>
<p><code>X</code>: Check directions from 0 to 5, if the original bee is already moving in this direction, just let the bee move on. If the original bee is moving in the opposite direction, don’t create a clone. For every other position, clone the original bee and set its direction to the appropriate direction and push it on the pointer stack.</p>
<p><code>E</code>: Check direction 0, then direction 3. If the original bee is moving in one of these directions, just let the bee move on and don’t create a clone. If the original bee is coming from any different direction, change the direction of the original bee to 0, push a clone of that bee on the pointer stack, moving in direction 3.</p>
<p><code>H</code>: Check direction 1, then direction 4. If the original bee is moving in one of these directions, just let the bee move on and don’t create a clone. If the original bee is coming from any different direction, change the direction of the original bee to 1, push a clone of that bee on the pointer stack, moving in direction 4.</p>
<p><code>W</code>: Check direction 2, then direction 5. If the original bee is moving in one of these directions, just let the bee move on and don’t create a clone. If the original bee is coming from any different direction, change the direction of the original bee to 2, push a clone of that bee on the pointer stack, moving in direction 5.</p>
<p>Here is a beeswax example program that prints <code>Hello, World!</code> to the console. This should demonstrate quite nicely how the bees that are created during initialization end up on the pointer stack. I leave it to the user to figure out the details. It shouldn’t be too hard:</p>
<pre><code>W o    l   `
 ``     e  H
 `*`r`#`l`_`ld!
  ``      \/
  , o
</code></pre>
<p>If a bee leaves the honeycomb during program execution, it gets marked as dead. Once all instructions of the pointer stack are executed, the dead bees get deleted, and the stack gets cleaned up, leaving the general order of bees intact.</p>

<h4><a id="user-content-relative-addressing" class="anchor" aria-hidden="true" href="#relative-addressing"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Relative addressing</h4>
<pre><code>Y       Drop local stack top value to cell at relative [r,c]=[2nd,3rd].
        If relative [r,c] are lower than absolute [1,1], nothing is dropped.
Z       Get value from cell at relative [a,b]=[2nd,3rd] and put it as top value on local stack.
        If relative [r,c] are lower than absolute [1,1], then local stack top value=0.
</code></pre>
<p>As everyone knows, bees don’t have a concept of negative numbers, but they discovered that they can use the most significant bit of an address to get around that. Thus, coordinates relative to a bee’s position are realized by the <a href="https://en.wikipedia.org/wiki/Two's_complement" rel="nofollow">two’s complements</a>
of the coordinates. This way, half of the 64-bit address space is available for local addressing without wraparound in each direction.</p>
<p>The maximum positve 64-bit two’s complement address is <code>9223372036854775807</code> or <code>0x7fffffffffffffff</code> in 64 bit hex.
All values from 0 up to this value translate identically to the same 64 bit value.
All values <code>n</code> in the opposite (negative) direction translate to <code>2^64-n</code>
For example
<code>n=-1</code> is addressed by <code>18446744073709551615</code>,
<code>n=-9223372036854775808</code> is addressed by <code>9223372036854775808</code>.</p>
<p>Always keep the <code>UInt64</code> wrap-around in mind!</p>

<p>Start Julia and enter <code>using BeeswaxEsolang</code> in the console. The first run should take a few seconds for precompiling.</p>
<p><em>Normal program execution</em></p>
<p>Run any program with <code>beeswax("program_name")</code>. Without further parameters the program runs until a limit of <em>1e6 ticks</em> to prevent lockup if there are errors in the code.</p>
<p><em>Available parameters</em></p>
<p><code>beeswax("program_name",limit)</code> lets you determine a maximum limit of ticks the program is supposed to run.</p>
<p><code>beeswax("program_name",debug_mode,limit)</code> lets you determine a maximum limit of ticks and a debug mode.</p>
<p><code>beeswax("program_name",debug_mode,seconds_between_steps,limit)</code> lets you set the wait time (Float/Int in seconds) between the execution of each step.</p>
<h4><a id="user-content-available-debug-modes" class="anchor" aria-hidden="true" href="#available-debug-modes"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Available debug modes</h4>
<pre><code>    0   No debug messages.
    1   Output of all local stack contents at every tick to STDOUT.
    2   Output of program code with locations of all bees, plus all stack contents, for every tick.
    3   Output of program code with locations of all bees, plus all stack contents, for every tick. Bees are colored red.
    4..8 The same as mode 3, but the screen is fully refreshed every tick.
</code></pre>
<p>Debug modes 9 and 10 are quick tools to check a short beeswax code snippet.</p>
<pre><code>     Example for a two-line program directly entered via console:
