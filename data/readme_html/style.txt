Where it differs, prefer the Julia Docs guide, above.</h2>
<p dir="auto">This document lays out, in a very rough draft form, the style guidelines
for Julia programming that I've started to impose on myself. I'd ask that
anyone making contributions to my packages consider following these
guidelines as well.</p>
<p dir="auto">These guidelines are designed to encourage writing highly readable code
that performs well. They make explicit what I personally think is good
style.</p>
<p dir="auto">They are numbered to make them easier to reference in a discussion of
code, but the numbering is not yet finalized.</p>

<p dir="auto">(1) File names end in <code>.jl</code>, except for shell scripts which should not have any explicit file type extension.</p>
<p dir="auto">(2) GitHub repo names end in <code>.jl</code>.</p>
<p dir="auto">(3) Package names <em>do not</em> end in <code>.jl</code>.</p>
<h1 dir="auto"><a id="user-content-whitespace-and-line-breaks" class="anchor" aria-hidden="true" href="#whitespace-and-line-breaks"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Whitespace and Line Breaks</h1>
<p dir="auto">(4) Use four spaces when indenting:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="function foo(n::Integer)

        x += i
    end
    return x
end"><pre class="notranslate"><code>function foo(n::Integer)
    x = 0
    for i in 1:n
        x += i
    end
    return x
end
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="function foo(n::Integer)
  x = 0
  for i in 1:n
    x += i
  end
  return x
end"><pre class="notranslate"><code>function foo(n::Integer)
  x = 0
  for i in 1:n
    x += i
  end
  return x
end
</code></pre></div>
<p dir="auto">(5) Never use tabs instead of space characters as whitespace in code.</p>
<p dir="auto">(6) Never place more than 80 characters on a line.</p>
<p dir="auto">(7) Always include a single space after a comma:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="x[1, 2]"><pre class="notranslate"><code>x[1, 2]
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="x[1,2]"><pre class="notranslate"><code>x[1,2]
</code></pre></div>
<p dir="auto">(8) Never insert a space before a comma:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="x[1, 2]"><pre class="notranslate"><code>x[1, 2]
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="x[1 , 2]
x[1 ,2]"><pre class="notranslate"><code>x[1 , 2]
x[1 ,2]
</code></pre></div>
<p dir="auto">(9) Always insert a single space before and after an operator, except for the <code>^</code> and <code>:</code> operators, which never have spaces around them:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="1 + 1
1^2
1:5"><pre class="notranslate"><code>1 + 1
1^2
1:5
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="1+1
1 ^ 2
1 : 5"><pre class="notranslate"><code>1+1
1 ^ 2
1 : 5
</code></pre></div>
<p dir="auto">(10) Contrary to many other style guidelines, the spacing before-and-after rule applies to keyword arguments as well:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="foo(a = 1)"><pre class="notranslate"><code>foo(a = 1)
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="foo(a=1)"><pre class="notranslate"><code>foo(a=1)
</code></pre></div>
<p dir="auto">(11) Use explicit parentheses with the <code>:</code> operator in complex expressions. Do not rely upon Matlab-like precedence rules.</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="1:(n - 1)"><pre class="notranslate"><code>1:(n - 1)
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="1:n - 1"><pre class="notranslate"><code>1:n - 1
</code></pre></div>

<p dir="auto">(17) Always add explicit zeros to the ends of floating point constants:</p>
<p dir="auto"><strong>Good style</strong>
1.0 + 2.0</p>
<p dir="auto"><strong>Bad style</strong>

<p dir="auto">(18) Always explicitly type all arguments to a function. Explicit typing makes code safer to use and clearer to an unfamiliar user:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="foo(x::Real, y::Real; z::Real = 1) = x + y + z"><pre class="notranslate"><code>foo(x::Real, y::Real; z::Real = 1) = x + y + z
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="foo(x, y; z = 1) = x + y + z"><pre class="notranslate"><code>foo(x, y; z = 1) = x + y + z
</code></pre></div>
<p dir="auto">(19) When the desired types for a function are too generic to be tightly typed in Julia, use an explicit <code>Any</code>. This makes it clear that you intended for your code to work with any type of input.</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="screamcase(x::Any) = uppercase(string(x))"><pre class="notranslate"><code>screamcase(x::Any) = uppercase(string(x))
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="screamcase(x) = uppercase(string(x))"><pre class="notranslate"><code>screamcase(x) = uppercase(string(x))
</code></pre></div>
<p dir="auto">(20) Don't explicitly introduce a parametric type rule for a function unless it's needed to ensure correctness:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="foo(x::String) = print(x)
foo(x::Vector) = print(x)
foo{T &lt;: Real}(x::Vector{T}) = sum(x)"><pre class="notranslate"><code>foo(x::String) = print(x)
foo(x::Vector) = print(x)
foo{T &lt;: Real}(x::Vector{T}) = sum(x)
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="foo{T &lt;: String}(x::T) = print(x)
foo{T &lt;: Any}(x::Vector{T}) = print(x)"><pre class="notranslate"><code>foo{T &lt;: String}(x::T) = print(x)
foo{T &lt;: Any}(x::Vector{T}) = print(x)
</code></pre></div>
<p dir="auto">(21) Try to order method definitions from least specific to most specific type constraints.</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="foo(x::Any) = print(x)

</code></pre></div>

<p dir="auto">(22) Avoid creating temporary arrays, especially in loops.</p>
<p dir="auto">(23) Ensure that functions return a single type for each type signature of inputs.</p>
<p dir="auto">(24) Ensure that the type of any variable's binding does not change over the body of a function.</p>
<h1 dir="auto"><a id="user-content-code-organization" class="anchor" aria-hidden="true" href="#code-organization"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Code Organization</h1>
<p dir="auto">(25) Most code should exist in a package, except for isolated scripts. Make ad hoc packages to organize your own work.</p>
<p dir="auto">(26) When writing packages, obey the package organization rules by placing code in <code>src</code> and tests in <code>test</code>.</p>
<h1 dir="auto"><a id="user-content-testing" class="anchor" aria-hidden="true" href="#testing"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Testing</h1>
<p dir="auto">(27) Always write a separate test file for every source file you write. Specifically, place the tests for <code>src/foo.jl</code> in <code>test/foo.jl</code>.</p>
<p dir="auto">(28) The contents of <code>test/foo.jl</code> should be surrounded by a module to keep variables from leaking out:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="module TestFoo
    @assert foo
end"><pre class="notranslate"><code>module TestFoo
    @assert foo
end
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="@assert foo"><pre class="notranslate"><code>@assert foo
</code></pre></div>
<p dir="auto">(29) Test the functionality of <code>src/foo.jl</code> by writing at least one test for every type/function definition in <code>src/foo.jl</code>. Ensure systematic code coverage.</p>
<p dir="auto">(30) Avoid explicit types for variables inside code unless there is potential for bugs that you need to catch.</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="function foo()
    x = 1
    return x
end"><pre class="notranslate"><code>function foo()
    x = 1
    return x
end
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="function foo()
    x::Int = 1
    return x
end"><pre class="notranslate"><code>function foo()
    x::Int = 1
    return x
end
</code></pre></div>
<h1 dir="auto"><a id="user-content-comments" class="anchor" aria-hidden="true" href="#comments"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Comments</h1>
<p dir="auto">(31) Avoid overcommenting code. Focus on writing code that makes sense by using informative variable names and simple constructions. If you need to document a non-trivial algorithm or data structure, move that documentation into a specification file where it can be formatted nicely with diagrams and other information. English language documents are much more readable when they're not constrained by the rules for code comments.</p>
<p dir="auto">(32) Write separate specification documentation for non-obvious algorithms.</p>
<h1 dir="auto"><a id="user-content-be-conservative" class="anchor" aria-hidden="true" href="#be-conservative"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path d="m7.775 3.275 1.25-1.25a3.5 3.5 0 1 1 4.95 4.95l-2.5 2.5a3.5 3.5 0 0 1-4.95 0 .751.751 0 0 1 .018-1.042.751.751 0 0 1 1.042-.018 1.998 1.998 0 0 0 2.83 0l2.5-2.5a2.002 2.002 0 0 0-2.83-2.83l-1.25 1.25a.751.751 0 0 1-1.042-.018.751.751 0 0 1-.018-1.042Zm-4.69 9.64a1.998 1.998 0 0 0 2.83 0l1.25-1.25a.751.751 0 0 1 1.042.018.751.751 0 0 1 .018 1.042l-1.25 1.25a3.5 3.5 0 1 1-4.95-4.95l2.5-2.5a3.5 3.5 0 0 1 4.95 0 .751.751 0 0 1-.018 1.042.751.751 0 0 1-1.042.018 1.998 1.998 0 0 0-2.83 0l-2.5 2.5a1.998 1.998 0 0 0 0 2.83Z"></path></svg></a>Be Conservative</h1>
<p dir="auto">Julia often gives you more freedom than you should use. Here are some guidelines for exhibiting self-control in the face of temptation.</p>
<p dir="auto">(33) Don't use <code>importall</code>. Don't even use <code>import</code>. Explicitly annotate the source of each extended function at the point of extension:</p>
<p dir="auto"><strong>Good style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="Base.mean(x::MyNewType) = 1.0"><pre class="notranslate"><code>Base.mean(x::MyNewType) = 1.0
</code></pre></div>
<p dir="auto"><strong>Bad style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="import Base.median
median(x::MyNewType) = 1.0"><pre class="notranslate"><code>import Base.median
median(x::MyNewType) = 1.0
</code></pre></div>
<p dir="auto"><strong>Worst style</strong></p>
<div class="snippet-clipboard-content notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="importall Base
median(x::MyNewType) = 1.0"><pre class="notranslate"><code>importall Base
median(x::MyNewType) = 1.0
</code></pre></div>
<p dir="auto">(34) Always use <code>for x in y</code>. Never use <code>for x = y</code>.</p>
<p dir="auto">(35) Always initialize <code>BigFloat</code> objects using a string literal, not a floating point literal. The floating point literal has low precision, which will be propagated to the <code>BigFloat</code>.</p>
