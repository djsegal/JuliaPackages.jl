<p dir="auto">The GracePlot.jl module is a simple control interface for Grace/xmgrace - providing more publication-quality plotting facilities to Julia.</p>
<ul dir="auto">
<li>GracePlot.jl is ideal for seeding a Grace session with plot data before fine-tuning the output with Grace itself.</li>
<li>Grace "templates" (.par) files can then be saved/re-loaded to maintain a uniform appearance in publication.</li>
<li>The user is encouraged to pre-process data using math facilities from Julia instead of those built-in to Grace.</li>
</ul>

<p dir="auto">By default, GracePlot.jl assumes Grace is accessible from the environment path as <code>xmgrace</code>.  To specify a different command/path, simply set the <code>GRACEPLOT_COMMAND</code> environment variable.</p>
<p dir="auto">The value of <code>GRACEPLOT_COMMAND</code> can therefore be set from <code>~/.julia/config/startup.jl</code> with the following:</p>

<p dir="auto">GracePlot.jl currently provides a relatively "bare-bones" interface (despite offering significant functionality).</p>
<ul dir="auto">
<li>Does not currently provide much in terms of input validation.</li>
<li>Does not support the entire Grace control interface.
<ul dir="auto">
<li>In particular: GracePlot.jl does not support Grace math operations.  Users are expected to leverage Julia for processing data before plotting.</li>
</ul>
</li>
<li>On certain runs, Grace will complain that some commands cannot be executed... almost like commands are sent too fast for Grace, or something...  Not sure what this is.  Try re-running.</li>
</ul>

<p dir="auto">GracePlot.jl will post-process SVG files in an attempt to support the W3C 1999 standard.  The changes enable most new web browsers to display the SVG outputs.  Note, however, that text will not appear correctly on these plots.</p>
<p dir="auto">The EPS format is therefore suggested if high-quality vector plots are desired.</p>
